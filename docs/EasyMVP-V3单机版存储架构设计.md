> 目标：定义 EasyMVP V3 单机版的正式存储方案，明确数据库选型、本地文件存储边界、目录结构、可靠性策略与未来迁移约束。

## 1. 设计结论

EasyMVP V3 单机版的正式存储方案建议定为：

1. 主数据库使用 `SQLite`
2. 数据库运行在 `WAL` 模式
3. 大体量产物使用本地文件系统存储
4. 数据库存放结构化元数据、索引、状态与引用关系
5. 文档与表结构设计按“`SQLite` 优先、未来可迁移 `Postgres`”约束

一句话：

> V3 单机版不是轻量化 SaaS，而是本地工作台；因此默认存储形态应是 `SQLite + Local Files`。

## 2. 为什么首选 SQLite

当前 V3 的产品边界是：

1. 单用户
2. 单机版
3. 本地优先
4. 开箱即用
5. 轻量化

在这个前提下，`SQLite` 比 `MySQL/Postgres` 更符合目标。

### 2.1 选择 SQLite 的理由

1. 无需额外部署数据库服务
2. 安装和迁移成本极低
3. 非常适合本地工作台应用
4. 事务能力足以覆盖 V3 当前状态流
5. 便于打包、备份、移动和恢复

### 2.2 现在不优先选 MySQL/Postgres 的原因

1. 会引入不必要的安装和运维负担
2. 会破坏“单机版、开箱即用”的产品定位
3. 当前没有多用户并发与远程服务端吞吐压力
4. 早上重数据库会把文档和实现都拖回 SaaS 后台心智

## 3. 存储分层

V3 的单机版存储应明确分成三层：

1. 结构化状态层
2. 文件产物层
3. 缓存与临时层

### 3.1 结构化状态层

使用 `SQLite` 存：

1. 项目
2. 计划对象
3. 任务对象
4. Run 绑定
5. 事件索引
6. 待处理问题
7. 验收运行状态
8. 证据元数据
9. 本地设置

### 3.2 文件产物层

使用本地文件系统存：

1. 日志原文
2. replay 明细
3. 截图
4. 视频
5. 导出产物
6. 大型 JSON 快照
7. 采证附件

### 3.3 缓存与临时层

使用本地临时目录存：

1. SSE 本地缓存
2. 临时导出文件
3. 中间态调试文件
4. 可重建索引缓存

缓存层内容应允许随时清理，不应成为正式事实来源。

## 4. 数据库存什么，文件系统存什么

这是 V3 存储设计最重要的边界。

### 4.1 必须进数据库

以下对象必须进入 `SQLite`：

1. `Project`
2. `ProjectCategory`
3. `CategoryProfile` 当前生效引用
4. `PlanDraft`
5. `PlanReviewResult`
6. `CompiledPlan`
7. `CompiledTask`
8. `DomainTask`
9. `BrainRunBinding`
10. `AcceptanceRun`
11. `ActionInboxItem`
12. `LiveEvent` 索引
13. `Evidence` 元数据
14. `Settings`

### 4.2 不应直接进数据库

以下内容不应直接作为大字段长期塞进库：

1. 全量日志原文
2. 录屏文件
3. 大量截图二进制
4. 导出视频或构建产物
5. 大体量 replay 文件
6. 高频调试 telemetry 原始流

这些内容应落本地文件系统，数据库只保留：

1. 文件路径
2. 摘要
3. 类型
4. 大小
5. hash
6. 来源对象引用
7. 生成时间

## 5. SQLite 的正式使用约束

### 5.1 模式

建议：

1. 启用 `WAL`
2. 开启外键约束
3. 使用显式事务
4. 关键表建立必要索引

### 5.2 并发假设

当前文档默认：

1. 单用户
2. 同机单主进程
3. 少量后台 worker
4. 可接受有限并发写入

这与 `SQLite` 的能力边界是匹配的。

### 5.3 表设计约束

为了降低未来迁移成本：

1. 避免依赖 SQLite 特有 SQL 黑魔法
2. 不使用复杂触发器堆业务逻辑
3. 业务规则留在 Orchestrator，不塞进数据库层
4. `json` 字段可用，但要控制粒度
5. 高频查询字段必须有单独列，不要全靠 JSON 扫描

## 6. 建议目录结构

建议为单机版定义统一数据根目录，例如：

```text
~/.easymvp/
  ├─ data/
  │   ├─ easymvp.db
  │   ├─ easymvp.db-wal
  │   └─ easymvp.db-shm
  ├─ projects/
  │   └─ {project_id}/
  │       ├─ workspace/
  │       ├─ evidence/
  │       ├─ runs/
  │       ├─ replay/
  │       ├─ exports/
  │       └─ cache/
  ├─ settings/
  │   └─ config.json
  ├─ backups/
  └─ temp/
```

## 7. 项目级目录建议

每个项目建议固定目录结构：

```text
projects/{project_id}/
  ├─ workspace/
  ├─ evidence/
  │   ├─ screenshots/
  │   ├─ videos/
  │   ├─ exports/
  │   └─ reports/
  ├─ runs/
  │   └─ {run_id}/
  │       ├─ logs/
  │       ├─ checkpoints/
  │       └─ artifacts/
  ├─ replay/
  ├─ cache/
  └─ diagnostics/
```

原则：

1. 一个 `run_id` 对应一个稳定目录
2. 一个证据文件应能回溯到项目、run、surface、journey
3. 页面展示不直接扫描目录，仍以数据库索引为准

## 8. 关键对象的存储落点

### 8.1 计划对象

建议全部落库：

1. `PlanDraft`
2. `PlanReviewResult`
3. `CompiledPlan`
4. `CompiledTask`

原因：

1. 这些是状态链路核心对象
2. 需要强引用关系和版本回溯
3. 需要支撑工作台与 Plan 页面聚合

### 8.2 Run 对象

建议：

1. `BrainRunBinding`、状态、事件索引落库
2. logs 原文落文件
3. checkpoint 元数据落库，必要时 payload 落文件
4. replay 索引落库，明细落文件

### 8.3 验收对象

建议：

1. `AcceptanceRun`、Issue、Judgement 落库
2. Evidence 元数据落库
3. 证据附件落文件

### 8.4 工作台对象

建议：

1. `ProjectSnapshot` 不持久化为长期事实表
2. `WorkspaceView / PlanView / AcceptanceView` 作为聚合结果按需生成
3. 必要时可增加短期缓存表，但不得替代领域对象

## 9. 备份与恢复

单机版必须优先考虑“本地可恢复”，而不是“大规模分布式容灾”。

### 9.1 备份最小单元

建议最小备份单元为：

1. `easymvp.db`
2. `projects/{project_id}/`
3. `settings/config.json`

### 9.2 恢复原则

建议：

1. 先恢复数据库
2. 再校验项目文件目录
3. 对丢失附件的证据做降级标记
4. 对丢失日志的 run 标记为 `replay_incomplete`

## 10. 清理与 retention

V3 单机版会持续产生日志、截图、导出文件，因此必须设计保留策略。

### 10.1 默认长期保留

1. 项目元数据
2. 计划对象
3. 当前生效验收结果
4. 关键证据索引

### 10.2 可清理内容

1. 临时缓存
2. 调试级 telemetry
3. 过期中间截图
4. 非关键 replay 明细
5. 已归档项目的低价值日志

### 10.3 清理动作要求

1. 清理文件前必须保留数据库索引状态
2. 清理后要把 evidence/run 标记成 `artifact_pruned`
3. 页面上必须能看出“索引仍在，但附件已清理”

## 11. 性能与容量边界

对当前单机版，以下假设是成立的：

1. 项目数量有限
2. 活跃 run 数有限
3. 查询主要是单用户交互查询
4. 写入主要集中在事件和状态回写

因此第一阶段重点不是分库分表，而是：

1. 关键索引
2. 日志与附件出库
3. 聚合接口避免扫全表
4. 首页只读必要聚合字段

## 12. 迁移到 Postgres 的兼容约束

虽然当前正式选型是 `SQLite`，但文档和表设计必须允许未来迁移。

### 12.1 必须遵守

1. 主键策略保持一致
2. 不把业务语义锁死在 SQLite 特有行为上
3. JSON 字段仅承载非高频检索内容
4. 高价值查询字段必须拆列
5. 文件存储路径与数据库解耦

### 12.2 迁移时应保持不变

1. 领域对象模型
2. 页面聚合接口
3. 事件 schema
4. 目录结构语义

### 12.3 迁移时允许变化

1. 连接层实现
2. SQL 方言细节
3. 索引策略
4. 归档和备份实现

## 13. 不该怎么做

不建议：

1. 一开始就引入 MySQL/Postgres 作为单机版默认依赖
2. 把截图、视频、导出文件直接塞进数据库 blob
3. 用数据库当日志文件仓库
4. 让页面直接扫磁盘目录组织业务数据
5. 把缓存表当正式事实表

## 14. 对后续文档的约束

本专题确定后，后续文档应遵守：

1. 表结构设计默认以 `SQLite` 为第一落点
2. Evidence 文档必须区分“元数据入库”和“附件落文件”
3. 回放/审计文档必须区分“索引入库”和“明细落文件”
4. 页面聚合文档不得假设页面直接读取文件系统
5. 设置文档应把数据根目录作为正式配置项
