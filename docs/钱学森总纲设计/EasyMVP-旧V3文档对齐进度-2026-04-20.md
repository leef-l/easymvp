## 3. 已对齐的关键结论

当前已经统一的口径包括：

1. `completed` 不能由运行成功或 `production_passed` 直接偷换，必须由 `CompletionVerdict` 驱动。
2. 旧 V3 的 `PlanDraft / PlanReviewResult / CompiledPlan / AcceptanceRun` 继续保留为背景与细稿对象，但当前 EasyMVP 实现主链优先以闭环 6 对象为准。
3. 页面不能只显示成功/失败，必须显示 contract gap、人工检查点、升级对象和返工来源。
4. `execution_mode` 不再适合作为新的顶层建模中心，应改为脑路由、验证通道和人工检查点口径。
5. `plan_compile` 是 `reviewing` 阶段内部动作，不应继续扩散为独立运行阶段。
6. `AcceptancePage` 顶部主锚点应改为 `VerificationResult + CompletionVerdict`，不能继续只锚定 `production_passed`。
7. `WorkspacePage` 主入口必须可见阻塞原因、升级对象和人工检查点，不能退化成活动流页面。
8. 首页与单项目组件层的阶段展示开始统一收口到 `reviewing / executing / accepting / reworking / completed`。
9. 多项目首页、项目卡、Release Gate 等壳层组件不再使用 `production_passed` 直接偷换“完成”。
10. 总页面规范和单项目线框图开始统一区分 `acceptance passed` 与 `completed`，并把单项目工作台重新确认为闭环解释主入口。
11. onboarding/首次进入文档开始从旧 `Design / Review / Compile` 启动叙事，收口到当前早期 `reviewing` 引导态。
12. 创建链路与初始化事件流开始从旧 `Design / Review` 启动表达，收口到“创建状态机 + 早期 `reviewing` 项目态”的双层语义。
13. 组件 props 与推荐动作规则开始补齐升级类问题字段和当前阶段枚举边界，避免前端实现继续沿用旧阶段名或压平高优先级异常。
14. onboarding 引导卡和顶部状态 props 也开始补齐早期 `reviewing` 语义以及人工检查点/升级对象所需字段边界。
15. DTO / repository / evidence index 这一批数据契约文档开始明确：`AcceptanceRun` 等旧对象可以保留，但不再被解释为最终验证与完成语义中心。
16. GoFrame DTO、easymvp-brain 合同 schema、实现架构拆分文档也开始明确：旧 acceptance 结构是现状兼容资产，不再是当前权威完成语义。
17. Go 服务架构、数据库 schema、单机启动调用链文档也开始统一补充：现有 acceptance 表和调用链可保留，但最终验证与完成语义应逐步收口到 `VerificationResult / CompletionVerdict`。
18. migration / SQLite / 建表 SQL 文档也开始统一补充：现有 `acceptance_runs` 结构继续保留为现实资产，但不再等于当前顶层完成语义。
19. SQLite 初始化、索引优化、Evidence 引用规范也开始统一补充：`workflow_acceptance_runs / production_status` 与 Evidence 保留规则都不再围绕旧 acceptance 中心单独解释。
20. 首页壳层 props 与索引页面映射也开始统一补充：`production_status` 仍可保留为现实字段，但不再被解释为最终完成状态。
21. 事务边界、首页聚合来源、首页与单项目 DTO 也已统一补充：旧 acceptance 查询与写库现实仍保留，但不再作为当前总纲下的唯一裁决中心。
