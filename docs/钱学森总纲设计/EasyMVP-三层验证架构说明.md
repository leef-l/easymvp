# EasyMVP 三层验证架构说明

> 更新时间：2026-04-20
> 目标：统一 EasyMVP 当前与后续的验证环境口径，避免把 GitHub Actions 误写成长期总纲

---

## 1. 结论

EasyMVP 的验证架构应固定为三层：

1. **低配开发机**
2. **高配验证环境**
3. **GitHub Actions**

其中：

- **长期正确口径**：高配验证环境
- **当前阶段替代实现**：GitHub Actions
- **不承担最终验收**：低配开发机

一句话：

> GitHub Actions 不是原则本身，它只是当前缺少高配验证机时的远端替代通道。

---

## 2. 为什么要三层分离

当前实际约束很明确：

- 本地或线上宿主机配置较低
- 无法稳定承担 Go / 前端 / 打包等重验证任务
- 但系统又必须有最终编译、最终测试、最终构建口径

所以必须把“开发环境”和“最终验证环境”分离。

这符合钱学森《工程控制论》的几个基本原则：

- **时滞显式处理**
- **资源约束下求最优**
- **稳定优先于盲目本地放大**
- **反馈闭环必须有可信测量端**

---

## 3. 三层定义

## 3.1 低配开发机

定位：

- 开发、编排、文档、轻量排查、轻量只读检查

允许承担：

- 代码编辑
- 文档修改
- 日志检查
- 工作流触发
- 轻量 API 联调
- 低成本只读验证

不承担：

- 最终编译
- 最终构建
- 重测试
- 大规模 bundle / build / image build

结论：

> 低配开发机负责“推进工作”，不负责“给最终通过结论”。

## 3.2 高配验证环境

定位：

- 最终编译、最终测试、最终构建、最终打包

应承担：

- Go 编译与测试
- 前端 typecheck / build / bundle
- 镜像构建
- 重验收
- 大规模集成验证

这才是长期应收敛的正式验证终点。

结论：

> 高配验证环境才是最终验证口径。

## 3.3 GitHub Actions

定位：

- 当前阶段的远端替代验证通道

它现在之所以重要，不是因为它天然更正确，而是因为：

- 宿主机配置不够
- 需要一个可重复、可隔离、可记录的远端重验证环境

所以当前阶段：

- 可以把 GitHub Actions 视为“临时高配验证环境”
- 其日志、artifact、workflow run 可作为当前验收证据

但后续一旦有独立高配验证机，应允许：

- 高配验证机成为主口径
- GitHub Actions 退为补充 CI / 公共流水线 / 对外协作验证通道

---

## 4. 当前实施口径

### 当前真实情况

因为当前服务器配置偏低，EasyMVP 现阶段采用：

- 宿主机负责开发与轻量操作
- GitHub Actions 负责重验证

所以当前操作规则可以概括为：

1. **不要在低配宿主机上直接跑重编译和重测试**
2. **需要最终验证时，走 GitHub Actions**
3. **GitHub Actions 当前等价承担“远端高配验证”的职责**

### 注意

这只是当前实施口径，不是长期抽象口径。

长期写法应该是：

- “最终验证必须在满足资源条件的高配验证环境中完成”

而不是：

- “最终验证永远只能是 GitHub Actions”

---

## 5. 对文档和铁律的统一表述

后续文档里，建议统一使用下面这套表述。

### 推荐表述

- **最终验证口径：高配验证环境**
- **当前替代实现：GitHub Actions**
- **本地低配机不承担最终验收**

### 不推荐表述

- GitHub Actions 是最终唯一验证口径
- 一律只认 GitHub Actions，未来也不变

---

## 6. 对 EasyMVP 的直接影响

这套三层验证架构会直接影响：

1. **Acceptance**
   - 应接受“高配验证环境结果”作为顶层证据类型
2. **Evidence**
   - 当前可先落 GitHub Actions 证据；后续扩展为通用远端验证证据
3. **Verification Contract**
   - 不要把 contract 写死为 GitHub Actions，要写成“远端高配验证通道”
4. **Rework**
   - 返工判断应基于远端验证结果，而不是低配宿主机上的局部运行

---

## 7. 最终建议

现阶段：

- 继续用 GitHub Actions 承担重验证

中期：

- 增加独立高配验证机
- 把“远端验证适配层”从 GitHub Actions 特定实现抽出来

长期：

- 验证口径固定为“高配验证环境”
- GitHub Actions 只是其中一个实现通道
