# EasyMVP 使用文档

> 更新日期：2026-04-08

本文以当前代码中的 `Workflow V2` 为准，覆盖项目创建、设计对话、审核、执行、返工、验收和协作入口。

## 系统概览

EasyMVP 是一个 AI 协作开发平台。当前 MVP 模块的主流程是：

```text
创建项目
  -> 设计阶段（与架构师 AI 对话）
  -> 方案审核
  -> 执行阶段
  -> 验收阶段
  -> 完成

中途可暂停、返工、重规划，或进入人工审核/人工验收。
```

默认角色分工：

- `architect`：需求分析、任务蓝图、重规划
- `implementer`：代码或执行任务落地
- `auditor`：审核与问题识别
- `coordinator`：汇总、优化、推进流程

## 开始前检查

正式创建项目前，建议先完成以下准备：

1. 在 [AI应用使用指南](AI应用使用指南.md) 中按顺序配置 `供应商 -> 套餐 -> 模型 -> 引擎`。
2. 在 `MVP -> 角色预设` 中为常用项目分类准备默认角色模板。
3. 如果是编码类项目，准备好 `workDir`：
   - 目录存在
   - 已初始化 Git 仓库
   - `ai` / `mvp` 服务对目录有读写权限
4. 进入 `MVP -> 系统概览` 查看系统检查结果，确保关键项通过后再新建项目。

## 标准使用流程

### 1. 创建项目

入口：`MVP -> 项目管理 -> 新建项目`

关键字段：

- `项目名称`：项目展示名
- `项目分类`：按后端分类配置动态加载
- `项目简介`：给架构师的初始上下文
- `工作目录`：编码类项目必填，非编码类可留空
- `架构师 AI 模型`：可选；不选则回退到分类默认预设
- `项目角色配置`：可选；只勾选需要落成“项目级覆盖”的角色

当前实现中，如果新建项目时没有勾选项目级角色，运行期会直接回退到分类默认预设，不会强制为每个项目复制一整套角色记录。

### 2. 与架构师 AI 对话

创建成功后会自动进入 `/mvp/chat` 对话页。

你可以在这里：

- 描述需求、约束、技术栈和交付要求
- 让架构师继续补充方案或重新拆解
- 手动点击“检查拆分”解析任务蓝图
- 查看当前草稿任务数

对话页使用 SSE 实时回流 AI 输出，适合连续多轮澄清需求。

### 3. 确认方案

当草稿任务已经准备好后，点击“确认方案”。

当前代码中的行为不是“直接开始执行”，而是：

1. 提交进入审核阶段
2. 运行系统预检、审计员审核、协调员优化
3. 只有 `error=0` 且 `warning=0` 时，才推进到执行阶段

如果审核阶段仍有 warning，系统会把完整问题列表发回架构师对话，并要求重新输出修订方案。
支持两种修订方式：

- 整体重排：输出完整 `{"tasks": [...]}`，系统会解析为新的方案版本
- 局部修订：输出 `{"task_patches": [...]}`，系统会直接回写当前方案中的指定任务

如果内容太长，架构师可以分多段发送；每段都应是独立合法 JSON。只有在系统明确要求使用 `[AUTO_CONTINUE_NEXT]` 续传标记的场景下，架构师在当前段末尾追加该标记，系统才会自动继续索取后续分段；普通“请继续”之类自然语言不会触发自动续取。

如果审核未通过，可以在审核工作台查看问题并选择：

- 调整方案后重新确认
- 人工通过审核
- 人工驳回回到设计阶段

### 4. 监控执行

常用入口：

- `MVP -> 项目管理`：看项目列表、阶段、进度和快速操作
- `项目 -> 工作流仪表盘`：看阶段总览、关键按钮和多标签页工作台
- `项目 -> 执行控制台`：看领域任务、批次、失败项、资源锁
- `MVP -> 任务管理 / 任务日志`：看任务详情和执行日志

执行阶段常见人工操作：

- 重试失败任务
- 跳过失败任务
- 暂停项目
- 恢复项目
- 触发重规划

### 5. 返工与验收

项目执行结束后会进入验收阶段。验收控制台会展示：

- 验收轮次
- 分数与决策
- 验收问题
- 验收证据

人工可执行的操作：

- 放行验收
- 驳回验收
- 重新验收
- 驳回并返工

如果返工被触发，项目会进入 `reworking` 状态，系统继续推进修复和重新验证。

### 6. 暂停、恢复与回到设计阶段

当前项目支持以下控制：

- `暂停项目`：停止当前推进，保留上下文
- `恢复项目`：恢复工作流继续推进
- `回到设计阶段`：清理现有方案、任务和工作空间，重新设计

“回到设计阶段”属于重置操作，只在确认要放弃当前执行结果时使用。

## 状态说明

项目 / 工作流常见状态：

| 状态 | 含义 |
|------|------|
| `designing` | 设计中，仍在与架构师沟通和拆解任务 |
| `reviewing` | 方案审核中 |
| `executing` / `running` | 执行中 |
| `reworking` | 返工中 |
| `accepting` | 验收中 |
| `paused` | 已暂停 |
| `completed` | 已完成 |
| `failed` | 已失败 |
| `canceled` | 已取消 |

领域任务常见状态：

| 状态 | 含义 |
|------|------|
| `draft` | 草稿任务，尚未确认 |
| `pending` | 待调度执行 |
| `running` | 执行中 |
| `completed` | 已完成 |
| `failed` | 失败 |
| `escalated` | 已升级，等待人工处理 |

## 协作入口

当前代码中已提供以下协作入口：

- `MVP -> Workflow -> 飞书协作`
- `MVP -> Workflow -> Telegram`
- `MVP -> 配置管理`
- `MVP -> 自治控制台`

这些页面可用于：

- 飞书 / Telegram 账号绑定与联通测试
- Bot 菜单配置
- 工作流自治模式与阈值查看
- MVP 全局配置维护

## 常见问题

### 1. 为什么确认方案后没有立刻开始写代码？

因为当前默认流程是“先审核，再执行”。如果审核工作台还有问题项，项目不会直接进入执行阶段。

### 2. 编码类项目一定要填 `workDir` 吗？

要。当前分类如果属于编码类，`workDir` 是必填项；未准备好 Git 仓库或权限时，执行器很容易在执行阶段失败。

### 3. 项目一定会生成项目级角色吗？

不一定。新建项目时不勾选项目角色覆盖，运行时会直接走分类默认预设。

### 4. 失败任务怎么处理？

优先去执行控制台或任务列表看失败原因，再决定重试、跳过、暂停排查，还是触发重规划。
