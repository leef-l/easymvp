# 项目分类配置化改造设计方案

## 1. 背景

当前系统中的项目分类同时承担了以下三种语义：

1. 前端展示名称
2. 业务能力选择键
3. 规则/模板/角色预设的匹配键

现状问题：

1. `mvp_project.project_category` 当前保存的是中文展示值，如“软件开发”“数据分析”“小说创作”。
2. Accept 规则种子使用的是英文键，如 `software_dev`、`document`。
3. 引擎、角色预设、审核、验收、自治等模块对项目分类的使用方式并不统一。

这会导致：

1. 新增分类时需要改代码。
2. 分类改名会影响规则匹配。
3. 同类项目无法复用上层能力模板。
4. 中文展示值被误当成系统主键，后期难以治理。

因此需要将项目分类从“硬编码字符串”升级为“可配置分类模型”。

## 2. 设计目标

1. 项目分类支持后期动态新增，不依赖代码硬编码。
2. 分类展示名允许修改，不影响系统行为。
3. Accept、RolePreset、Autonomy、执行策略等能力按稳定编码匹配。
4. 支持“分类专属规则”和“家族级通用规则”两级复用。
5. 保留数据权限口子，便于后续按部门或租户扩展。
6. 提供平滑迁移路径，兼容当前已有中文分类数据。

## 3. 总体方案

项目分类拆成三层：

1. `display_name`
   用于前端展示和用户输入。
2. `category_code`
   系统内部稳定编码，作为业务主键。
3. `family_code`
   上层能力家族，用于规则、模板、执行策略复用。

最终依赖关系：

`project -> category_code -> family_code -> rules/templates/strategies`

系统后续禁止直接把展示名称作为业务主键使用。

## 4. 分类模型

### 4.1 分类编码

建议新增稳定编码，例如：

1. `software_dev`
2. `game_dev`
3. `novel_writing`
4. `animation_writing`
5. `movie_writing`
6. `data_analysis`
7. `product_design`

### 4.2 分类家族

建议家族编码：

1. `coding`
2. `creative`
3. `analysis`

示例映射：

1. `software_dev -> coding`
2. `game_dev -> coding`
3. `novel_writing -> creative`
4. `animation_writing -> creative`
5. `movie_writing -> creative`
6. `data_analysis -> analysis`
7. `product_design -> analysis`

### 4.3 展示名

展示名用于 UI，例如：

1. `software_dev -> 软件开发`
2. `game_dev -> 游戏开发`
3. `novel_writing -> 小说创作`
4. `data_analysis -> 数据分析`

展示名可修改，但 `category_code` 不应随意变更。

## 5. 数据模型

### 5.1 新增表 `mvp_project_category`

```sql
CREATE TABLE `mvp_project_category` (
  `id` bigint NOT NULL COMMENT '主键ID',
  `category_code` varchar(64) NOT NULL COMMENT '稳定分类编码',
  `display_name` varchar(64) NOT NULL COMMENT '展示名称',
  `family_code` varchar(32) NOT NULL COMMENT '能力家族编码',
  `description` varchar(255) DEFAULT NULL COMMENT '分类说明',
  `status` tinyint NOT NULL DEFAULT 1 COMMENT '1启用 0停用',
  `sort` int NOT NULL DEFAULT 100 COMMENT '排序',
  `created_by` bigint NOT NULL DEFAULT 0 COMMENT '创建人',
  `dept_id` bigint NOT NULL DEFAULT 0 COMMENT '部门ID',
  `created_at` datetime NOT NULL COMMENT '创建时间',
  `updated_at` datetime NOT NULL COMMENT '更新时间',
  `deleted_at` datetime DEFAULT NULL COMMENT '软删除时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_category_code` (`category_code`),
  KEY `idx_family_code` (`family_code`),
  KEY `idx_status_sort` (`status`, `sort`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='项目分类配置表';
```

### 5.2 调整表 `mvp_project`

建议：

1. 保留现有 `project_category`
2. 新增 `category_code`

```sql
ALTER TABLE `mvp_project`
ADD COLUMN `category_code` varchar(64) DEFAULT NULL COMMENT '项目分类编码' AFTER `project_category`,
ADD KEY `idx_category_code` (`category_code`);
```

说明：

1. 迁移期保留 `project_category` 作为兼容展示字段。
2. 新逻辑统一优先读取 `category_code`。
3. 待全部代码收敛后，可考虑将 `project_category` 改为冗余展示字段，或彻底移除。

## 6. 匹配策略

### 6.1 Accept 规则匹配

当前 Accept 规则不应再直接用 `project.project_category` 文本值匹配。

调整后规则加载顺序：

1. 先按 `category_code` 精确匹配专属规则
2. 若无专属规则，再按 `family_code` 匹配家族规则
3. 二者都存在时，可按优先级合并

建议：

1. `mvp_accept_rule.project_type` 字段后续更名为 `target_code`
2. 新增 `target_type`

可选结构：

1. `target_type = category`
2. `target_type = family`

如果暂不改表，可先约定：

1. `project_type` 可存 `software_dev`
2. `project_type` 也可存 `family:coding`

但长期更推荐显式 `target_type + target_code` 结构。

### 6.2 RolePreset 匹配

当前 `mvp_role_preset.project_category` 也是按中文分类绑定。

建议改为：

1. 先支持 `category_code`
2. 如无专属模板，则按 `family_code` 继承默认模板

这意味着：

1. `software_dev` 和 `game_dev` 可以共用 `coding` 家族模板
2. 如 `game_dev` 后续需要特化，再覆盖专属模板

### 6.3 审核、自治、执行策略

后续以下模块统一按 `category_code/family_code` 取策略：

1. 审核模板
2. Accept 模板
3. 执行器推荐策略
4. Autonomy 策略
5. 项目报告模板

禁止继续直接依赖中文展示值。

## 7. 运行时读取规则

建议新增 `CategoryResolver`，作为统一入口。

职责：

1. 根据 `project_id` 解析分类配置
2. 返回：
   - `category_code`
   - `display_name`
   - `family_code`
3. 提供兼容映射逻辑

建议结构：

```go
type CategoryInfo struct {
    CategoryCode string
    DisplayName  string
    FamilyCode   string
}
```

统一接口：

```go
ResolveProjectCategory(ctx, projectID) (*CategoryInfo, error)
```

后续模块一律通过该接口读取分类，不直接摸 `project_category`。

## 8. 兼容迁移方案

### 8.1 迁移阶段

分三步：

1. 新增 `mvp_project_category` 与 `mvp_project.category_code`
2. 初始化分类映射并回填现有项目
3. 逐步将业务代码切换到 `category_code`

### 8.2 初始映射

建议第一版建立如下映射：

1. `软件开发 -> software_dev -> coding`
2. `游戏开发 -> game_dev -> coding`
3. `小说创作 -> novel_writing -> creative`
4. `动漫创作 -> animation_writing -> creative`
5. `漫剧创作 -> comic_drama_writing -> creative`
6. `大电影创作 -> movie_writing -> creative`
7. `动画创作 -> animation_project -> creative`
8. `数据分析 -> data_analysis -> analysis`
9. `产品设计 -> product_design -> analysis`

### 8.3 过渡兼容逻辑

过渡期 `CategoryResolver` 逻辑：

1. 若 `mvp_project.category_code` 有值，则优先查分类表
2. 若为空，则根据历史 `project_category` 做兼容映射
3. 映射成功后可异步回填 `category_code`

这样可以避免一次性迁移失败导致主链不可用。

## 9. 数据权限预留

项目分类配置也是业务配置对象，需要保留权限口子。

冻结要求：

1. `mvp_project_category` 保留 `created_by / dept_id`
2. 分类列表查询后续支持 `ApplyDataScope`
3. 超级管理员可维护全部分类
4. 普通用户默认只读使用，不直接写分类配置

分类配置属于系统级基础数据，但不能堵死后续多租户或多部门扩展。

## 10. 代码改造点

### 10.1 后端

需要修改：

1. `admin-go/app/mvp/internal/engine/workflow.go`
   项目创建时写入 `category_code`
2. `admin-go/app/mvp/internal/workflow/stage/accept/service.go`
   改为通过 `CategoryResolver` 获取 `category_code/family_code`
3. `admin-go/app/mvp/internal/workflow/acceptance/rule_engine.go`
   支持按分类和家族加载规则
4. `admin-go/app/mvp/internal/engine/review.go`
   规则或审核模板不要再直接吃中文分类
5. `admin-go/app/mvp/internal/engine/config.go`
   现有 family 映射可下沉到分类配置表，避免双维护
6. `admin-go/app/mvp/internal/logic/project/*`
   项目创建/更新接口增加 `category_code`

### 10.2 数据层

新增：

1. `mvp_project_category` 实体/DAO/Repo
2. `CategoryResolver`

### 10.3 前端

需要调整：

1. 项目创建表单改为读取分类配置列表
2. 前端展示仍显示 `display_name`
3. 提交时使用 `category_code`

## 11. 第一阶段最小改造

如果当前不想一次改完，可先做最小可用版：

1. 新增 `category_code`
2. 新增 `mvp_project_category`
3. Accept 改走 `category_code/family_code`
4. RolePreset 暂时保留旧逻辑
5. 提供兼容映射函数

这样可以先解决当前最紧急的问题：

`Accept 规则因中英文分类不一致而加载失败`

## 12. 最终推荐方案

最终推荐不是“把中文分类改成英文”这么简单，而是：

1. 分类配置化
2. 引入稳定编码 `category_code`
3. 引入复用层 `family_code`
4. 统一通过 `CategoryResolver` 读取
5. 规则、模板、自治策略全部按编码体系运行

## 13. 结论

项目分类不应写死在代码里，也不应继续把展示名称当系统主键。

最合理的长期方案是：

1. `display_name` 负责展示
2. `category_code` 负责稳定匹配
3. `family_code` 负责能力复用

这样后续新增分类、改名、复用规则、扩展自治能力，都不需要反复改代码。
