# 21 · BrainKernel 错误模型 v1

> **定位**：本文档是 BrainKernel 内部以及 BrainKernel 与外部（Orchestrator、HTTP、UI、LLM 工具回路、其他大脑 sidecar）之间传递"失败"这一事实的**正式契约**。是 [02-BrainKernel设计.md](./02-BrainKernel设计.md) §13 的下位规格，同时是 [20-协议规格.md](./20-协议规格.md) §9 所引用的 `error.data` 字段的 schema 源头。
>
> **规格性质**：本文档使用 RFC 2119 关键词（**MUST** / **MUST NOT** / **SHOULD** / **SHOULD NOT** / **MAY**）。任何错误传递偏离 MUST 条款即为**错误模型污染**，主进程有权拒绝该错误、记审计日志、并在下一次 `health.ping` 把违规 sidecar 降级为 degraded。
>
> **适用范围**：本规格适用于所有 BrainKernel 主进程、所有 sidecar（无论 Go / Rust / Python / TypeScript 写的）、所有把 BrainKernel 错误向外传递的接口（HTTP API、SSE 流、gRPC、WorkflowRun 事件、tool_result）。`MCPAdapterRunner` 桥接的外部 MCP server 错误在进入主进程之前 **MUST** 先被 adapter 转译成本模型定义的结构。
>
> **版本**：错误模型版本 `v1`。v1 冻结后，`Class` 枚举值和 `error_code` 命名 **MUST NOT** 做破坏性改动（只能新增，不能改名 / 删除 / 改含义）。

## 1. 为什么要把错误模型独立成文档

错误是分布式系统里最昂贵的一类事件，却几乎是所有项目里最随意的一类代码。回顾 EasyMVP legacy 引擎曾经踩过的坑：

- **失败原因丢失**：Aider 执行器把 stderr 拼成 string 塞进 `task.error_message`，上层只能 grep，决策器永远看不懂"这是网络闪断"还是"这是工具 bug"。
- **重试策略靠猜**：调度器对所有失败统一"重试 3 次"，结果把 quota exceeded 也重试 3 次，白白烧钱；把明确的 compile error 也重试 3 次，白白浪费时间。
- **UI 直接暴露栈**：前端把 Go panic 的 stack trace 原样呈给用户，既不安全也不友好。
- **错误链断裂**：sidecar → 主进程 → Orchestrator → HTTP → UI 五层传递，每一层都会包一层自己的 error 类型，最后 root cause 找不到了。
- **跨语言不对齐**：Python 大脑抛的 `TimeoutError` 和 Go 主进程认识的 `context.DeadlineExceeded` 根本对不上，上层不得不写一堆 `if strings.Contains(...)`。

02 §13 的错误分类表只给了一个 snake_case 字符串枚举，远远不够一个严肃的多语言 sidecar 架构使用。本文档补上：

1. **一个跨语言都能实现的结构体**（BrainError）
2. **一个可机读的四维分类**（Class），让重试器不再靠猜
3. **一个可聚合的 fingerprint**，让同一类错误在观测平台自动聚类
4. **一条自底向上的转译链**，保证任何错误在经过 5 层之后 root cause 都还在
5. **一张重试决策矩阵**，让 `retryable` 字段有明确语义，而不是靠调用方心情

对应 02 的决策追溯锚点：**决策 4**（fault_policy 可配）和 02 §13（错误分类与恢复策略）。本文档是这两节的技术细化和跨语言规格化。

## 2. 设计原则

本节列出错误模型的 6 条设计原则。后续任何章节的 MUST/SHOULD 都可以追溯到这 6 条原则之一。

### 原则 1：错误是一等公民数据，不是 string

错误 **MUST** 是**结构化对象**，不是字符串拼接。主进程和 sidecar 之间传递的 error **MUST** 能被 JSON 编码为 [BrainError](#3-brainerror-核心结构) 结构。日志和 UI 可以只展示 `message` 字段，但底层链路上**禁止**只传 string。

### 原则 2：可机读优先于可读

Class、error_code、retryable 这三个字段是**给机器看**的，不允许为了"日志更好看"改动它们。调度器 / 熔断器 / 重试器 / 工作流引擎只看这三个字段做决策，从不看 `message`。`message` 才是给人看的。

### 原则 3：root cause 永远可追溯

每一次错误转译（比如 sidecar 把工具的 stderr 包成 `tool_execution_failed`、主进程把它包成 `brain_task_failed`）**MUST** 把上一层原始错误放进 `cause` 字段。一个传播 5 层的错误，最后应该能通过 `err.Cause.Cause.Cause.Cause` 追回最底层是什么。**MUST NOT** 在包装时丢弃 cause。

### 原则 4：fingerprint 保持稳定

同一类错误（比如"S3 connection reset"）在 1000 次发生时应产生**相同的 fingerprint**，以便 Sentry / Grafana / Loki 等观测平台自动聚类。fingerprint **MUST NOT** 包含时间戳、请求 id、随机 id、完整堆栈等每次都不同的字段。

### 原则 5：安全消息隔离（对外 vs 对内）

有些错误含敏感信息（内部路径、环境变量、凭证片段、SQL）。这些信息 **MUST** 只留在主进程侧日志和 trace 中，**MUST NOT** 出现在返回给 UI 的 HTTP body 或返回给 LLM 的 tool_result 中。错误模型提供 `internal_only` 字段来区分。

### 原则 6：错误是 WorkflowRun 事件源之一

每一个 Class ≠ `Transient` 的错误 **MUST** 产出至少一条 `mvp_workflow_event`，让 V2 工作流引擎能够基于错误重建状态、重放事件流。`Transient` 错误可以只进 metrics，不写事件表（否则会把事件表淹没）。

## 3. BrainError 核心结构

### 3.1 Go 语言规范实现

BrainError 是 BrainKernel 主进程内唯一允许跨层传递的错误类型。业务代码 **MUST** 返回 `*BrainError`，**MUST NOT** 返回原始的 `errors.New(...)` / `fmt.Errorf(...)` 直接穿过 Runner / Kernel 边界。

```go
// package: admin-go/app/mvp/internal/workflow/executor/brainkernel/errmodel

// BrainError 是 BrainKernel v1 错误模型的唯一标准承载结构。
type BrainError struct {
    // --- 机器决策字段（原则 2） ---
    Class     ErrorClass `json:"class"`              // 四维分类，决策器只看这个
    ErrorCode string     `json:"error_code"`         // snake_case，对应 §5 命名规范
    Retryable bool       `json:"retryable"`          // 明确语义：调度器是否该重试

    // --- 标识与聚合字段（原则 4） ---
    Fingerprint string `json:"fingerprint"`          // 16 位 hex，§6 算法
    TraceID     string `json:"trace_id,omitempty"`   // W3C Trace Context
    SpanID      string `json:"span_id,omitempty"`

    // --- 人读字段 ---
    Message string `json:"message"`                  // 一句话描述，给 UI / 日志
    Hint    string `json:"hint,omitempty"`           // 可选的自助排查提示

    // --- 可追溯字段（原则 3） ---
    Cause *BrainError `json:"cause,omitempty"`       // 上一层 root cause（递归结构）

    // --- 安全隔离字段（原则 5） ---
    InternalOnly *InternalDetail `json:"-"`          // MUST NOT 被 JSON 导出到 wire
    // InternalOnly 会被主进程落日志 / trace，但绝不进 HTTP body 和 tool_result

    // --- 上下文锚点 ---
    BrainID    string    `json:"brain_id,omitempty"`    // 哪个大脑抛的
    SidecarPID int       `json:"sidecar_pid,omitempty"` // 哪个 sidecar 进程
    OccurredAt time.Time `json:"occurred_at"`           // 发生时间
    Attempt    int       `json:"attempt,omitempty"`     // 这是第几次尝试（给重试器看）

    // --- 建议动作 ---
    Suggestions []string `json:"suggestions,omitempty"` // 结构化的可执行建议
}

// InternalDetail 仅存在于主进程内存，永不上 wire。
type InternalDetail struct {
    StackTrace string            // 完整 Go 栈 / sidecar 抛的语言栈
    EnvSnapshot map[string]string // 失败时的相关环境变量（已脱敏）
    RawStderr   string            // sidecar 未脱敏的 stderr
    RawRequest  json.RawMessage   // 引发错误的原始请求（仅在 debug 模式）
    Fields      map[string]any    // 任意附加结构化字段
}
```

### 3.2 字段级 MUST 清单

| 字段 | 必填？ | MUST 条款 |
|------|--------|-----------|
| `class` | 是 | **MUST** 是 §4 定义的 6 个枚举值之一，不接受 "unknown" |
| `error_code` | 是 | **MUST** 是 snake_case ASCII，长度 ≤ 64，**MUST** 在附录 A 保留清单里注册 |
| `retryable` | 是 | **MUST** 与 class 保持一致（详见附录 B 矩阵） |
| `fingerprint` | 是 | **MUST** 是 16 位 hex 字符串 |
| `message` | 是 | **MUST** 非空，**SHOULD** ≤ 200 字符 |
| `occurred_at` | 是 | **MUST** 是 UTC 时间戳 |
| `cause` | 否 | 存在时 **MUST** 本身也是合法 BrainError |
| `internal_only` | 否 | **MUST NOT** 进入任何 JSON 序列化输出（Go 里 tag `json:"-"`）|
| `suggestions` | 否 | **MUST** 是可执行文字，不是"请检查配置"这种废话 |

### 3.3 构造器规范

为了杜绝业务代码"随手写 `&BrainError{Class: "Transient"}`"这种绕过校验的做法，本规格要求：

- **MUST** 通过构造器 `errmodel.New(code, opts...)` 创建 BrainError
- **MUST** 通过包装器 `errmodel.Wrap(cause, code, opts...)` 包装上游错误
- **MUST NOT** 在业务代码中 `fmt.Errorf` 一个 BrainError 再 `errors.As` 回来

标准构造器签名：

```go
// New 创建一个新的 BrainError，class 和 retryable 由 code 查表决定。
func New(code ErrorCode, opts ...Option) *BrainError

// Wrap 包装一个上游错误：
//   - 如果 cause 是 *BrainError，会把 cause 放进 new.Cause，保留链条；
//   - 如果 cause 是普通 error，会转译成 Class=InternalBug 的 BrainError 再作为 cause；
//   - 如果 cause 是 nil，等价于 New。
func Wrap(cause error, code ErrorCode, opts ...Option) *BrainError

// Option 风格构造：
var err = errmodel.New(
    codes.ToolExecutionFailed,
    errmodel.WithMessage("code.run_tests exit 1"),
    errmodel.WithBrainID("implementer"),
    errmodel.WithSuggestions("Run `go test ./auth -v` locally"),
    errmodel.WithInternalStderr(rawStderr),
)
```

`errmodel` 包是整个 BrainKernel 的**单一错误制造入口**。任何其他包返回错误 **MUST** 走这个包。

### 3.4 跨语言等价实现

协议规格要求大脑 SDK 跨语言实现相同结构。各语言字段名 **MUST** 保持 snake_case JSON 编码一致：

```python
# Python sidecar SDK
@dataclass
class BrainError:
    class_: str                # 映射到 JSON "class"
    error_code: str
    retryable: bool
    fingerprint: str
    message: str
    cause: Optional["BrainError"] = None
    hint: Optional[str] = None
    trace_id: Optional[str] = None
    suggestions: List[str] = field(default_factory=list)
    occurred_at: datetime = field(default_factory=datetime.utcnow)
```

```rust
// Rust sidecar SDK
#[derive(Serialize, Deserialize, Debug)]
pub struct BrainError {
    pub class: ErrorClass,
    pub error_code: String,
    pub retryable: bool,
    pub fingerprint: String,
    pub message: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub hint: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub cause: Option<Box<BrainError>>,
    #[serde(default)]
    pub suggestions: Vec<String>,
    pub occurred_at: DateTime<Utc>,
}
```

```typescript
// TypeScript sidecar SDK
export interface BrainError {
  class: ErrorClass;
  error_code: string;
  retryable: boolean;
  fingerprint: string;
  message: string;
  hint?: string;
  cause?: BrainError;
  trace_id?: string;
  span_id?: string;
  suggestions?: string[];
  occurred_at: string; // ISO 8601
}
```

跨语言实现 **MUST** 通过[合规测试 C-E-01](#14-合规性测试矩阵)：往合规测试 fixture 里塞一个错误，反序列化后所有字段能正确访问。

## 4. Class 四维分类

Class 是整个错误模型最核心的决策字段。调度器 / 重试器 / 熔断器 / 告警器**只看 Class**决定行动。一个错误被归错 Class，会导致整个恢复策略被误导。所以本节规格有意很严格。

### 4.1 六个枚举值（v1 冻结）

v1 定义 **6 个** ErrorClass 枚举值。v1 之后**只能新增，不能改名/删除/改语义**。

| Class | 中文名 | 含义 | 典型例子 | 默认 retryable |
|-------|--------|------|----------|----------------|
| `Transient` | 瞬时故障 | 外部依赖短暂不可用，原请求重放大概率成功 | 网络闪断 / DB deadlock / LLM API 5xx / rate limit 但不是 quota | **true** |
| `Permanent` | 永久失败 | 重试同样请求永远失败，必须改输入或改代码 | 编译错误 / 测试失败 / 语法错误 / 找不到文件 / 参数非法 | **false** |
| `UserFault` | 用户输入错 | 用户提供的 prompt / 配置 / 文件有问题 | prompt 缺必填字段 / 上传文件格式错 / 项目路径不存在 | **false** |
| `QuotaExceeded` | 配额耗尽 | 钱 / token / 速率 / 并发 被打满，等待或升级后可恢复 | LLM API quota / 每分钟请求数超限 / 单日 token 上限 | **false**（但需要特殊处理） |
| `SafetyRefused` | 安全拒绝 | LLM / 策略层 / 沙箱拒绝执行，不可重试，需人工介入 | Anthropic safety refusal / 沙箱拒绝越权 / 策略闸门否决 | **false** |
| `InternalBug` | 内部 bug | 主进程 / sidecar 自己的代码 bug，不是外部原因 | panic / assertion failure / schema 校验意外失败 | **false**（但要报警） |

### 4.2 分类原则

- 一个错误只能属于**一个**主 Class。如果边界模糊，按"影响恢复策略最关键的那个维度"归类。
- 归类 **MUST NOT** 基于 HTTP status code 单点判断。例子：LLM API 返回 429 可能是 Transient（秒级重试）也可能是 QuotaExceeded（日级配额），调用方要看 body 才能分。
- `Transient` 是**唯一默认可重试**的 Class。其他 Class 的 `retryable` 默认是 false。`QuotaExceeded` 要等 cooldown，`SafetyRefused` 需要人工 override，`InternalBug` 要发版修，所以这三个重试没意义。
- **`UserFault` 永远不重试**。哪怕 UI 上用户点了"重试"按钮，也应是重新让用户改输入后再提交，不是靠调度器重放原请求。

### 4.3 和 Go 标准错误的映射

主进程有很多地方会拿到 Go 标准库或第三方库的 error。`errmodel.Wrap` **MUST** 按以下表进行映射：

| Go 原始错误 | 映射到 Class | 映射到 error_code |
|-------------|-------------|-------------------|
| `context.Canceled` | 不是错误 | 特殊：返回 `ErrCancelled`（详见 §8） |
| `context.DeadlineExceeded` | `Transient` | `deadline_exceeded` |
| `net.Error` 且 `Temporary()==true` | `Transient` | `network_transient` |
| `net.OpError` 且 `Op=="dial"` | `Transient` | `dial_failed` |
| `io.EOF` （读 sidecar stdout 时突然遇到） | `Transient` | `sidecar_stdout_eof` |
| `syscall.EPIPE` | `Transient` | `sidecar_stdin_broken_pipe` |
| `os/exec.ExitError` | `Permanent` | `sidecar_exit_nonzero` |
| `json.SyntaxError` / `json.UnmarshalTypeError` | `Permanent` | `frame_parse_error` |
| `sql.ErrNoRows` | `UserFault` | `record_not_found` |
| MySQL 1213 (deadlock) | `Transient` | `db_deadlock` |
| MySQL 1062 (duplicate key) | `UserFault` | `db_unique_violation` |
| LLM API 429 + `Retry-After` ≤ 60s | `Transient` | `llm_rate_limited_shortterm` |
| LLM API 429 + 日配额 | `QuotaExceeded` | `llm_quota_exhausted_daily` |
| LLM API 5xx | `Transient` | `llm_upstream_5xx` |
| LLM safety refusal | `SafetyRefused` | `llm_safety_refused` |
| panic 被 recover 住 | `InternalBug` | `panicked` |

本表**MUST**通过合规测试 C-E-02 验证。新增未列出的 Go 错误映射 **MUST** 同步更新本表并更新附录 A 保留清单。

### 4.4 反模式

以下写法**禁止**出现在业务代码里：

```go
// ❌ 反模式 1：猜 Class
if strings.Contains(err.Error(), "timeout") {
    return errmodel.New(codes.DeadlineExceeded, ...)
}

// ✅ 正确：用 errors.Is / errors.As
if errors.Is(err, context.DeadlineExceeded) {
    return errmodel.New(codes.DeadlineExceeded, ...)
}
```

```go
// ❌ 反模式 2：把一切塞成 Transient
return errmodel.New(codes.Unknown, errmodel.WithClass("Transient"))
// 这会导致编译错误都被重试 3 次，浪费时间和钱。

// ✅ 正确：走 Wrap 让映射表决定 Class
return errmodel.Wrap(err, codes.ToolExecutionFailed)
```

```go
// ❌ 反模式 3：丢失 cause
return errmodel.New(codes.SidecarFailed, errmodel.WithMessage(err.Error()))

// ✅ 正确
return errmodel.Wrap(err, codes.SidecarFailed)
```

## 5. error_code 命名规范

error_code 是比 Class 更细粒度的**字符串常量**。Class 回答"该不该重试 / 该不该报警"，error_code 回答"到底出了什么事"。

### 5.1 命名规则（v1 冻结）

- **MUST** 是 `snake_case` ASCII
- **MUST** 长度 ≤ 64 字符
- **MUST** 使用"主语_动词_对象"或"领域_事件"结构，不用含糊的 "failed" "error" "problem"
- **SHOULD** 以领域前缀开头便于聚类：`sidecar_*` / `tool_*` / `llm_*` / `db_*` / `policy_*` / `workflow_*` / `brain_*`
- **MUST** 在[附录 A 保留清单](#附录-a--error_code-保留清单-v1)里注册才能在 wire 上使用
- **MUST NOT** 在未注册的情况下编造新的 error_code

### 5.2 Go 端常量定义

```go
// package: admin-go/app/mvp/internal/workflow/executor/brainkernel/errmodel/codes

package codes

type Code string

// sidecar 进程级
const (
    SidecarStartFailed    Code = "sidecar_start_failed"
    SidecarExitNonzero    Code = "sidecar_exit_nonzero"
    SidecarHung           Code = "sidecar_hung"
    SidecarStdoutEOF      Code = "sidecar_stdout_eof"
    SidecarStdinBrokenPipe Code = "sidecar_stdin_broken_pipe"
    SidecarCrashed        Code = "sidecar_crashed"
    SidecarProtocolViolation Code = "sidecar_protocol_violation"
    SidecarVersionMismatch Code = "sidecar_version_mismatch"
)

// 工具执行
const (
    ToolNotFound          Code = "tool_not_found"
    ToolExecutionFailed   Code = "tool_execution_failed"
```

## 6. Fingerprint 算法

fingerprint 的目的：让观测平台（Sentry / Grafana Loki / 自建日志中心）能把 1000 次同源错误自动聚类成一个事件。

### 6.1 算法规格

Fingerprint **MUST** 按以下步骤计算：

1. 取以下字段拼成 canonical string（用 `|` 分隔）：
   - `error_code`
   - `brain_id`（可空则用 `""`）
   - `cause.error_code`（可空则用 `""`）
   - `message` 的**归一化**版本（见 §6.2）
2. 用 SHA-256 计算 hash
3. 取前 8 字节，转小写 hex（共 16 字符）

```go
func Fingerprint(e *BrainError) string {
    canonical := strings.Join([]string{
        string(e.ErrorCode),
        e.BrainID,
        causeCode(e.Cause),
        normalize(e.Message),
    }, "|")
    sum := sha256.Sum256([]byte(canonical))
    return hex.EncodeToString(sum[:8])
}
```

### 6.2 message 归一化规则

`message` 可能带动态值（路径、id、时间、数字），直接 hash 会导致同一类错误出现 1000 个 fingerprint。归一化 **MUST** 做以下替换：

| 模式 | 替换为 |
|------|--------|
| 数字 `\d+` | `<N>` |
| uuid / 雪花 id | `<ID>` |
| 绝对路径 `/[^ ]+` | `<PATH>` |
| 时间戳 ISO 8601 | `<TIME>` |
| 十六进制长串（≥8 位）| `<HEX>` |
| 引号内用户字符串 | `<STR>` |

例如 `tool_execution_failed: code.run_tests exit 1 in /tmp/worktree/abc123/auth_test.go at 2026-04-11T08:23:11Z` 归一化后是 `tool_execution_failed: code.run_tests exit <N> in <PATH> at <TIME>`。归一化后 fingerprint 对同类错误稳定。

### 6.3 禁止事项

fingerprint 计算 **MUST NOT** 包含：

- `occurred_at` / `trace_id` / `span_id`（每次都不同）
- `sidecar_pid`（重启就变）
- `attempt`（重试次数）
- 完整堆栈
- 任何随机 id / 请求 id

合规测试 C-E-03 会验证：同一归一化输入的 fingerprint 必须稳定；不同归一化输入的 fingerprint 必须不同。

## 7. 错误传播链与转译规则

### 7.1 五层传播模型

一个错误从 sidecar 内部的工具调用失败开始，要经过 5 层才到达用户眼前：

```
(L1) sidecar 内部业务代码
  │  原始 error（可能是 Python Exception / Rust anyhow / Go error）
  ▼
(L2) sidecar 协议层
  │  转成 BrainError，经 stdio JSON-RPC error 帧发回主进程
  ▼
(L3) 主进程 Runner / Kernel
  │  根据 fault_policy 决定：重试 / 降级 / 抛给上层
  │  如果抛给上层 → 用 errmodel.Wrap 再包一层（保留 cause）
  ▼
(L4) Orchestrator / WorkflowRun 引擎
  │  把 BrainError 写入 workflow_event、推进状态机、决定是否进 rework 阶段
  ▼
(L5) HTTP API / SSE / UI
     只暴露 message + error_code + suggestions（MUST 脱敏）
```

每一层 **MUST** 遵守：

1. **MUST** 把上层传来的错误放进新错误的 `cause`，**MUST NOT** 丢掉
2. **SHOULD** 在新错误里加本层专有的上下文（"在 executor.ExecuteDomainTask 时发生"）
3. **MUST NOT** 把 Class 从严重变轻或从轻变严重——只透传
4. **MUST NOT** 在 L5 暴露 `InternalOnly` / `cause.internal_only` / `trace_id` 给 UI（`trace_id` 只放 HTTP response header 供后台查询）

### 7.2 sidecar → 主进程（L2 → L3）

sidecar 通过 [20-协议规格.md §9](./20-协议规格.md#9-错误帧编码) 定义的 error 帧把 BrainError 发回。主进程 **MUST**：

- 校验 `data.class` 是合法枚举；若不是，本地改写成 `InternalBug` + `sidecar_protocol_violation`
- 校验 `data.retryable` 是 boolean；若不是，视为 false
- 校验 `data.error_code` 在附录 A；若不在，改写成 `unknown` 并告警
- 校验 `data.fingerprint` 是 16 位 hex；若不是，本地重算
- 以上任何一项不合规，**MUST** 在 metrics 里增加 `errmodel_violation_total{brain_id}`，并在 5 分钟内超阈值时把该 sidecar 降级为 `degraded`

### 7.3 主进程 → Orchestrator（L3 → L4）

主进程把错误抛给 Orchestrator 时 **MUST** 走 errmodel.Wrap：

```go
brainErr, err := runner.ExecuteTool(ctx, req)
if err != nil {
    return nil, errmodel.Wrap(err, codes.BrainTaskFailed,
        errmodel.WithMessage(fmt.Sprintf("brain %q failed to execute %q", brainID, req.Tool)),
        errmodel.WithBrainID(brainID),
    )
}
```

Orchestrator 看到的永远是 `*BrainError`，**MUST NOT** 看到原始 error 类型。

### 7.4 Orchestrator → HTTP（L4 → L5）

Orchestrator 抛给 HTTP 层时 **MUST** 做**脱敏**转换：

```go
// utility/response/error_render.go
func RenderBrainError(c *gin.Context, err *errmodel.BrainError) {
    // 写 trace_id 到 response header 供后台关联查日志，但不进 body
    c.Header("X-Trace-Id", err.TraceID)

    // internal_only 绝对不能进 body
    publicBody := map[string]any{
        "error_code": err.ErrorCode,
        "class":      err.Class,
        "retryable":  err.Retryable,
        "message":    err.Message,          // 已经是脱敏后的一句话
        "hint":       err.Hint,
        "suggestions": err.Suggestions,
    }
    // 递归暴露 cause，但也只暴露安全字段，跳过 internal_only
    if err.Cause != nil {
        publicBody["cause"] = publicCauseView(err.Cause)
    }

    c.JSON(httpStatusForClass(err.Class), publicBody)
}
```

HTTP status code 映射：

| Class | HTTP status |
|-------|-------------|
| `Transient` | 503 Service Unavailable |
| `Permanent` | 500 Internal Server Error（但经过脱敏） |
| `UserFault` | 400 Bad Request |
| `QuotaExceeded` | 429 Too Many Requests |
| `SafetyRefused` | 451 Unavailable For Legal Reasons |
| `InternalBug` | 500 Internal Server Error |

### 7.5 跨 sidecar 传播（大脑 A → 大脑 B）

一个大脑调用另一个大脑时，A 收到 B 的 error 帧 **MUST NOT** 直接把 B 的 BrainError 透传给自己的调用方，而是 **MUST** Wrap 成 A 自己命名空间的错误：

```python
# A sidecar 调用 B
try:
    result = await kernel_rpc("brain.call", {"target": "B", "method": "..."})
except BrainError as b_err:
    raise BrainError.wrap(b_err, code="upstream_brain_failed",
                          message=f"brain B refused: {b_err.message}")
```

这保证了错误传播链每一跳都有"谁在调用谁"的上下文，不会出现"主进程收到一个说不清是谁抛的 unknown error"。

## 8. 重试 / 熔断 / 降级决策

本节定义**唯一官方**的错误决策引擎。主进程调度器 / Runner / Orchestrator 的所有重试逻辑 **MUST** 通过这个引擎走，**MUST NOT** 自己写 if/else。

### 8.1 决策函数签名

```go
type Decision struct {
    Action      Action        // Retry / Fail / DegradeBrain / Quarantine / AskHuman
    BackoffHint time.Duration // 如果 Action == Retry，建议等多久再试
    MaxRetries  int           // 剩余重试预算
    ReasonLog   string        // 记给日志的人读原因
}

// Decide 是主进程的唯一决策入口。fault_policy 影响它的行为。
func Decide(err *BrainError, ctx DecideContext) Decision
```

`DecideContext` 里包含：

- 当前 fault_policy（`fail_fast` / `best_effort` / `retry`）
- 当前已重试次数
- 当前 sidecar 的 health state（healthy / degraded / quarantined）
- 当前大脑的错误预算剩余
- WorkflowRun 当前阶段（reviewing / executing / rework / accepting）

### 8.2 决策矩阵

以下决策矩阵 **MUST** 被 Decide 实现。合规测试 C-E-04 会逐行验证。

| 输入 Class | retryable | fault_policy | 当前 health | 动作 |
|-----------|-----------|--------------|-------------|------|
| `Transient` | true | 任意 | healthy | Retry（指数退避，上限 3 次） |
| `Transient` | true | 任意 | degraded | Retry（退避加倍，上限 1 次） |
| `Transient` | true | 任意 | quarantined | Fail（已隔离不允许再试） |
| `Transient` | false | 任意 | 任意 | Fail（违反 class 约束，同时报 violation） |
| `Permanent` | 任意 | `fail_fast` | 任意 | Fail 立即 |
| `Permanent` | 任意 | `best_effort` | 任意 | Fail 但不中断其他任务 |
| `Permanent` | 任意 | `retry` | 任意 | Fail（retry 策略不包括 Permanent） |
| `UserFault` | 任意 | 任意 | 任意 | Fail + 原样返回 UI 让用户改输入 |
| `QuotaExceeded` | 任意 | 任意 | 任意 | Fail + 写入 quota_event，触发 cooldown，暂停该大脑 |
| `SafetyRefused` | 任意 | 任意 | 任意 | AskHuman（进入人工节点，见 §10） |
| `InternalBug` | 任意 | 任意 | 任意 | DegradeBrain（该 sidecar 计一次 bug 配额），连续 ≥ 3 次 → Quarantine + P1 告警 |

### 8.3 指数退避参数

Retry 动作的退避参数：

- 初始：**1 秒**
- 倍数：**2×**
- 最大单次等待：**30 秒**
- 总重试次数：**≤ 3**（healthy 大脑）/ **≤ 1**（degraded 大脑）
- 抖动：**±20%**（避免重试风暴）

### 8.4 错误预算（error budget）

每个 sidecar 在每个观察窗口（默认 5 分钟）内有一个错误预算：

- `Transient` 错误：预算 `50` 次 / 5 分钟
- `Permanent` 错误：预算 `10` 次 / 5 分钟
- `InternalBug` 错误：预算 `3` 次 / 5 分钟

触达任一预算时：

- **MUST** 把 sidecar 降级为 `degraded`
- 降级态下重试预算减半、并发度降至 1
- 再超一次（`Transient` ≥ 100 / `Permanent` ≥ 20 / `InternalBug` ≥ 5）→ 升级到 `quarantined`
- `quarantined` 状态持续 10 分钟后自动回到 `healthy` 重新观察；但在 `quarantined` 期间新请求直接被 Decide 拒绝

`QuotaExceeded` 不计入常规预算，它走独立的 quota cooldown 路径。`SafetyRefused` 也不计入，它永远走 AskHuman。

### 8.5 熔断与半开态

降级和隔离共同构成两级熔断。从 `quarantined` 回到 `healthy` 前 **MUST** 有一个 **半开态**：

- 从 `quarantined` → `half_open`：主进程放一个探测请求（`health.ping` 或空执行）进去
- 如果探测成功 → 转 `healthy`
- 如果探测失败 → 回到 `quarantined` 再等 10 分钟

半开态 **MUST NOT** 对真实业务请求开放。合规测试 C-E-05 会验证此条。

## 9. 毒消息防御

"毒消息"指的是那种**重放必 panic** 的错误请求。如果不处理，调度器会反复重试它，每次都把 sidecar 打 panic，把整个大脑拖死。本节规定主进程 **MUST** 实现毒消息防御。

### 9.1 毒消息判定

符合以下任一条件 → 判定为毒消息：

1. 同一个 fingerprint 在**同一个 sidecar 进程生命周期**内连续出现 ≥ 3 次 → 立即标记毒消息
2. 同一个 fingerprint 导致 `InternalBug` Class 出现 ≥ 2 次 → 立即标记
3. 同一个请求（按 idempotency key / request hash）在不同 sidecar 实例上都导致 `panicked` 或 `assertion_failed` → 立即标记

### 9.2 毒消息处置

判定为毒消息后主进程 **MUST**：

- **立即停止对这条消息的任何重试**（哪怕 Class = Transient）
- 把请求 payload 落盘到 `dead_letter_queue` 表（一张独立隔离表）
- 把错误升级为 `AskHuman`，附加毒消息标签
- 在 metrics 里报 `poison_message_detected_total`
- 在 5 分钟内同类毒消息 ≥ 5 条 → P1 告警

### 9.3 与原则 6 的关系

毒消息 **MUST** 产生 `workflow_event` 和 `dead_letter_queue` 两份记录。前者是业务事件流，后者是诊断证据。

## 10. 安全拒绝升级通道

当 LLM 或策略闸门返回 `SafetyRefused`，系统 **MUST NOT** 重试或绕过，**MUST** 走人工升级通道。这是合规底线。

### 10.1 触发条件

- `Class == SafetyRefused`
- `error_code ∈ { llm_safety_refused, policy_gate_denied, tool_sandbox_denied }`
- 策略闸门（`mvp_policy_rule` / `mvp_risk_gate_rule`）主动拒绝

### 10.2 升级动作

主进程 **MUST**：

1. 把错误写入 `mvp_human_checkpoint` 表，状态 = `pending`
2. WorkflowRun 相应阶段状态转成 `paused`（对应 16 铁律：人工最高权限）
3. 在 UI 出现 "需要人工审核" 的操作入口，展示：
   - message
   - 触发的规则 / 模型拒绝理由
   - suggestions
   - 重新生成 / 覆盖放行 / 终止任务 三个按钮
4. 人工操作 **MUST** 写审计日志（`oplog`）包括谁在什么时间做了什么决定

### 10.3 不允许做的事

- **MUST NOT** 在主进程自动重试 `SafetyRefused`
- **MUST NOT** 把 `SafetyRefused` 改写成其他 Class 绕过闸门
- **MUST NOT** 默认放行：人工决定前，该任务保持 paused

## 11. 把错误交还给 LLM：tool_result 污染防御

Agent Loop 中，工具失败的错误会作为 `tool_result` 返回给 LLM 供它反思 / 重试 / 换策略。但 **原样返回 BrainError** 会带来两个问题：

- **信息泄露**：internal_only 里的栈 / 路径 / 环境变量会被 LLM 看到，可能被 prompt injection 反复引导输出出来
- **污染 context**：InternalBug 的长栈占掉大量 token，把 Prompt Cache 打碎（详见 Round 2 的 22-Agent-Loop规格.md）

### 11.1 tool_result 脱敏视图

返回给 LLM 的 tool_result **MUST** 只包含以下字段：

```json
{
  "ok": false,
  "error": {
    "class": "Permanent",
    "error_code": "tool_execution_failed",
    "message": "code.run_tests failed: 3 tests in auth_test.go failed",
    "hint": "Check auth_test.go and ensure auth middleware is mocked",
    "suggestions": [
      "Inspect the failing tests with verbose output",
      "Check if dependencies were updated"
    ]
  }
}
```

**MUST NOT** 包含的字段：

- `fingerprint` / `trace_id` / `span_id`（LLM 不需要）
- `cause.internal_only`（强泄露源）
- `sidecar_pid` / `brain_id`（让 LLM 知道内部拓扑没意义）
- `stack_trace`（污染 context）
- `raw_stderr`（污染 + 泄露）

### 11.2 对 prompt injection 的基线防御

用户工具输入 / stderr 可能含有 `"ignore all previous instructions"` 之类的注入 payload。把 stderr 原样转成 `message` 返回给 LLM = 把攻击向量直接喂给下一轮。所以：

- `message` **MUST** 是主进程**自己生成**的一句话摘要，**MUST NOT** 是 stderr 原样回显
- 原始 stderr **MUST** 只进 `internal_only.raw_stderr`
- 如果 sidecar 想把工具输出传回 LLM 用于反思，**MUST** 走单独的 `tool_result.stdout` 字段（有长度上限和内容过滤），**MUST NOT** 借 `error.message` 夹带

### 11.3 错误反思循环限制

同一个 `fingerprint` 的错误 **MUST NOT** 被 Agent Loop 在同一次任务内连续反思超过 **2 次**。第 3 次再出现 → Decide 直接返回 Fail，不再给 LLM 反思机会。这防止 LLM 陷入"我再改一下 / 再改一下 / 再改一下"的死循环（这是 Agent Loop 里非常真实的失败模式）。

- `error.fingerprint`
- `error.brain_id`
- `error.cause_code`（递归只取最深一层的 error_code）

### 13.3 Logs（必须）

`class ∈ { Permanent, UserFault, QuotaExceeded, SafetyRefused, InternalBug }` 的错误 **MUST** 落结构化日志（JSON 行），字段包含整个 BrainError 树（含 internal_only）。`class == Transient` 的错误 **SHOULD** 走采样日志（1%）避免日志洪峰。

### 13.4 审计事件（必须）

以下事件 **MUST** 写入 `oplog`：

- `SafetyRefused` 触发人工节点
- 人工对毒消息做决定
- sidecar 被 quarantine
- `errmodel_violation` 导致大脑降级
- fault_policy 被运行时修改

## 14. 合规性测试矩阵

以下测试用例是 **BrainKernel v1 错误模型合规测试**。任何实现（主进程和 sidecar SDK）**MUST** 全部通过才能声称合规。

| 编号 | 名称 | 验证点 |
|------|------|--------|
| C-E-01 | 跨语言序列化 | BrainError 在 Go / Python / Rust / TS 之间 round-trip 后所有字段无损 |
| C-E-02 | Go 错误映射表 | §4.3 每一行 Go 错误 → 正确的 Class + error_code |
| C-E-03 | Fingerprint 稳定性 | 归一化后相同输入 fingerprint 相同；时间戳 / id 不影响 fingerprint |
| C-E-04 | Decide 决策矩阵 | §8.2 表每一行输入产生期望的 Action |
| C-E-05 | 熔断半开态 | `quarantined` → `half_open` → `healthy` 状态机无跳转错位 |
| C-E-06 | sidecar 协议违规处理 | 非法 Class / 非 hex fingerprint 被主进程改写成 InternalBug + violation 计数 |
| C-E-07 | 错误传播保留 cause | 五层 Wrap 后最底层 error_code 仍可通过 `err.Cause.Cause.Cause.Cause` 取到 |
| C-E-08 | HTTP 脱敏 | RenderBrainError 产出的 body 不含 internal_only / stack / raw_stderr / trace_id |
| C-E-09 | tool_result 脱敏 | §11.1 字段白名单严格执行 |
| C-E-10 | 毒消息检测 | 同 fingerprint 连 3 次 → 停止重试 + 落 dead_letter_queue + AskHuman |
| C-E-11 | SafetyRefused 升级 | 触发后 WorkflowRun 状态转 paused + 人工检查点记录 + 禁止自动重试 |
| C-E-12 | Quota cooldown | QuotaExceeded 触发后该大脑在 cooldown 期内新请求被 Decide 直接 Fail |
| C-E-13 | 反思循环限制 | 同 fingerprint 在 Agent Loop 内超 2 次 → 强制 Fail |
| C-E-14 | 预定义 code 范围 | 所有 `error_code` 必须在附录 A 保留清单内 |
| C-E-15 | InternalOnly 隔离 | `internal_only` 在 JSON 序列化输出中不存在 |
| C-E-16 | 退避抖动 | 指数退避 ±20% 抖动落在 [0.8x, 1.2x] 区间 |
| C-E-17 | 错误预算独立计数 | Transient / Permanent / InternalBug 三个预算独立 |
| C-E-18 | 构造器单一入口 | 所有 BrainError 必须来自 errmodel.New 或 errmodel.Wrap（静态分析规则） |
| C-E-19 | Wrap nil 语义 | Wrap(nil, code) ≡ New(code) |
| C-E-20 | 合规指标上报 | errmodel_violation_total 在违规发生时递增，counter 语义正确 |

合规测试的执行环境与实现：详见 Round 3 的 25-测试策略.md。
