# 23 · BrainKernel 安全模型 v1

> **定位**：本文档是 BrainKernel 的**安全宪法**，定义主进程、sidecar、工具调用、LLM 访问、凭证管理、审计、合规拒绝路径的**正式规格**。是 [02-BrainKernel设计.md](./02-BrainKernel设计.md) §6（Tool Registry 与 Guardrail）和 §12.5.7（LLMAccess 三模式）的下位规格，同时与 [21-错误模型.md §10](./21-错误模型.md#10-安全拒绝升级通道) 的 SafetyRefused 升级通道配合工作。
>
> **规格性质**：本文档使用 RFC 2119 关键词。安全条款**默认从严**：只要文档没有明确 "MAY" 允许，就视为 "MUST NOT"。任何对安全条款的例外 **MUST** 有明确的人工审批和审计记录。
>
> **设计前提**：BrainKernel 运行在**部分可信**环境。主进程可信，sidecar 半可信（第三方大脑可能有 bug 也可能恶意），工具输入/输出不可信，LLM 输出不可信，用户输入半可信。本模型的所有条款都围绕"如何在上述信任分布下仍然能跑业务"展开。
>
> **版本**：安全模型版本 `v1`。v1 冻结后信任边界、凭证协议、审计事件类型 **MUST NOT** 破坏性修改。

## 1. 威胁模型

本节明确 v1 要防范的威胁。任何**没列入此清单**的威胁都不在 v1 保护范围内（需要再升级时写进 v2）。

### 1.1 对手类型

| 对手 | 能力 | 动机 | 防御优先级 |
|------|------|------|-----------|
| **恶意用户 prompt** | 通过 UI 或 API 提交 prompt | 越权访问、数据泄漏、越界执行 | P0 |
| **被污染的工具输出** | 读到的文件 / stderr / 网页含注入载荷 | 劫持 LLM 继续指令执行 | P0 |
| **恶意第三方 sidecar** | 作为插件加载的大脑 SDK | 窃取凭证、越权写文件、滥用 LLM 配额 | P1 |
| **LLM 幻觉** | 生成看似合法但越权的 tool_call | 误伤、绕过 guardrail | P1 |
| **compromised LLM provider** | 云端 LLM API 被劫持 | 泄漏 prompt / 篡改响应 | P2（依赖 provider 自身安全）|
| **本地多租户 | 同一台机器多个项目并行 | cross-tenant 数据泄漏 | P2 |
| **网络中间人** | sidecar ↔ LLM 流量 | TLS 已覆盖 | P3 |

### 1.2 资产清单

以下资产 **MUST** 被本模型保护：

- **LLM API 凭证**（Anthropic / OpenAI / Gemini / DeepSeek / 自建）
- **项目源代码**（worktree 目录及其 git history）
- **审计日志完整性**（禁止篡改）
- **用户身份与会话令牌**
- **BrainPlan / artifact 内容**（可能含业务逻辑与 PII）
- **`mvp_config` 中的所有 secret 字段**
- **Run / Turn 级成本数据**（防止被伪造用于逃避配额）

### 1.3 不在 v1 范围内

本模型 v1 **不解决**以下问题（明确声明以避免误用）：

- 抵御能物理访问服务器的对手（硬盘加密不在本规格范围内）
- 抵御 ring-0 kernel rootkit
- 抵御 Anthropic / OpenAI / Gemini 官方 API 被 compromise（交由 provider 自己负责）
- 抵御 LLM 生成正确 tool_call 但语义错误的"合法误伤"（这是业务正确性问题，不是安全问题）
- 完整的侧信道攻击防御

## 2. 信任边界

### 2.1 五个信任域

```
┌──────────────────────────────────────────────────────┐
│ Trust Zone 0 · Kernel Core                           │
│   - 主进程                                             │
│   - errmodel / authentication / audit logger         │
│   - credentials vault                                │
│   （完全可信，是 TCB）                                 │
└──────────────────────────────────────────────────────┘
           │ 信任降级：通过 Runner 启动
           ▼
┌──────────────────────────────────────────────────────┐
│ Trust Zone 1 · Built-in Brain Sidecars               │
│   - central_brain / code_brain / verifier_brain ...  │
│   （半可信：代码由自己维护，但受 sandbox 和 guardrail 约束）│
└──────────────────────────────────────────────────────┘
           │ 信任降级：通过 BrainRegistry 动态加载
           ▼
┌──────────────────────────────────────────────────────┐
│ Trust Zone 2 · 3rd-party Brain Sidecars              │
│   - 用户从插件市场加载的大脑                              │
│   （不可信：代码完全黑盒，必须签名 + sandbox + LLMAccess=proxied） │
└──────────────────────────────────────────────────────┘
           │ 信任降级：tool.invoke
           ▼
┌──────────────────────────────────────────────────────┐
│ Trust Zone 3 · Tool Execution                        │
│   - git / bash / docker / playwright / pytest ...    │
│   （不可信输入：文件内容 / stderr / stdout 都可能被注入）    │
└──────────────────────────────────────────────────────┘
           │ 信任降级：tool_result 回填给 LLM
           ▼
┌──────────────────────────────────────────────────────┐
│ Trust Zone 4 · LLM Output                            │
│   - LLM 生成的 text / tool_call                      │
│   （不可信：可能幻觉、可能被 prompt injection 劫持）       │
└──────────────────────────────────────────────────────┘
```

### 2.2 跨边界 MUST 条款

| 边界 | MUST |
|------|------|
| Zone 0 → Zone 1 | 凭证下发 **MUST** 按 §5 LLMAccess 模式决定 |
| Zone 1/2 → Zone 0（反向 RPC）| **MUST** 经过 guardrail 两层白名单 + rate limit + schema 校验 |
| Zone 2 对任何 Zone | **MUST** 默认禁止直连 LLM（proxied only）|
| Zone 3 → Zone 4 | tool_result **MUST** 通过 [22 §10 Sanitizer 流水线](./22-Agent-Loop规格.md#10-工具结果污染防御) |
| Zone 4 → Zone 3 | LLM 的 tool_call **MUST** 再过 schema + guardrail + resource lock |
| 任何边界 | 跨边界数据传递 **MUST NOT** 保留 Zone 0 的 `InternalDetail`（见 [21 §3](./21-错误模型.md#3-brainerror-核心结构) 安全隔离字段）|

### 2.3 Zone 2（3rd-party sidecar）特殊约束

Zone 2 的大脑 **MUST** 遵守以下额外限制：

1. **MUST NOT** 使用 `LLMAccess=direct`（只能 `proxied`，强制走主进程代理）
2. **MUST NOT** 访问宿主文件系统（通过 Docker / gVisor / 非 root 用户启动）
3. **MUST** 在 `brain.describe` 时声明全部 tool 白名单，运行时新增工具 → 拒绝
4. **MUST** 通过 §10 签名机制验证
5. **SHOULD** 在独立 network namespace，只能访问主进程代理端口
6. **SHOULD** 独立 cgroup，CPU / memory / FD 配额按 Zone 2 默认值限制

## 3. Sandbox 边界规格

### 3.1 Sandbox 的四个维度

| 维度 | 控制对象 | 默认策略 |
|------|----------|----------|
| **FS** 文件系统 | 允许读/写的路径 | worktree 目录内 read-write，宿主其他路径 read-only 或拒绝 |
| **NET** 网络 | 允许访问的域名 | 默认空白名单（明确列出每个允许的外部服务）|
| **PROC** 进程 | 允许执行的二进制 | 白名单（git / node / python / pytest 等具名）|
| **SYS** 系统调用 | seccomp 过滤器 | 禁 `mount` / `ptrace` / `kexec_load` / `bpf` 等高危 syscall |

### 3.2 FS 边界

**Worktree 内**：

- read / write / create / delete：**允许**
- chmod / chown：**允许**（但 setuid/setgid bit 强制清除）
- 符号链接：**允许创建**，但 **MUST NOT** 指向 worktree 外
- hard link：**禁止**创建跨越 worktree 边界的 hard link

**Worktree 外**：

- `/etc` / `/proc/self/environ` / `~/.ssh` / `~/.aws` / `~/.config` / `/root`：**MUST NOT** 读
- `/tmp` 和 `/var/tmp`：默认拒绝，tool 可按需申请临时目录（由 Kernel 分配子目录）
- 系统共享库：**MAY** 读
- 其他所有路径：**MUST** 拒绝

### 3.3 NET 边界

- **LLM 代理端口（仅 proxied）**：MUST 允许
- **工具显式声明的域名**：MAY 允许（例如 code_brain 需要 `pypi.org` / `registry.npmjs.org` / `github.com`）
- **元数据服务 IP**（`169.254.169.254`、`metadata.google.internal`、`metadata.azure.com`）：**MUST** 拒绝（防 SSRF 窃取云凭证）
- **回环 127.0.0.1**：默认拒绝（防绕过 proxy）
- **RFC1918 私网**：默认拒绝
- **公网**：默认拒绝，必须明确列入域名白名单

### 3.4 PROC 边界

每个大脑 `brain.describe` 里 **MUST** 声明 `allowed_binaries`：

```yaml
allowed_binaries:
  - git
  - node
  - npm
  - pnpm
  - pytest
  - go
```

Runner 通过 seccomp / ptrace / runtime wrapper 强制执行。任何未列入的二进制 → 阻止 + 记审计。

### 3.5 SYS 边界

基础 seccomp 策略（默认拒绝）：

- `mount` / `umount2` / `pivot_root` / `chroot`
- `kexec_load` / `kexec_file_load`
- `ptrace`（禁止工具追踪其他进程）
- `bpf`（禁止加载 eBPF 程序）
- `init_module` / `finit_module` / `delete_module`
- `settimeofday` / `clock_settime`

允许清单：标准的 file IO / net / mm / sched 系统调用。

### 3.6 Sandbox 实现可选项

| 级别 | 实现 | 适用 |
|------|------|------|
| L0 | 无沙箱（仅 guardrail）| 开发环境、Zone 1 built-in 大脑 |
| L1 | 独立 user + seccomp + resource rlimit | 生产环境、Zone 1 大脑 |
| L2 | Docker + user namespace + seccomp + AppArmor | Zone 2 第三方大脑默认 |
| L3 | gVisor / Firecracker microVM | 高风险任务（code_brain 跑不可信代码）|

每个大脑在 `ai_engine_config` 里 **MUST** 声明最低 sandbox 级别，Runner 按声明启动对应沙箱。

## 4. 凭证管理

### 4.1 Secret 分类

| 类型 | 例子 | 存储位置 |
|------|------|----------|
| **LLM API Key** | Anthropic / OpenAI / Gemini | `ai_plan.api_key` 字段（列级加密）|
| **Git Access Token** | GitHub PAT / Gitlab Deploy Token | `mvp_config` secret 段 |
| **数据库密码** | MySQL / Redis | 环境变量 / 外部 Secret Manager |
| **JWT 签名密钥** | 用户认证 | `mvp_config` secret 段 |
| **服务间 mTLS 证书** | sidecar ↔ 主进程 | 文件 + 文件系统 ACL |
| **Webhook 签名密钥** | GitHub / Slack 回调 | `mvp_config` secret 段 |

### 4.2 存储 MUST

1. 任何 secret **MUST NOT** 以明文落盘到项目目录、worktree、git 仓库
2. 任何 secret **MUST NOT** 出现在 `.env` 文件被 commit 进 git
3. DB 里的 secret 字段 **MUST** 至少做**列级加密**（AES-256-GCM + master key 外置）
4. master key **MUST** 来自环境变量或外部 KMS，**MUST NOT** 写在代码 / config / DB
5. secret 热更新 **MUST** 通过主进程统一接口，**MUST NOT** 重启生效
6. 内存中的 secret **SHOULD** 使用 `mlock` 防 swap，离开作用域 **SHOULD** 显式置零

### 4.3 Vault 接口

主进程 **MUST** 内置一个 `Vault` 接口，是所有 secret 的**唯一出口**：

```go
type Vault interface {
    Get(ctx context.Context, key SecretRef) (Secret, error)
    Rotate(ctx context.Context, key SecretRef) error
    List(ctx context.Context, tenant string) ([]SecretRef, error)
}

type Secret struct {
    Value       []byte    // 使用后 MUST 置零
    Fingerprint string    // 非敏感 hash，用于日志
    ExpiresAt   time.Time
    Scope       []string  // 授权给哪些大脑
}
```

**MUST 条款**：

- 任何业务代码 **MUST NOT** 从 `os.Getenv` / 配置文件直接读 secret，**MUST** 走 Vault
- Vault.Get **MUST** 记审计：谁 / 什么时间 / 什么目的 / 是否成功
- Secret.Value 离开调用栈后 **MUST** 被置零（`runtime.KeepAlive` + `clear(slice)`）
- 日志 **MUST** 只记 `Fingerprint`，**MUST NOT** 记 `Value`

### 4.4 日志脱敏

Logger 层 **MUST** 注册一个 secret redactor：

- 扫描 log record 所有 string 字段
- 匹配常见 secret 模式（`sk-*` / `ghp_*` / `AKIA*` / JWT 三段式 / Bearer 头）
- 命中 → 替换为 `<redacted:sk-***xyz>`
- 匹配 Vault 内所有 Secret 的已知 `Fingerprint` 对应的 `Value` 前缀 → 替换

redactor 匹配失败不是致命错，但 **MUST** 递增 `log_redaction_miss_total` 指标供审计复查。

## 5. LLMAccess 三模式的凭证策略

（本节是 02 §12.5.7 的安全侧细化）

### 5.1 三模式回顾

| 模式 | sidecar 能否直连 LLM | 凭证流向 | 记账归属 |
|------|---------------------|----------|----------|
| `proxied`（默认）| **否** | sidecar 不拿 key，所有 llm.complete 反向 RPC 经主进程 | 主进程统一记账 |
| `direct` | **是** | 主进程 `initialize` 时下发临时 key | sidecar 主动 trace.emit 上报 usage |
| `hybrid` | **按需** | 主进程仅在 sidecar 显式请求时下发短期 key | 混合 |

### 5.2 每种模式的安全约束

#### proxied

- **适用**：Zone 0 / Zone 1 / Zone 2 默认都用这个
- **约束**：
  - sidecar **MUST NOT** 在 `initialize` 接收任何 LLM 凭证字段
  - 主进程 **MUST** 拒绝 `initialize.llm_credentials` 字段出现在 proxied 模式
  - sidecar 所有 LLM 调用 **MUST** 走 `llm.complete` / `llm.stream`
  - 主进程是**唯一的 LLM API 客户端**，唯一的 token bucket 持有者
- **优点**：零凭证泄漏风险 + 完美成本可控
- **缺点**：每次 LLM 调用多一次 IPC，延迟 +1ms

#### direct

- **适用**：Zone 1 内置大脑的高频任务（Prompt Cache 可直连以减少转发）
- **约束**：
  - 主进程 `initialize.llm_credentials` **MUST** 下发的是**短期派生密钥**，不是 master key
  - 短期 key 有效期 **MUST** ≤ Run 剩余时间 + 5 分钟 grace
  - sidecar 拿到 key 后 **MUST** 在内存中持有，**MUST NOT** 落盘
  - sidecar **MUST** 在每次 LLM 调用后 `trace.emit(llm_usage_reported)`，缺漏 **MUST** 触发熔断
  - sidecar 进程退出时主进程 **MUST** 立即吊销该短期 key
  - Zone 2 大脑 **MUST NOT** 使用 direct
- **短期派生密钥方案**：若 provider 支持（Anthropic workspace scoped key），主进程在主 key 下派生 run-scoped sub-key；若不支持（OpenAI），主进程必须**代理**而不是下发，即降级到 proxied

#### hybrid

- **适用**：极少数对延迟敏感的 Zone 1 大脑
- **约束**：
  - sidecar 默认用 proxied，需要 direct 时发 `llm.requestDirectAccess` 反向 RPC
  - 主进程 **MUST** 按 policy 决定是否放行
  - 放行后下发的 key 仅对**单一 Run** 有效
  - 每次申请 **MUST** 审计

### 5.3 Zone 2 的强制 proxied

Zone 2 第三方大脑 **MUST NOT** 在任何情况下使用 direct 或 hybrid。CI 静态检查（§11）会扫描 `BrainDescription.yaml` 里的 `llm_access` 字段，Zone 2 注册时若不是 `proxied` → 构建失败。

## 6. 工具级沙箱策略

### 6.1 Guardrail 两层白名单（规格化）

[02 §6.2](./02-BrainKernel设计.md#62-guardrail-两层白名单) 定义了两层白名单概念。本节给出规格化字段。

```yaml
# BrainDescription.guardrail
guardrail:
  # L1: 工具注册白名单
  allowed_tools:
    - code.read_file
    - code.write_file
    - code.run_tests
    - code.run_cmd
    - code.git_commit

  # L2: 参数级沙箱策略
  sandbox_policy:
    fs:
      write_allow:   ["src/**", "tests/**"]
      write_deny:    ["src/migrations/**", ".gitignore", "admin-go/manifest/sql/**"]
      read_allow:    ["**"]
      read_deny:     [".env*", "~/.ssh/**", "~/.aws/**"]
    net:
      allow_domains: ["pypi.org", "registry.npmjs.org", "github.com"]
      deny_ips:      ["169.254.169.254", "127.0.0.0/8", "10.0.0.0/8"]
    proc:
      allow_binaries: ["git", "node", "npm", "pnpm", "pytest"]
      deny_binaries:  ["*"]   # 默认拒绝剩余
    env:
      allow_keys: ["PATH", "HOME", "LANG", "GOFLAGS"]
      # 所有其他环境变量 MUST 被清除
```

**MUST 条款**：

1. 任何未显式 allow 的资源 **MUST** 默认拒绝
2. deny 规则优先级 **MUST** 高于 allow（显式拒绝不可覆盖）
3. glob 匹配 **MUST** 使用 `doublestar` 语义，`**` 可跨目录
4. 路径匹配 **MUST** 先规范化（`filepath.Clean`）再比对，防 `../` 绕过
5. 符号链接 **MUST** 按"最终解析路径"而非"表面路径"匹配

### 6.2 Resource Lock 安全语义

除了白名单，本节补充 [22 §7.2](./22-Agent-Loop规格.md#72-批次与并发) 的 resource lock 的安全面：

- lock 的 key **MUST** 是规范化的绝对路径
- lock 的 scope **MUST** 是 Run，Run 结束后主进程 **MUST** 释放该 Run 持有的所有 lock
- lock 饥饿检测：单个 Run 持有 lock > 10 分钟 **MUST** 触发告警
- 死锁检测：检测 Run A 等 Run B 的 lock 且 B 等 A → fail 持有时间更长的 Run

### 6.3 Shell 注入防御

`code.run_cmd` 类工具接受字符串命令。这是高风险界面。规格要求：

- **禁止** sidecar 把 LLM 输出的原始字符串直接拼接成 shell 命令。Runner 必须用 **参数数组**形式 exec（`exec.Command(bin, args...)`）
- **禁止** 主进程使用 `sh -c` / `bash -c` 作为工具调度入口
- 工具 schema **MUST** 用数组参数而非单字符串（LLM 输出结构化工具参数）
- 若必须接受 shell 字符串，**MUST** 经过一个"命令行 AST 解析 + 白名单比对"层

## 7. 合规拒绝路径

### 7.1 合规拒绝的触发场景

| 场景 | 具体触发 |
|------|----------|
| LLM safety refusal | Anthropic / OpenAI / Gemini 返回 safety / content_filter / refusal |
| 沙箱拒绝 | Guardrail L2 命中 deny |
| 策略闸门拒绝 | `mvp_policy_rule` 判定 |
| 风险闸门拒绝 | `mvp_risk_gate_rule` 判定（例如"生产环境禁止 DROP TABLE"）|
| 凭证策略拒绝 | Zone 2 大脑请求 direct LLM access |

### 7.2 统一拒绝流程

所有合规拒绝 **MUST** 走同一个流程（对齐 [21 §10 SafetyRefused 升级通道](./21-错误模型.md#10-安全拒绝升级通道)）：

```
  Runner 检测到合规拒绝
       │
       ▼
  抛 BrainError{Class=SafetyRefused, error_code=<具体>}
       │
       ▼
  Decide() → AskHuman
       │
       ▼
  WorkflowRun 状态 → paused
       │
       ▼
  写 mvp_human_checkpoint{status=pending}
       │
       ▼
  写审计日志 oplog{event=safety_refused_blocked}
       │
       ▼
  UI 展示"需要人工审核"面板
       │
       ▼
  人工三选一：继续放行 / 终止任务 / 修改后重试
       │
       ▼
  人工决策 MUST 写 oplog
```

### 7.3 合规拒绝绝不绕过

**MUST NOT** 条款：

- **MUST NOT** 把 `SafetyRefused` 自动改写成其他 Class
- **MUST NOT** 在 Decide 里对 `SafetyRefused` 返回 Retry
- **MUST NOT** 允许 sidecar 调 `brain.overrideSafetyRefused` 之类的 API（此类 API 不存在）
- **MUST NOT** 人工 override 无审计

### 7.4 人工放行后的责任

一旦人工放行 SafetyRefused 任务：

- oplog 记录的 `actor` 字段 **MUST** 是真实用户 id，不能是 `system` / `anonymous`
- 放行决定 **MUST** 带理由（UI 必填字段）
- 该任务后续产生的所有副作用 **MUST** 在 audit trail 里关联这次放行决定
- 如果任务随后引发事故，audit trail 能回溯到这次放行

## 8. 审计事件规格

### 8.1 审计事件的三性

审计事件 **MUST** 满足：

- **完整性**（Integrity）：一旦写入不可改（append-only + hash chain）
- **不可否认性**（Non-repudiation）：每条事件带 actor id 和时间戳
- **可追溯性**（Traceability）：同一 Run 的所有事件可串成一条链

### 8.2 存储

审计事件 **MUST** 落入 `oplog` 表（复用 EasyMVP 现有审计设施）。oplog 记录 **MUST**：

- 包含 prev_hash（指向上一条事件的 SHA-256），形成 hash chain
- prev_hash 校验失败 → 触发 P0 告警（可能被篡改）
- 定期把 oplog 的最新 chain tip hash 写到外部不可变存储（例如每天一条 WORM 存储或加时间戳的外部系统）

### 8.3 必记事件（v1）

完整清单见[附录 A](#附录-a--审计事件类型清单)。以下是**不可省略**的事件类别：

- LLM 凭证下发 / 吊销 / 轮换
- 任何 SafetyRefused 触发
- 任何 Zone 2 大脑加载 / 卸载
- 任何人工 override（放行 / 终止）
- 任何 Vault.Get 失败
- 任何 fault_policy 运行时修改
- 任何 sidecar 被 quarantine
- 任何 oplog hash chain 异常

### 8.4 事件格式

```json
{
  "id": "<snowflake>",
  "timestamp": "2026-04-11T08:23:11.123Z",
  "event_type": "safety_refused_blocked",
  "actor": {
    "type": "user" | "system" | "brain",
    "id": "<user_id or brain_id>",
    "session_id": "<jwt_jti>"
  },
  "subject": {
    "type": "run" | "brain" | "tool_call" | "config",
    "id": "<id>"
  },
  "context": {
    "run_id": 123,
    "workflow_run_id": 456,
    "trace_id": "01HQ...",
    "dept_id": 789,
    "tenant": "..."
  },
  "payload": { /* 事件特定字段 */ },
  "severity": "info" | "warn" | "error" | "critical",
  "prev_hash": "<sha256 of previous oplog row>",
  "hash": "<sha256 of this row without hash field>"
}
```

## 9. PII 与敏感数据处理

### 9.1 PII 识别

Runner **MUST** 在以下数据流通过 PII 扫描：

- 进入 prompt 的 user messages
- tool_result 的 text 部分
- 写入 artifact 的内容
- 写入审计日志的 payload

PII 类型（v1 基本集）：

- 邮箱地址
- 手机号（国际 + 中国常见格式）
- 身份证号（中国 18 位）
- 信用卡号（Luhn 校验）
- IPv4 / IPv6 地址
- JWT / API key 模式

### 9.2 处理策略

| 位置 | 策略 |
|------|------|
| 进 prompt | 默认**不过滤**（用户可能需要让 LLM 帮忙处理自己的数据），但 **MUST** 在审计日志打标 |
| tool_result 进 LLM | 默认**不过滤**（和进 prompt 一样，但走 §10 sanitizer） |
| 进 artifact 存储 | **MAY** 按 tenant 配置掩码 |
| 进日志 / metrics | **MUST** 掩码（邮箱 → `u***@example.com`，手机 → `138****1234`） |
| 进 metrics label | **MUST NOT** 使用原始 PII，只能用 hash |

### 9.3 数据留存

- Run 结束后，`BrainPlan` / `artifact` 默认保留 **90 天**
- 90 天后 **MUST** 归档或匿名化
- 用户请求删除 **MUST** 级联删除所有副本（artifact / 审计 payload 字段 / trace）
- 但 `oplog` 本体 **MUST NOT** 被级联删除（审计不可篡改）。删除请求本身也是一条 oplog 事件

## 10. Supply Chain 与 sidecar 签名

### 10.1 Binary 来源约束

Zone 1 built-in sidecar：

- **MUST** 由项目 CI 构建
- **MUST** 作为主进程 binary 的一部分随版本发布（or 同源仓库）
- **MUST** 在启动时校验自己的 SHA-256 与 release manifest 一致

Zone 2 third-party sidecar：

- **MUST** 携带有效的**签名**（Sigstore / GPG / Anthropic signed manifest）
- **MUST** 在 `brain.describe` 返回 `signature_chain`
- 主进程 **MUST** 验证签名链，任一节点失败 → 拒绝加载
- **SHOULD** 发布在官方 registry，主进程按 checksum pinning 加载
- **MUST** 在启动 sidecar 前运行 static scanner（可选：trivy / snyk）

### 10.2 依赖来源

sidecar 的运行时依赖（Python packages / npm modules）**MUST** 走受管理的内部镜像：

- **MUST NOT** 在 sidecar 运行时直接 `pip install` 从 pypi
- **MUST** 预装到镜像，镜像本身由 CI 构建 + 签名

### 10.3 自更新禁止

sidecar 运行过程中 **MUST NOT**：

- 自己 `pip install` 新包
- 从网络下载新 binary 并 exec
- 修改自己的代码

任何上述行为命中 → P0 告警 + 立即 quarantine。

## 12. 合规性测试矩阵

| 编号 | 名称 | 验证点 |
|------|------|--------|
| C-S-01 | Worktree 逃逸 | 通过 `../../etc/passwd` 读取宿主敏感文件 → 必须拒绝 |
| C-S-02 | 符号链接逃逸 | worktree 内创建指向 `/etc/shadow` 的 symlink → 读时被规范化后拒绝 |
| C-S-03 | 元数据服务 SSRF | tool 尝试 `curl http://169.254.169.254/` → 被 NET 拒绝 |

| C-S-08 | Shell 注入阻断 | tool 参数带 `; rm -rf /` → exec 用数组形式，不被解释为 shell |
| C-S-09 | oplog hash chain | 伪造一条 oplog → hash 校验失败 → P0 告警 |
| C-S-10 | SafetyRefused 不可绕过 | Runner 尝试把 SafetyRefused 改写 Class → 违规计数 + 拒绝 |
| C-S-11 | 合规拒绝后 paused | Guardrail L2 拒绝后 WorkflowRun 状态 = paused |
| C-S-12 | 人工放行审计 | 无理由放行被拒；有理由放行写 oplog |
| C-S-13 | sidecar 签名验证 | 伪造 signature → 加载失败 |
| C-S-14 | sidecar 自更新阻断 | sidecar 尝试 `pip install` → 沙箱拒绝 + P0 |
| C-S-15 | PII 日志掩码 | 日志 `user 13812345678 login` → `user 138****5678 login` |

| C-S-17 | Zone 边界错误不泄露 internal | HTTP body 检查无 stack / internal_only |
| C-S-18 | 跨租户隔离 | tenant A 的 Run 不能读 tenant B 的 worktree |
| C-S-19 | sandbox 资源耗尽 | sidecar fork bomb → cgroup 限制触发 |
| C-S-20 | 凭证 rotation 无重启 | Vault.Rotate 后 in-flight 请求仍成功使用旧 key，新请求自动使用新 key |
