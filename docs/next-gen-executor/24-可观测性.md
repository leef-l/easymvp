# 24 · BrainKernel 可观测性 v1

> **定位**：本文档是 BrainKernel 的**可观测性正式规格**，定义主进程、sidecar、Agent Loop、工具调用、审计事件在 **Metrics / Traces / Logs** 三个信号维度上的契约。是 [02-BrainKernel设计.md §11](./02-BrainKernel设计.md) Trace/Audit Logger 的下位规格，同时承接 [21-错误模型 §13](./21-错误模型.md#13-审计与观测约束) 和 [22-Agent-Loop规格 §3.4 / §4.5](./22-Agent-Loop规格.md) 的指标命名约束。
>
> **规格性质**：本文档使用 RFC 2119 关键词。指标名称、trace attribute 命名、log 字段名 **MUST** 严格对齐本规格——任何"随手新增的指标名"都会让 Grafana / Sentry / Loki 里的仪表盘失配，导致事故时找不到关键信号。
>
> **版本**：可观测性规格版本 `v1`。v1 冻结后 metric 名称、label 集合、trace attribute key、log 字段 key **MUST NOT** 改名 / 删除 / 改语义，只能新增。

## 1. 为什么要把可观测性独立成规格

Agent 系统出事故的时候，"找不到现场"是最痛的一件事。三种典型故障模式：

- **静默失败**：sidecar 还在心跳、metric 还在出数，但 Run 实际卡在某个 tool 上 40 分钟没动。没有 per-step 的 trace span → 看不到卡哪
- **成本爆炸**：某个 Run 烧掉 $30。事后复盘想知道是哪个 Turn 哪个 prompt 把 cache 打碎了。没有 prompt_cache_hit_ratio 的 per-turn 记录 → 只能凭猜
- **指标名漂移**：v1.0 的指标叫 `agent_step_total`，v1.2 改名 `loop_turn_count`，仪表盘全废。Alertmanager 规则静默失效三周没人发现

本规格把命名、维度、语义冻结在 v1，让后续两年都能以同一套 Grafana dashboard + 同一套 SLO 运作。

对应 02 的决策追溯锚点：**决策 4**（fault_policy 可配）、**决策 6**（Runner 三方法的心跳接口）、**决策 7**（LLMAccess 记账归属）——这三条决策都依赖完整的可观测性支撑才能真正"按数据决策"。

## 2. 三信号设计原则

### 2.1 Metrics / Traces / Logs 的分工

| 信号 | 本质 | 成本 | 查询方式 | 用途 |
|------|------|------|----------|------|
| **Metrics** | 时间序列 + label | 低 | PromQL / time range | 实时监控、SLO、告警 |
| **Traces** | 树形 span + 事件 | 中 | 按 trace_id 或 attribute 查单次 | 故障定位、性能分析 |
| **Logs** | 自由格式事件 | 高 | 全文 / 字段搜索 | 审计、疑难排查 |

**设计原则**：

1. **Metrics 回答 "有没有坏" 和 "坏了多少"**。不回答"为什么坏"。高基数 label（run_id / trace_id / user_id）**MUST NOT** 进 metric label——否则 Prometheus 会 OOM
2. **Traces 回答 "一次请求具体发生了什么"**。一个 trace 覆盖一个 Run 的完整生命周期，可下钻到单次 tool_call
3. **Logs 回答 "某个事件的完整上下文"**。结构化日志，按 trace_id 可与 Traces 关联
4. **三信号 MUST 通过 `trace_id` 相互关联**。一条告警 → 打开 trace → 查相关日志 → 这是正常的 on-call workflow

### 2.2 黄金指标（RED + USE）

BrainKernel v1 **MUST** 为以下维度同时覆盖 RED + USE 指标：

- **RED**（Rate / Error / Duration）：`brain.run` / `llm.complete` / `tool.invoke` 三个操作
- **USE**（Utilization / Saturation / Errors）：sidecar 进程、LLM bucket、tool worker pool

## 3. OpenTelemetry 集成

### 3.1 为什么用 OTel 而不是 Prometheus 原生

OTel 是 CNCF 的 vendor-neutral 标准，允许后端自由切换（Jaeger / Tempo / DataDog / 自建）。BrainKernel **MUST** 通过 OTel SDK 发射所有信号。

- **Metrics**：通过 OTel Metrics SDK → OTLP 导出到 Prometheus（经 OTel Collector）或直连 VictoriaMetrics
- **Traces**：通过 OTel Traces SDK → OTLP 导出到 Tempo / Jaeger
- **Logs**：通过 OTel Logs SDK → OTLP 导出到 Loki / Elasticsearch

### 3.2 Resource 属性

每个 OTel signal **MUST** 携带以下 `Resource` 属性：

```
service.name         = "brainkernel"
service.version      = "<git sha>"
service.instance.id  = "<hostname>-<pid>"
deployment.env       = "dev" | "staging" | "prod"
brainkernel.tenant   = "<tenant>"
brainkernel.region   = "<region>"
```

### 3.3 Context 传播

Trace Context **MUST** 通过 20 号协议规格的 JSON-RPC 消息传递。具体机制：

- 主进程发起 `brain.run` 时生成 root trace_id 和 span_id
- 写入请求的 `ChatRequest.Metadata.trace_id` 和 `trace_state`
- sidecar 收到后 **MUST** 把 trace context 作为当前 span 的 parent
- sidecar 发起反向 RPC（`tool.invoke` / `llm.complete`）**MUST** 携带 `$trace` 字段，内容是 W3C Trace Context 的 `traceparent` header 编码

### 3.4 OTLP 端点配置

```yaml
# config/otel.yaml
otel:
  exporter: "otlp"
  endpoint: "http://otel-collector:4317"
  protocol: "grpc"
  timeout: "10s"
  headers:
    Authorization: "Bearer ${OTEL_AUTH_TOKEN}"  # MUST 走 Vault
  sample_rate:
    traces: 0.1      # 见 §9 采样
    metrics: 1.0
    logs: 1.0
  batch:
    max_export_batch_size: 512
    export_interval: "5s"
    export_timeout: "30s"
```

## 4. Metrics 清单

（完整清单见[附录 A](#附录-a--metric-完整清单)，本节列出最关键的一部分作为心智模型）

### 4.1 命名规范（冻结）

- **MUST** 使用 `snake_case`
- **MUST** 以 `brain_` 开头（与业务 metric 区分）
- **MUST** 按 OTel metric 类型后缀：counter 用 `_total`，histogram 用 `_seconds` / `_bytes`
- **MUST NOT** 在 metric name 里编码 label 值（不要 `brain_run_total_code_brain_success`，用 labels）
- **MUST NOT** 出现高基数 label（run_id / user_id / tool_arg_hash）

### 4.2 必记 metric 家族

| metric 家族 | 类型 | labels | 关键作用 |
|-------------|------|--------|----------|
| `brain_run_total` | counter | `brain_id, result` | 一个 Run 的结果分布（completed/failed/cancelled）|
| `brain_run_duration_seconds` | histogram | `brain_id, result` | Run 挂钟时长 |
| `brain_turn_total` | counter | `brain_id` | Turn 数量（Run 内平均 Turn 数可衍生）|
| `brain_turn_duration_seconds` | histogram | `brain_id` | 单 Turn 时长（LLM 延迟 + tool 批次）|
| `llm_call_total` | counter | `provider, model, result` | LLM 调用次数 |
| `llm_call_duration_seconds` | histogram | `provider, model` | LLM 调用时长 |
| `llm_tokens_total` | counter | `provider, model, direction` | token 用量（direction ∈ input / output / cache_read / cache_write）|
| `llm_cost_usd_total` | counter | `provider, model, brain_id` | 成本累计 |
| `prompt_cache_hit_ratio` | gauge | `provider, model, layer` | 对齐 [22 §3.4 SLO](./22-Agent-Loop规格.md#34-cache-命中率-slo) |
| `prompt_cache_break_total` | counter | `layer, reason` | Cache 被打破 |
| `tool_call_total` | counter | `tool_name, result` | 工具调用次数 |
| `tool_call_duration_seconds` | histogram | `tool_name, result` | 工具时长 |
| `tool_timeout_total` | counter | `tool_name` | 超时次数（对齐 [22 §8](./22-Agent-Loop规格.md#8-单工具超时与预算)）|
| `brain_error_total` | counter | `brain_id, class, error_code, retryable` | 对齐 [21 §13.1](./21-错误模型.md#131-metrics必须) |
| `sidecar_health_state` | gauge | `brain_id, state` | 0/1，对应 healthy/degraded/quarantined |
| `sidecar_heartbeat_age_seconds` | gauge | `brain_id` | 距最近一次 `health.ping` 的秒数 |
| `sidecar_process_count` | gauge | `brain_id, state` | 同一大脑的 sidecar 实例数 |
| `rate_limited_total` | counter | `provider, model` | 上游 429 次数 |
| `waiting_queue_depth` | gauge | `provider, model` | 等待上游的请求队列深度 |
| `streaming_backpressure_total` | counter | `brain_id` | Sidecar 反向背压次数 |
| `reflection_budget_exhausted_total` | counter | `brain_id` | 反思预算耗尽次数 |
| `poison_message_detected_total` | counter | `brain_id` | 毒消息检测 |
| `errmodel_violation_total` | counter | `brain_id, violation_type` | 错误模型合规违规 |
| `audit_event_total` | counter | `event_type, severity` | 审计事件总数 |
| `oplog_hash_chain_broken_total` | counter | - | hash chain 破坏（必须恒为 0）|

### 4.3 禁止出现的反模式

| 反模式 | 为什么 |
|--------|--------|
| `label run_id` | 每 Run 不同，时间序列无限增长 |
| `label user_id` | 同上 + PII 风险 |
| `label trace_id` | 同上 |
| `label error_message` | 变化多，基数爆炸 |
| `metric 名拼 label 值` | 破坏命名一致性 |
| counter 被 decrement | OTel counter 单调递增 |
| gauge 用作 counter | 重启丢数 |

### 4.4 Cardinality 预算

每个 metric 家族的**总 label 组合数** **MUST** ≤ **5000**。超出 → 静态检查阻断。估算规则：

```
cardinality = ∏ |labelᵢ|
```

例如 `brain_error_total{brain_id, class, error_code, retryable}`：

- brain_id 上限 20
- class 6
- error_code ~40
- retryable 2

总 = 20 × 6 × 40 × 2 = 9600 → 超限 → **MUST** 拆掉 `error_code` 或做聚合

v1 选择去掉 `retryable`（retryable 与 error_code 强相关，查 error_code 就能推出 retryable），改为 4800。

## 5. Traces 规格

### 5.1 Span 层级

一个 `brain.run` 产生的 trace 树：

```
brain.run (root span)
├── initialize
│   └── sidecar.handshake
├── agent_loop.turn [0]
│   ├── llm.complete
│   │   ├── provider.api_call
│   │   └── prompt_cache.check
│   └── tool_batch [0]
│       ├── tool.invoke [code.read_file]
│       ├── tool.invoke [code.git_diff]
│       └── guardrail.check
├── agent_loop.turn [1]
│   ├── llm.complete
│   └── tool_batch [1]
│       └── tool.invoke [code.run_tests]
│           └── sandbox.exec
└── run.finalize
    ├── artifact.flush
    └── audit.emit
```

### 5.2 Span MUST 属性

每个 span **MUST** 携带：

- `brainkernel.run_id`
- `brainkernel.brain_id`
- `brainkernel.turn_index`（agent_loop.turn 以下的 span）
- `brainkernel.component`（kernel / runner / provider / tool / sidecar）

agent_loop.turn span **MUST** 额外携带：

- `llm.provider`
- `llm.model`
- `llm.input_tokens`
- `llm.output_tokens`
- `llm.cache_read_tokens`
- `llm.cache_creation_tokens`
- `llm.stop_reason`
- `llm.cost_usd`

tool.invoke span **MUST** 携带：

- `tool.name`
- `tool.input_size_bytes`
- `tool.output_size_bytes`
- `tool.is_error`
- `tool.timeout_ms`
- `tool.duration_ms`

出错的 span **MUST** 遵循 OTel semantic conventions：

- `span.status_code = ERROR`
- `span.status_description = <BrainError.message>`
- `exception.type = BrainError`
- `exception.message = <message>`
- `exception.stacktrace = <internal_only.stack_trace>`（仅后端可见）
- 加事件 `exception` 时间点对齐错误发生时刻

### 5.3 Span Event

以下时间点 **MUST** 作为 span event 记录：

- `llm.call.start` / `llm.call.end`
- `prompt_cache.hit` / `prompt_cache.miss`
- `tool_result.truncated`（tool_result 被 pruning）
- `tool_result.sanitized`（prompt injection 命中）
- `reflection.attempted` / `reflection.accepted`
- `loop.suspected` / `loop.confirmed`（死循环检测）
- `guardrail.denied`
- `sidecar.restarted`

### 5.4 Trace 命名

span 名 **MUST** 是"动词 + 资源"形式，稳定且不含动态值。例子：

- ✅ `llm.complete` / `tool.invoke` / `brain.run`
- ❌ `llm.complete claude-opus-4-6`（provider/model 放属性不放名字）
- ❌ `brain.run #123`（id 放属性）

## 6. Logs 规格

### 6.1 结构化日志

所有日志 **MUST** 是 JSON 行格式。Go 端使用 `slog` + OTel bridge。字段约定：

```json
{
  "timestamp": "2026-04-11T08:23:11.123Z",
  "level": "info",
  "msg": "llm call completed",
  "component": "agent_loop",
  "brain_id": "code_brain",
  "run_id": 123,
  "turn_index": 5,
  "trace_id": "01HQ...",
  "span_id": "5e2b...",
  "llm": {
    "provider": "anthropic",
    "model": "claude-opus-4-6",
    "input_tokens": 4532,
    "output_tokens": 812,
    "duration_ms": 1830,
    "cost_usd": 0.0234
  }
}
```

### 6.2 日志级别语义

| level | 语义 |
|-------|------|
| `debug` | 开发态细节；生产环境默认关闭；仅在排障时临时打开 |
| `info` | 正常业务事件（Run 开始 / 结束、LLM 调用完成、tool 完成）|
| `warn` | 预期内的异常（Transient 错误、重试、降级）|
| `error` | 预期外的失败（Permanent / InternalBug）|
| `critical` | 需要立即人工介入（SafetyRefused、sidecar panic、oplog 破损）|

### 6.3 必记日志事件（v1）

| 事件 | level |
|------|-------|
| Run 开始 / 结束 | info |
| 每个 Turn 的 LLM 调用完成 | info |
| 每次 tool.invoke 完成 | info |
| Transient 错误（采样 1%） | warn |
| Permanent 错误（全量）| error |
| SafetyRefused（全量） | critical |
| sidecar 重启 | warn |
| sidecar quarantine | error |
| 审计事件（全量）| info+ |
| Prompt Cache 被打破 | warn |
| reflection 预算耗尽 | warn |
| 死循环检测触发 | error |
| 毒消息检测 | critical |
| oplog hash chain 异常 | critical |

### 6.4 日志脱敏

Logger **MUST** 内置 [23 §4.4 secret redactor](./23-安全模型.md#44-日志脱敏)，所有写出的 string 字段都经过扫描和替换。

Log 字段 **MUST NOT** 包含：

- LLM 凭证
- 用户密码 / JWT token
- `BrainError.InternalOnly.RawStderr` 中未过 sanitizer 的原文（可进 trace 的 exception attribute，但不进 log）
- PII 原文（按 [23 §9](./23-安全模型.md#9-pii-与敏感数据处理) 掩码）

## 7. SLO 与告警矩阵

### 7.1 SLO 目标

| 指标 | 目标 | 观察窗口 |
|------|------|----------|
| `brain.run` 成功率 | ≥ 95% | 24h |
| `brain.run` P95 时长 | ≤ 10 分钟 | 1h |
| `llm.complete` P99 延迟 | ≤ 30s | 1h |
| `tool.invoke` 成功率 | ≥ 99% | 1h |
| `prompt_cache_hit_ratio{layer=L1}` | ≥ 95% | 1h |
| `prompt_cache_hit_ratio{layer=L2}` | ≥ 90% | 1h |
| `sidecar_heartbeat_age_seconds` | ≤ 60s | 实时 |
| `errmodel_violation_total` | 0 | 任意 |
| `poison_message_detected_total` | 0 | 24h |
| `oplog_hash_chain_broken_total` | 0 | 任意 |

### 7.2 告警矩阵

| 告警 ID | 级别 | 条件 | 处置 |
|--------|------|------|------|
| AL-01 | P0 | `oplog_hash_chain_broken_total > 0` | 立即冻结写入、启动取证 |
| AL-02 | P0 | `poison_message_detected_total > 0` 持续 1 min | 停止相关 Run |
| AL-03 | P0 | `sandbox_escape_attempt` 审计事件 | 隔离 sidecar |
| AL-04 | P1 | `brain_error_total{class="InternalBug"}` 5min ≥ 3 | 查日志定位 |
| AL-05 | P1 | `sidecar_health_state{state="quarantined"} > 0` | 人工评估 |
| AL-06 | P1 | `brain.run` 成功率 10 min < 80% | 查批量失败原因 |
| AL-07 | P2 | `prompt_cache_break_total{layer="L1"} > 0` 1h | 排查 L1 注入 |
| AL-08 | P2 | `errmodel_violation_total` 5 min ≥ 5 | 查违规 sidecar |
| AL-09 | P2 | `llm_cost_usd_total` 日增 > $50 per brain | 查异常成本 |
| AL-10 | P2 | `tool_timeout_total{tool_name="code.run_tests"}` 1h > 20 | 查测试拖慢 |
| AL-11 | P2 | `rate_limited_total` 5 min > 50 | 查 provider 压力 |
| AL-12 | P3 | `reflection_budget_exhausted_total` 1h > 10 | 审查 prompt 设计 |
| AL-13 | P3 | `sidecar_heartbeat_age_seconds > 60s` 持续 2 min | sidecar 异常 |
| AL-14 | P3 | `streaming_backpressure_total` 1h > 50 | sidecar 处理慢 |
| AL-15 | P3 | `SafetyRefused` 24 h ≥ 10 | 检查是否策略误伤 |

### 7.3 Error Budget 策略

以 `brain.run` 成功率为例，SLO = 95% → 错误预算 = 5%：

- 周错误预算 = 5% × 周 Run 数
- 错误预算用完 → **冻结新功能上线**，优先修复故障
- 预算恢复 → 解除冻结

## 9. 采样策略

### 9.1 Trace 采样

全量 trace 成本太高。策略：

- **错误全采**：任何产生 `span.status = ERROR` 的 trace **MUST** 被强制采样（`ParentBased(TraceIdRatio(0.1))` + `always_on_for_errors`）
- **慢请求全采**：Run duration > P99 的 **MUST** 强制采样
- **关键大脑全采**：`central_brain` / `verifier_brain` 的 trace 全采
- **其他**：按 10% 随机采样

### 9.2 Log 采样

- `debug` / `info` 级别：默认 10% 采样
- `warn` / `error` / `critical`：100% 采样
- `Transient` 错误：1% 采样（量大不重要）
- `Permanent` / `UserFault` / `SafetyRefused` / `InternalBug`：100% 采样

### 9.3 Metric 采样

Metrics **MUST NOT** 采样——时间序列需要完整窗口才能做聚合。为控制基数走 §4.4 cardinality 预算而不是采样。

## 10. 成本与性能预算

### 10.1 可观测性自身的成本上限

| 信号 | 上限 |
|------|------|
| Metric 时间序列总数 | ≤ 200k |
| Trace 日导出量 | ≤ 20 GB |
| Log 日导出量 | ≤ 50 GB |
| OTel exporter CPU | ≤ 主进程 5% |
| OTel exporter 内存 | ≤ 512 MB |

### 10.2 性能约束

- span 创建延迟 **MUST** ≤ 10 µs
- metric record 延迟 **MUST** ≤ 5 µs
- log 写入不阻塞主路径（异步写 + batch flush）
- OTel Exporter crash **MUST NOT** 影响主进程业务

### 10.3 降级策略

当 OTel exporter 出问题（connection refused / 队列打满）时：

- Metric：本地 ring buffer 缓 1 min，丢弃最旧
- Trace：丢弃（按 9.1 采样已经抽稀）
- Log：写入本地文件作为 fallback，同时告警
