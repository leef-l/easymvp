# 26 · BrainKernel 持久化与恢复 v1

> **定位**：本文档是 BrainKernel 的**持久化层正式规格**，定义 BrainPlan、Artifact、Run 状态、审计日志如何落盘、如何保证原子性、如何在主进程崩溃或 sidecar 丢失后恢复。是 [02-BrainKernel设计.md §7 (ArtifactStore) / §8 (PlanStore)](./02-BrainKernel设计.md) 的下位规格，同时与 [23-安全模型.md §8 (oplog hash chain)](./23-安全模型.md#8-审计事件规格) 和 [24-可观测性.md](./24-可观测性.md) 配合。
>
> **规格性质**：本文档使用 RFC 2119 关键词。持久化是"救命层"——所有其他层出错都可以靠它恢复，但它自己出错就再也救不回来。任何偏离 MUST 条款都可能导致数据丢失或损坏。
>
> **版本**：持久化规格版本 `v1`。v1 冻结后 schema、文件格式、CAS key 算法、Resume 协议 **MUST NOT** 破坏性修改；只能加字段、加表、加格式版本号。

## 1. 为什么要独立一份持久化规格

Agent 系统的持久化有三个容易被低估的点：

1. **长 Run 会跨进程**。一个 50 Turn 的 Run 可能跑 2 小时，中途主进程被 SIGKILL、被 deploy、被 OOM——完成率依赖 Resume 能力。没有 Resume = 所有长 Run 失败就重来，成本和体验都崩。

2. **Artifact 会被多次引用**。LLM 看到一个文件的 tool_result 可能跨 10 个 Turn 继续讨论。如果每个 Turn 都把文件原文塞进 message，context 立刻爆炸。只能靠 CAS：存一次、引用多次。CAS 的 key 必须稳定、去重必须可靠——否则磁盘会慢慢爆炸。

3. **独立产品 + 集群产品并存**。02 §7 / §8 明确了 ArtifactStore 和 PlanStore 是**双轨设计**：独立产品版用 SQLite + 本地文件，集群版用 MySQL + 对象存储。双轨实现必须语义等价，否则把"本地跑通的测试"迁到集群会出现看不见的 bug。

本规格把这三个问题的约束写成 MUST 条款。

对应 02 的决策追溯锚点：本规格与决策 4（fault_policy）、决策 5（sidecar 零例外）、决策 7（LLMAccess 记账）都相关——故障处理、sidecar 崩溃恢复、成本数据持久化都走本层。

## 2. 持久化对象清单

v1 需要持久化的对象按生命周期分三类：

### 2.1 长生命周期（Run 生命周期之外存在）

| 对象 | 存储 | 用途 |
|------|------|------|
| `BrainPlan` | SQLite/MySQL | 大脑的中央大脑规划（类似 TODO list，但含决策理由） |
| `Artifact`（文件 / 二进制）| CAS 存储 | 工具输出、LLM 生成的文件 |
| `oplog` 审计事件 | MySQL | 审计不可篡改 hash chain |
| `mvp_workflow_event` | MySQL | 工作流事件流 |
| `mvp_config` | MySQL | 配置与灰度开关 |

### 2.2 中生命周期（Run 持续期间）

| 对象 | 存储 | 用途 |
|------|------|------|
| `Run Checkpoint` | SQLite/MySQL | Turn 边界的状态快照 |
| `Messages`（L3 history）| SQLite/MySQL + 溢出到 artifact | 对话历史，长了会溢出 |
| `Tool Call 记录` | SQLite/MySQL | 每次工具调用的完整 request/response |
| `Usage / Cost` | SQLite/MySQL | 按 Turn 粒度的 token 与成本记账 |

### 2.3 短生命周期（Run 结束即丢弃）

| 对象 | 存储 | 用途 |
|------|------|------|
| `Prompt Cache 本地索引` | 主进程内存 | L1/L2 hash → cache handle |
| `Token Bucket 状态` | 主进程内存 | Rate limit |
| `Inflight RPC 窗口` | 主进程内存 | 协议层 |
| `Streaming 缓冲` | 主进程内存 | SSE 透传 |

短生命周期对象 **MUST NOT** 落盘（写磁盘成本 >> 重算成本）。

## 3. 双轨存储架构

### 3.1 两条轨道

```
┌───────────────────────────────────────────────────────────┐
│ 独立产品版本（Solo mode）                                    │
│   BrainPlan:  SQLite WAL, ~/.easymvp/brain.db              │
│   Artifact:   CAS on local fs, ~/.easymvp/artifacts/       │
│   oplog:      SQLite WAL, ~/.easymvp/oplog.db              │
│ 目标：单机跑一切，零外部依赖                                  │
└───────────────────────────────────────────────────────────┘
                           ↕ (同接口)
┌───────────────────────────────────────────────────────────┐
│ 集群版本（Cluster mode）                                    │
│   BrainPlan:  MySQL `mvp_brain_plan` 表                    │
│   Artifact:   S3/MinIO CAS bucket                          │
│   oplog:      MySQL `oplog` 表（hash chain）                │
│ 目标：水平扩展 + HA                                          │
└───────────────────────────────────────────────────────────┘
```

### 3.2 接口统一

两条轨道 **MUST** 实现相同的 Go 接口：

```go
type PlanStore interface {
    Create(ctx context.Context, plan *BrainPlan) (PlanID, error)
    Get(ctx context.Context, id PlanID) (*BrainPlan, error)
    Append(ctx context.Context, id PlanID, delta PlanDelta) (PlanVersion, error)
    GetVersion(ctx context.Context, id PlanID, version PlanVersion) (*BrainPlan, error)
    List(ctx context.Context, filter PlanFilter) ([]PlanSummary, error)
}

type ArtifactStore interface {
    Put(ctx context.Context, content io.Reader, meta ArtifactMeta) (ArtifactRef, error)
    Get(ctx context.Context, ref ArtifactRef) (io.ReadCloser, *ArtifactMeta, error)
    Exists(ctx context.Context, ref ArtifactRef) (bool, error)
    List(ctx context.Context, filter ArtifactFilter) ([]ArtifactRef, error)
    Delete(ctx context.Context, ref ArtifactRef) error  // 只用于合规删除请求
}

type RunCheckpointStore interface {
    Save(ctx context.Context, run_id int64, checkpoint *RunCheckpoint) error
    Load(ctx context.Context, run_id int64) (*RunCheckpoint, error)
    ListResumable(ctx context.Context) ([]int64, error)
}
```

### 3.3 语义等价 MUST

- 同一输入在两条轨道上 **MUST** 产生语义等价的输出
- CAS key 算法 **MUST** 两轨一致（都是 sha256 + 规范化）
- Resume 协议 **MUST** 两轨一致
- 合规测试 **MUST** 在两条轨道上都跑

### 3.4 切换时机

- 开发 / 单机部署 → 独立版（默认）
- 多实例 / HA → 集群版
- 切换通过 `mvp_config.storage_mode = solo | cluster`
- **不允许**同一实例同时开两种模式
- 切换 **MUST** 通过一次性迁移脚本（`solo → cluster` 的数据导出/导入）

## 4. SQLite WAL 本地 BrainPlan

### 4.1 为什么 WAL

SQLite 的默认 rollback journal 模式**不允许并发读写**。BrainKernel 主进程可能有多个 goroutine 同时读 BrainPlan（trace logger 读老版本 + runner 写新 delta），默认模式会锁死。

WAL (Write-Ahead Logging) 允许：

- 多个 reader 与一个 writer 并发
- crash 后自动 recover 到最后 checkpoint
- fsync 成本可配

**MUST** 条款：

1. 所有 SQLite 连接 **MUST** 启用 `PRAGMA journal_mode=WAL`
2. **MUST** 启用 `PRAGMA synchronous=NORMAL`（不是 FULL，性能权衡）
3. **MUST NOT** 启用 `synchronous=OFF`（会丢数据）
4. **MUST** 启用 `PRAGMA foreign_keys=ON`
5. **MUST** 设 `PRAGMA busy_timeout=5000`（5 秒）
6. 每 1000 次写触发一次 `PRAGMA wal_checkpoint(TRUNCATE)`

### 4.2 Schema

完整 schema 见[附录 A](#附录-a--sqlite-schema)。核心表：

- `brain_plan` - 主计划表（含 version 字段）
- `brain_plan_delta` - append-only 的增量日志
- `run_checkpoint` - Turn 边界快照
- `usage` - Turn 级 token 记账
- `tool_call_log` - 工具调用完整历史
- `artifact_meta` - CAS artifact 元数据
- `oplog` - 审计事件（独立 SQLite 文件 `oplog.db`）

### 4.3 连接池

SQLite 是单写多读：

- 写连接池：**1 个连接**（写串行）
- 读连接池：**8 个连接**（并发读）
- 读连接 **MUST NOT** 开启事务写
- 写连接 **MUST** 用 `BEGIN IMMEDIATE` 而不是 `BEGIN`（避免死锁）

### 4.4 事务边界

- 一个 Turn 结束时的状态更新 **MUST** 是单个事务（原子性）
- 事务内容：更新 `brain_plan` + append `brain_plan_delta` + 写 `run_checkpoint` + 写 `usage` + 写 `tool_call_log`
- 事务失败 **MUST** 全部回滚
- 事务 **MUST NOT** 跨 Turn

## 5. MySQL 集群版 BrainPlan

### 5.1 表结构增量

在现有 EasyMVP 数据库（55 张表）基础上**新增** 4 张表（详见[附录 B](#附录-b--mysql-表结构增量)）：

- `mvp_brain_plan`（与 SQLite schema 对齐）
- `mvp_brain_plan_delta`
- `mvp_run_checkpoint`
- `mvp_brain_usage`

**MUST 条款**：

1. 所有新表 **MUST** 通过 golang-migrate 迁移文件创建（对齐 [CLAUDE.md 十四铁律](../../CLAUDE.md)）
2. 所有新表 **MUST** 有 `created_at` / `updated_at` / `deleted_at` / `created_by` / `dept_id` 字段（对齐十三铁律）
3. 所有新表 **MUST** 用 `utf8mb4` + `utf8mb4_unicode_ci`
4. `brain_plan` 与 `brain_plan_delta` 之间 **MUST** 有外键
5. **MUST** 有适当 index 支持：`run_id` 查找、`brain_id` 过滤、`created_at` 范围扫描

### 5.2 BrainPlan 的 MVCC 与 Append-only

BrainPlan 是 append-only 结构：

- 每次修改 **MUST** 追加一条 `brain_plan_delta`，不覆盖
- `brain_plan.version` 字段每次递增（乐观锁）
- `brain_plan.current_state` 字段是最新状态的 JSON snapshot（性能优化，可重算）
- 读历史版本：重放 delta 到指定版本

### 5.3 并发控制

多主进程实例可能同时写同一个 `brain_plan`（HA 场景）：

- **MUST** 用乐观锁：UPDATE ... WHERE version = ?
- 冲突 → 返回 `db_version_conflict` → 调用方重试（读最新 + 重新应用变更）
- **MUST NOT** 用 `SELECT ... FOR UPDATE` 长事务（容易死锁）

## 6. Artifact CAS 存储

### 6.1 什么是 CAS

CAS = Content Addressable Storage。Artifact 的 key = `sha256(content)`。好处：

- **自动去重**：同内容只存一次
- **不可变**：key 不可变，一致性天然
- **可靠**：content 与 key 绑定，无法篡改

### 6.2 Key 算法

```go
func ArtifactKey(content []byte, meta ArtifactMeta) ArtifactRef {
    hash := sha256.Sum256(content)
    // 参考 Git blob 格式前缀避免歧义
    return ArtifactRef(fmt.Sprintf("sha256/%x", hash))
}
```

**MUST 条款**：

- **MUST** 用 sha256，不用 sha1 / md5（抗碰撞）
- **MUST** 对 content 的**原始字节**做 hash，**MUST NOT** 对规范化后的 JSON 做 hash
- Key 的字符串格式 **MUST** 是 `sha256/<64位小写hex>`
- 新增 hash 算法 **MUST** 走新的前缀（`sha256/` / `blake3/`）而不是复用

### 6.3 元数据

Artifact 的 meta 独立存储，不进入 hash：

```go
type ArtifactMeta struct {
    Ref         ArtifactRef      // sha256/abc...
    MimeType    string           // text/plain / image/png / ...
    SizeBytes   int64
    CreatedBy   string           // brain_id
    CreatedAt   time.Time
    RunID       int64
    TurnIndex   int
    Caption     string           // 人读描述
    RefCount    int              // 被多少 Turn 引用
    Tags        map[string]string
}
```

### 6.4 本地文件 CAS 布局（独立版）

```
~/.easymvp/artifacts/
├── sha256/
│   ├── ab/
│   │   └── cdef1234...789.bin          # <hash[0:2]>/<hash>.bin
│   ├── cd/
│   │   └── ef...abc.bin
│   └── ...
└── meta.sqlite                          # 独立 SQLite 文件存 ArtifactMeta
```

- 两级子目录（前 2 hex 字节）防单目录文件数爆炸
- meta.sqlite 独立文件（不与 brain.db 混用），便于并发
- Put 流程：写临时文件 → fsync → rename 到最终位置 → 更新 meta → commit

### 6.5 S3/MinIO CAS 布局（集群版）

```
s3://easymvp-artifacts/
└── sha256/
    └── ab/cd/abcdef1234...
```

- Bucket lifecycle 规则：90 天未访问 → 迁移到 Glacier
- Object lock：WORM 模式防篡改
- Metadata 存 MySQL `mvp_artifact_meta` 表

### 6.6 GC 与引用计数

CAS 会泄漏：Run 失败后没更新 refcount 导致 orphan。**MUST** 实现后台 GC：

- 每夜跑一次 GC：扫 meta 表找 `ref_count=0 且 created_at < 30 days ago` 的 artifact
- 删除前二次校验：grep `brain_plan_delta` 表是否仍引用
- 双重确认后才删

### 6.7 去重 race condition

并发 Put 同 content 可能导致重复写：

- A 和 B 同时计算出 key = `sha256/abc`
- A 写 meta (INSERT)，B 写 meta (冲突)
- B 的 INSERT 因 unique key 冲突失败，**MUST** 视为"已存在"而不是错误，refcount +1

用 MySQL `INSERT ... ON DUPLICATE KEY UPDATE ref_count = ref_count + 1` 或 SQLite `INSERT OR REPLACE` + 手动 refcount 原子更新。

## 7. Run Resume 协议

### 7.1 Resume 的价值

长 Run 跨进程恢复的能力是生产级 Agent 系统的**核心能力**。无 Resume 的系统只能跑 <10 分钟的 Run，否则一次重启就前功尽弃。

### 7.2 Checkpoint 粒度

Checkpoint 在 **Turn 边界** 产生（不是更细）：

- 每个 Turn 结束时写一条 `run_checkpoint` 记录
- Checkpoint 内容是"恢复这个 Run 所需的最小状态集"
- **MUST NOT** 在 Turn 中间 checkpoint（会导致中间状态不一致）

### 7.3 Checkpoint 内容

```go
type RunCheckpoint struct {
    RunID             int64
    TurnIndex         int               // 下一 Turn 编号
    BrainID           string
    State             RunState          // Initialized/Running/ToolBatch
    LastLLMResponse   json.RawMessage   // 最后一次 LLM 响应
    LastToolResults   []ToolResult      // 最后一批 tool_result
    MessagesRef       ArtifactRef       // L3 messages 已落 CAS 的引用
    SystemRef         ArtifactRef       // L1+L2 已落 CAS 的引用
    ToolsRef          ArtifactRef       // tool schemas 已落 CAS 的引用
    CostSnapshot      CostSnapshot      // 已累计的成本
    TokenSnapshot     TokenSnapshot
    BudgetRemaining   Budget
    TraceParent       string            // W3C trace parent
    CreatedAt         time.Time
}
```

### 7.4 Resume 流程

主进程启动时 **MUST**：

1. 调 `RunCheckpointStore.ListResumable()` → 拿到需要 resume 的 run_id 列表
2. 筛选条件：
   - 最后 heartbeat < 30min（太久的不 resume，视为失败）
   - WorkflowRun 状态仍是 `executing` / `reworking`
   - 最后 checkpoint 的 `state` 不是 `Completed` / `Failed` / `Cancelled`
3. 对每个可 resume 的 run：
   a. 加载 RunCheckpoint
   b. 加载 BrainPlan 到 checkpoint.TurnIndex
   c. 重新拉起对应 sidecar（如果没在跑的话）
   d. 给 sidecar 发 `brain.resume` 方法（v1 可选，初期可走重新 `brain.run` 的简化路径）
   e. Runner 从 TurnIndex 继续 loop

### 7.5 brain.resume 方法（可选）

v1.0 初期不强制 sidecar 支持 resume，降级为"从头重跑"策略：

- **简化策略（v1.0）**：resume 时用 `brain.run` 重新跑一次，但给 sidecar 传入已完成的 messages + 成本快照。sidecar 从 messages 的最后位置继续，避免重复调 LLM。
- **完整策略（v1.1）**：新增 `brain.resume` 方法，让 sidecar 完整恢复内部状态机。

### 7.6 Crash 恢复的幂等保证

Resume 必然会导致某些操作"重放"。所有可能重放的操作 **MUST** 是幂等的（见 §8）。

### 7.7 Resume 失败的处理

如果 Resume 失败（sidecar 启动不起来 / checkpoint 损坏 / BrainPlan 缺失 delta）：

- **MUST** 把 Run 状态设为 `Failed` with `error_code=resume_failed, class=InternalBug`
- **MUST** 通知 WorkflowRun 进入 `paused`（让人工决策：重跑 / 放弃）
- **MUST** 写审计事件 `run_resume_failed`
- **MUST NOT** 反复尝试 resume 同一 run（有 resume_attempts 上限 3 次）

## 8. 幂等写规则

### 8.1 幂等的关键

Resume 和重试都可能导致同一操作执行多次。所有持久化写操作 **MUST** 幂等。

### 8.2 实现方式

1. **自然幂等**（最优）：操作本身重复执行效果相同
   - 例：CAS Put（同 key 第二次是 no-op + refcount +1）
   - 例：UPSERT by natural key

2. **显式去重**：带 idempotency_key 字段
   - 每次 Turn 分配一个 `turn_uuid`
   - 写 `run_checkpoint` 时带 `turn_uuid`
   - 重放时先查 `turn_uuid` 是否已存在，存在则跳过

3. **版本号校验**：
   - BrainPlan 用 version 字段
   - 带 version 的写入冲突时失败，需要调用方处理

### 8.3 MUST 清单

所有写路径 **MUST** 归类到以上三种之一：

| 写路径 | 策略 |
|--------|------|
| `BrainPlan.Append` | 版本号校验（optimistic lock） |
| `Artifact.Put` | 自然幂等（CAS） |
| `RunCheckpoint.Save` | 自然幂等（replace by run_id + turn_index） |
| `TokenUsage.Record` | 显式 idempotency_key（turn_uuid） |
| `Cost.Charge` | 显式 idempotency_key |
| `oplog.Append` | 不需要幂等（oplog 重复是可接受的，但 hash chain 需 resync） |
| `WorkflowEvent.Emit` | 显式 idempotency_key |

### 8.4 非幂等操作的处理

以下操作**不可能**幂等：

- 调用外部 LLM 付费 API（每次调用都花钱）
- 发送真实邮件 / webhook
- 文件系统不可恢复的副作用

这些操作 **MUST**：

1. 放在 checkpoint **之后**，不在 checkpoint 事务中
2. 失败可容忍（最多重发一次）
3. 对它们的重发走独立的 "at-most-once" 或 "at-least-once" 标记

## 9. 原子性与一致性

### 9.1 一个 Turn 的写原子性

一个 Turn 结束的写操作集合：

- 追加 `brain_plan_delta`
- 更新 `brain_plan.version` + `current_state`
- 写 `run_checkpoint`
- 写 `usage`（token + cost）
- 写 `tool_call_log`
- 可选：新增 `artifact_meta`

这些 **MUST** 在**单个事务**内完成。任一失败 → 全回滚。

### 9.2 跨表一致性约束

- `brain_plan_delta.plan_id` 必须存在于 `brain_plan.id`（外键）
- `run_checkpoint.run_id` 必须存在于 `mvp_workflow_run.id`
- `tool_call_log.run_id` 同上
- `artifact_meta.ref_count` 与 `brain_plan_delta` 的引用次数 **MUST** 最终一致（eventual，通过 GC 对账）

### 9.3 主进程 crash 的容忍

- 事务中 crash → SQLite/MySQL 自动 rollback
- 事务已提交 + sidecar 已收到 tool_result 但主进程未写 oplog → 下次启动时走 oplog gap detection 补写（oplog 允许少一条但不允许多一条错的）
- Sidecar 已执行工具副作用（写文件 / push git）但 checkpoint 未更新 → 下次 resume 时 sidecar 重放同一 tool_call，因工具幂等不会造成二次副作用（git commit 的 hash 会被判重）

### 9.4 隔离级别

MySQL 集群版 **MUST** 用 `REPEATABLE READ`（InnoDB 默认）。  
SQLite WAL 已保证 serializable-ish snapshot isolation。

## 10. 归档与清理

### 10.1 数据生命周期

| 对象 | 保留期 | 归档目标 |
|------|--------|----------|
| 正在跑的 Run 数据 | 无期 | - |
| 已完成 Run 的 BrainPlan | 90 天 | 冷存（Glacier / 离线备份）|
| Artifact | 90 天 | 同上 |
| 审计 oplog | **永久**（MUST NOT 删除）| WORM 存储 |
| Run checkpoint | 30 天后清理 | - |
| Usage 记账 | 1 年 | 冷存 |

### 10.2 归档流程

每夜 1 次：

1. 扫 `mvp_brain_plan` 找 `created_at < 90 days ago AND archived = 0`
2. 导出为 tar.gz：`{run_id}.plan.tar.gz`
3. 上传到归档存储
4. 校验上传完整性（sha256）
5. 标记 `archived=1, archived_at=?, archive_ref=?`
6. 清理本地磁盘（但**保留元数据行**，不删 row）

### 10.3 合规删除

当用户请求删除个人数据（GDPR 等）：

1. 标记相关 row 的 PII 字段为 null 或匿名 hash
2. 删除关联的 artifact（按 ref 软删 + GC 真删）
3. **MUST NOT** 删除 `oplog`（oplog 保留事实，但可以掩码 PII 字段）
4. 记录一条 `compliance_deletion` 审计事件

## 11. 备份与灾难恢复

### 11.1 备份策略

| 类型 | 频率 | 保留 |
|------|------|------|
| MySQL full dump | 每日 | 30 天 |
| MySQL binlog | 实时 | 7 天 |
| SQLite brain.db | 每小时 | 7 天 |
| SQLite oplog.db | 每小时 | **永久** |
| Artifact CAS | 每小时增量 | 同主存 |
| 配置（mvp_config）| 每次变更 | 永久 |

### 11.2 RPO / RTO

| 场景 | RPO | RTO |
|------|-----|-----|
| 主进程 crash | 0（事务已提交则无丢失） | < 30s（重启 + resume） |
| MySQL 主库 crash | < 1s（binlog 同步）| < 5min（故障切换） |
| 整个集群灭失 | < 1h（binlog + full dump）| < 4h（从备份恢复 + 重建） |
| 存储介质损坏 | 到最近增量备份 | 同上 |

### 11.3 恢复演练

每季度 1 次恢复演练：

- 从备份重建 MySQL
- 从归档恢复一段 Artifact
- 从 oplog 回放重建任一 Run 的完整历史
- 验证 hash chain 完整性
- 验证恢复后新 Run 能正常跑

### 11.4 异地备份

集群版 **MUST**：

- MySQL binlog 实时异地复制
- Artifact bucket 跨 region 复制
- oplog 锚点写到外部不可变存储（每日一条）

## 附录 B · MySQL 表结构增量

> **实际迁移文件**：`admin-go/manifest/sql/mysql/000012_brain_kernel_v1.up.sql` / `.down.sql`（已落地）。
> 下文 DDL 为规格参考，与迁移文件在字段语义上一致。**迁移文件遵循 EasyMVP 现有风格约定**：`bigint unsigned` 主键、反引号、小写关键字、每字段中文 COMMENT、不使用 DB 外键约束（一致性走应用层）。

### B.1 EasyMVP 约定强制覆盖项

下列点与 SQLite schema（附录 A）不同，是 MySQL 实现必须遵守的 EasyMVP 铁律：

| 约定 | 规则 |
|------|------|
| 主键类型 | `bigint unsigned`，雪花 ID（应用层生成，不用 AUTO_INCREMENT） |
| 外键 | **禁用** DB 外键，一致性在应用层 Repo 层保证（避免分库分表与软删除冲突） |
| `created_by` / `dept_id` | `bigint unsigned`，用户 ID / 部门 ID。**禁止**用字符串存 actor 名（actor 字段用 `varchar`，是 brain_id 或 system 标识，不同于 created_by） |
| `deleted_at` | 业务表必备（append-only 日志表如 `mvp_brain_plan_delta` / `mvp_brain_usage` 可豁免） |
| 字符集 | `utf8mb4` / `utf8mb4_unicode_ci` 全表统一 |
| 索引命名 | `idx_<表>_<列>` / `uk_<表>_<列>`，反引号包裹 |
