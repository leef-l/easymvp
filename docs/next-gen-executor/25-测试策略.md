# 25 · BrainKernel 测试策略 v1

> **定位**：本文档是 BrainKernel 的**测试战略总纲**，定义如何验证主进程、sidecar、协议层、错误模型、Agent Loop、安全边界、可观测性在**真实与对抗环境**下仍然正确。汇总并承接 20 / 21 / 22 / 23 / 24 五份规格文档里的合规测试矩阵，把它们组织为可执行的测试分层与执行流水线。
>
> **规格性质**：本文档使用 RFC 2119 关键词。测试策略是防止回归的**唯一自动化保障**——任何偏离 MUST 条款的测试方案都会导致 CI 绿灯但生产红灯。
>
> **版本**：测试策略版本 `v1`。v1 冻结后的测试编号（C-01 / C-E-01 / C-L-01 / C-S-01 / C-O-01 等前缀规则）**MUST NOT** 改名。新增合规测试 **MUST** 追加编号不得复用。

## 1. 为什么要独立一份测试策略

Agent 系统的测试有三个"反直觉"特点，把它和普通后端测试区分开：

1. **非确定性是默认态**。LLM 调用同样的 prompt 可能返回不同的 tool_call，传统的 golden snapshot 测试在 Agent 领域几乎无效。
2. **失败模式是"慢性病"**。单元测试全绿 → 集成测试全绿 → 放生产跑 10 个小时 → context compaction 把关键事实删了 → LLM 开始编造 → 用户收到错误代码。这类 bug 单纯增加单元测试覆盖率 100% 也抓不到。
3. **成本驱动**。真实 LLM 调用贵而慢，不可能每次 PR 都跑 1000 个 LLM 调用做 E2E。必须有 replay / mock / fake 的完整 fixture 基础设施。

普通后端的 "Test Pyramid + Mock Everything" 不够用。本规格定义 7 层测试金字塔 + 2 套独立 replay 基础设施 + 自动化红队，覆盖上述三个特点。

对应 02 的决策追溯锚点：所有关键决策（1 ~ 8）都依赖本规格定义的测试矩阵把"写出来"与"真的对"两件事分开。

## 2. 测试金字塔与分层

### 2.1 七层金字塔

```
                          ┌─────────────────┐
                          │ 10%  Red Team   │  ← 安全红队 + 混沌
                          ├─────────────────┤
                          │ 10%  LLM Replay │  ← 真实/半真实 LLM fixture
                          ├─────────────────┤
                          │ 15%  E2E        │  ← 完整 Run 在 fake sidecar 下跑
                          ├─────────────────┤
                          │ 15%  Integration│  ← 主进程 + 真 sidecar + mock LLM
                          ├─────────────────┤
                          │ 15%  Cross-lang │  ← Go/Python/Rust/TS SDK 一致性
                          ├─────────────────┤
                          │ 15%  Contract   │  ← 协议/错误模型/Agent Loop 合规
                          ├─────────────────┤
                          │ 20%  Unit       │  ← 纯函数 + 结构体 + edge case
                          └─────────────────┘
```

百分比代表**相对运行时预算**，不是数量。Unit 数量最多但每条最快；Red Team 数量最少但耗时最长。

### 2.2 各层目标

| 层 | 目标 | 运行频率 | 时长预算 |
|----|------|----------|----------|
| Unit | 确认纯函数 / 结构体 / 边界条件正确 | 每次 PR | < 3 min |
| Contract | 协议 / 错误模型 / Agent Loop 的 MUST 条款 | 每次 PR | < 5 min |
| Cross-lang | Go 主进程 与 Python/Rust/TS SDK 跨语言同构 | 每次 PR | < 5 min |
| Integration | 主进程 + 真 sidecar + mock LLM，验证粘合层 | 每次 PR | < 10 min |
| E2E | 完整 Run 流程，fake sidecar + scripted LLM | 每次 PR | < 15 min |
| LLM Replay | 基于录制的真实 LLM 响应重放，确保升级后行为稳定 | 每 nightly | < 1 h |
| Red Team | 注入对抗性 payload 验证安全边界 | 每 weekly | ≤ 4 h |

## 4. 单元测试规格

### 4.1 覆盖范围

Unit 测试 **MUST** 覆盖：

- 所有 `errmodel` 包的函数（构造器 / Wrap / 脱敏 / fingerprint 算法）
- 所有 `stdio protocol` 包的 parser / writer state machine
- 所有 `agent_loop` 包的 纯函数（prompt 渲染 / cache 标记 / budget 追踪 / 死循环检测）
- 所有 `vault` / `sanitizer` / `guardrail` 包
- 所有 `decide` 决策函数

### 4.2 MUST 条款

- Unit 测试 **MUST NOT** 依赖网络 / 数据库 / 文件系统（允许 `testing/fstest`）
- Unit 测试 **MUST** 在 100ms 内完成单条
- Unit 测试 **MUST** 使用 table-driven 模式（`cases := []struct{...}{...}`）
- 错误路径 **MUST** 覆盖（不是只测 happy path）
- 任何 public 函数 **MUST** 至少有一个 Unit case

### 4.3 Code Coverage 目标

| 包 | 覆盖率目标 |
|----|-----------|
| `errmodel` | ≥ 95% |
| `stdio protocol` | ≥ 90% |
| `vault` / `sanitizer` / `guardrail` | ≥ 90% |
| `agent_loop` 纯函数 | ≥ 85% |
| `runner` / `kernel` | ≥ 70% |
| 全仓 | ≥ 75% |

覆盖率 **MUST** 通过 `go test -coverprofile` 记录并在 PR 上展示 diff。覆盖率下降 **SHOULD** 阻断合并（允许 reviewer override 并留评论）。

## 5. 合约测试（Contract Tests）

合约测试 = 把规格文档里每条 MUST 条款翻译成一个可执行用例。这是本规格的**战略武器**——它让规格文档成为"活文档"，规格变则测试变则实现变。

### 5.1 实现约定

合约测试 **MUST**：

- 单独放在 `admin-go/app/mvp/internal/workflow/executor/brainkernel/compliance/` 目录
- 文件命名 `compliance_<spec>_<id>_test.go`，如 `compliance_protocol_c01_test.go`
- 每个测试函数名 **MUST** 包含规格编号：`TestC_01_FrameFormatContentLength`
- 每个测试 **MUST** 在注释或 doc comment 里引用规格章节
- 测试失败信息 **MUST** 指明哪条 MUST 条款没满足

```go
// TestC_01_FrameFormatContentLength verifies [20-协议规格.md §2.2]:
// "A frame MUST start with "Content-Length: <N>\r\n\r\n" followed by exactly N bytes of UTF-8 JSON body."
func TestC_01_FrameFormatContentLength(t *testing.T) {
    // table-driven cases, including edge cases:
    // - minimum frame (N=2: "{}")
    // - maximum frame (N=16MiB)
    // - N=0 → reject
    // - N > 16MiB → reject with frame_too_large
    // - missing \r\n\r\n → reject
    // - body shorter than N → wait for more bytes
    // - body longer than N → only consume N, next frame starts right after
}
```

### 5.2 合规测试的生命周期

```
规格起草 → MUST 条款  
           ↓
         创建 compliance_<id>_test.go (初始是 skipped)
           ↓
         实现代码
           ↓
         解开 skip，测试通过
           ↓
         PR 合并
```

**MUST NOT** 在没有 compliance test 的情况下声明某条规格被实现。PR 描述里要明确列出本次 PR 新解锁的合规测试编号。

### 5.3 不能替代的测试

Contract 测试**只验证 MUST 条款**，不验证业务行为。它回答"这个实现是否符合协议"，不回答"这个功能是否对用户有用"——后者是 E2E 的任务。

## 6. 集成测试

### 6.1 作用域

集成测试的边界：**真主进程 + 真 sidecar 子进程 + mock LLM provider + mock 外部服务**。不跨网络调真 LLM。

### 6.2 必覆盖场景

- 主进程启动 → 加载 sidecar → 发 `initialize` → 收到握手结果
- 完整 `brain.run` 执行到 `Completed`
- `brain.cancel` 在各个阶段（initializing / llm waiting / tool executing）都能生效
- sidecar 崩溃后主进程能恢复（重启 + zombie reap）
- sidecar 发送非法帧后主进程能降级而不 panic
- 多 sidecar 并发执行时 token bucket 公平分配
- `llm.complete` 反向 RPC 被上游 429 时中心化退避生效
- 5 个 sidecar 并发 quarantine 压力测试
- Hash chain append-only 在 1k 条审计事件下保持完整

### 6.3 Mock LLM Provider

集成测试使用一个 `FakeProvider`，**MUST** 支持：

- 按 request hash 返回 pre-scripted 响应
- 模拟 5xx / 429 / timeout
- 模拟 streaming chunks
- 模拟 tool_use / safety refusal / stop_reason 变体
- 模拟 Prompt Cache hit 与 miss

FakeProvider 实现放在 `admin-go/.../compliance/fake_provider.go`，作为合约测试和集成测试的共享 fixture。

### 6.4 Mock 数据库

集成测试 **MUST** 使用真实的 MySQL（Docker 启动临时实例），不使用 sqlite / in-memory mock——这是为了确保 migration 与 charset 问题在集成层就被抓到（对齐十四铁律 golang-migrate）。

## 7. 跨语言 SDK 一致性测试

### 7.1 为什么需要

大脑 sidecar 可能由 Go / Python / Rust / TypeScript 写。同一份规格必须在四种语言里**行为等价**。跨语言一致性测试用"黑盒驱动同一组 fixture"的方式验证。

### 7.2 测试框架

在 `docs/next-gen-executor/tests/cross-lang/` 目录放一个**语言无关的测试套件**：

```
cross-lang/
├── fixtures/
│   ├── frame/            # 协议帧二进制 fixture
│   ├── brain_error/      # JSON 错误对象 fixture
│   ├── chat_request/     # ChatRequest fixture
│   └── expected/         # 期望结果
├── runners/
│   ├── run_go.sh
│   ├── run_python.sh
│   ├── run_rust.sh
│   └── run_ts.sh
├── harness/              # 各语言的 harness binary
│   ├── go/main.go
│   ├── python/harness.py
│   ├── rust/src/main.rs
│   └── ts/harness.ts
└── verify.sh             # 对比各语言输出
```

Harness 的约定：

- 输入：stdin 接收一条 fixture 指令（`parse_frame /path/to/bin.frame`）
- 输出：stdout 输出 canonical JSON（按 sorted keys）
- 退出码：0 = 成功，非 0 = 失败
- verify.sh 跑 fixture × 语言笛卡尔积，对比 canonical 输出

### 7.3 必覆盖 fixture

| fixture 类别 | 数量 | 验证 |
|--------------|------|------|
| 合法帧 parse | 20 | 四语言解析出相同 JSON |
| 非法帧 reject | 10 | 四语言都拒绝且错误分类相同 |
| BrainError round-trip | 15 | 四语言序列化后字节一致 + 反序列化后字段一致 |
| ChatRequest canonical | 10 | 四语言 canonical 序列化字节一致 |
| fingerprint 计算 | 10 | 四语言对相同 BrainError 产出相同 fingerprint |
| Decide 决策输入输出 | 15 | 四语言对同输入产出同动作 |

合计 80 个 fixture × 4 语言 = 320 个用例。

### 7.4 自动化

跨语言测试在 CI 里跑一个独立 job（`cross-lang-compliance`）。Go 主进程 PR 与各语言 SDK repo 都触发。任何一个语言的输出 diverge → 阻断。

## 8. 端到端 Scenario 测试

### 8.1 Scenario 定义

E2E 测试是**场景驱动**的，每个场景是一段完整的用户故事：

```yaml
# tests/e2e/scenarios/fix_failing_test.yaml
name: Fix failing test
description: |
  用户提交一个测试失败的仓库，code_brain 读取测试日志、
  定位 bug、修改源文件、重跑测试、提交修复。
setup:
  project: "fixtures/projects/failing_auth_test"
  initial_git_sha: "abc123"
brains:
  central: central_brain@v1
  code: code_brain@v1
  verifier: verifier_brain@v1
llm:
  mode: replay
  cassette: "cassettes/fix_failing_test.json"
input:
  goal: "Fix the failing test in auth_test.go and get it passing"
expected:
  run_status: "completed"
  git_sha_changes: true
  final_test_status: "passing"
  max_cost_usd: 0.50
  max_turns: 30
  assertions:
    - type: file_modified
      path: "src/auth.go"
    - type: test_passes
      suite: "go test ./auth"
    - type: no_safety_refused
    - type: no_loop_detected
    - type: audit_event_present
      event_type: "brain_run_completed"
```

### 8.2 必覆盖场景（v1 基线）

| # | 场景 | 目的 |
|---|------|------|
| E-01 | Fix failing test | 验证完整 read → edit → verify 循环 |
| E-02 | Add new feature | 验证 plan → implement → test → commit |
| E-03 | Refactor with preservation | 验证 diff review + 不破坏测试 |
| E-04 | Detect and refuse malicious change | 验证 SafetyRefused 升级路径 |
| E-05 | Handle flaky tests | 验证 Transient 重试 |
| E-06 | Context compaction over long run | 50+ Turn 运行后仍保持正确 |
| E-07 | Multi-brain delegation | central_brain delegates 2 个 code_brain 并行 |
| E-08 | Sidecar crash mid-run | kill -9 sidecar 后主进程恢复 |
| E-09 | Cost budget exhaustion | 成本超限时优雅失败 |
| E-10 | Human pause & resume | 中途 paused → 人工 review → resume |

### 8.3 环境

E2E 测试在**隔离 Docker Compose 栈**里跑：

- MySQL（临时实例）
- 主进程（真 binary）
- sidecar（真 binary，通过 Docker sandbox 启动）
- FakeProvider（按 cassette 回放）
- Git 仓库（fixture project）

每个 scenario 跑完后 **MUST** 清理环境（tear down）。

## 9. 混沌测试

### 9.1 目标

验证主进程在**基础设施故障**下仍能保持正确性：

- 断开 sidecar stdin（模拟 sidecar 卡住）
- kill -9 sidecar（模拟 OOM）
- 注入网络抖动（上游 LLM 丢包）
- DB 连接池耗尽
- 磁盘满（写 artifact 失败）
- 时钟跳跃（NTP 调整）
- 主进程自己被 SIGSTOP 1 秒

### 9.2 混沌场景

| # | 场景 | 预期 |
|---|------|------|
| Chaos-01 | sidecar stdin 写入 hang 300s | 主进程心跳超时 → 重启 sidecar |
| Chaos-02 | sidecar stdout 吐大量垃圾帧 | 主进程降级 sidecar 为 degraded，不崩 |
| Chaos-03 | kill -9 sidecar | 主进程 reap zombie + 重启 |
| Chaos-04 | 上游 LLM 随机 30% 5xx | 主进程按 Transient 重试，不雪崩 |
| Chaos-05 | DB 随机 1% deadlock | 主进程正确重试，数据不丢 |
| Chaos-06 | 磁盘满 | 主进程拒绝新写入，已有 Run 完成时降级 |
| Chaos-07 | 时钟倒流 10s | 审计时间戳正常，trace 不错位 |
| Chaos-08 | 主进程 SIGSTOP 1s | sidecar 心跳延迟但不重启 |
| Chaos-09 | OTel Collector 离线 | Metric 进 ring buffer，主进程不崩 |
| Chaos-10 | MySQL 重启 | 主进程恢复连接，in-flight Run 通过 resume 继续 |

### 9.3 实现

混沌测试使用 `tc netem` / `chaos-mesh` / 自定义 hook 注入故障。每次 nightly 跑 1 次。

## 10. 安全测试与红队

### 10.1 自动化安全测试

基线安全测试（每次 PR 跑）：

- Guardrail 白名单边界 fuzzing
- 路径遍历 (`../`) 与规范化绕过
- 符号链接逃逸
- Shell 注入 payload
- Prompt injection fixture
- Unicode RTL / zero-width 攻击
- TLS 证书验证
- CSRF / CORS
- JWT 篡改

### 10.2 红队演练

每月 1 次，由 **独立团队**（不是功能开发者）尝试：

- 让 code_brain 读取 `~/.ssh/id_rsa`
- 让 browser_brain 访问 `http://169.254.169.254/`
- 在 tool_result 里隐藏 prompt injection 让 LLM 泄漏 system prompt
- 通过构造 tool 参数触发 shell 注入
- 构造畸形帧让 parser crash
- 伪造 oplog 篡改记录
- 冒充 sidecar 签名
- 长时间占用 token bucket 做 DOS

### 10.3 红队报告

每次红队演练 **MUST** 产出报告，包含：

- 尝试的攻击列表
- 成功 / 部分成功 / 失败数量
- 对于成功的攻击，**MUST** 在 1 周内修复并加入自动化回归测试
- 所有红队发现的 CVE-like 问题 **MUST** 有 CVSS 打分

## 11. LLM Replay 测试

### 11.1 Cassette 机制

LLM 调用全部通过 `CassetteProvider` 录制 / 回放：

```
First run (RECORD mode):
  FakeProvider → Real Anthropic API → response → save to cassette.json

Subsequent runs (REPLAY mode):
  FakeProvider → match by request hash → return cached response
```

Cassette 文件格式：

```json
{
  "meta": {
    "recorded_at": "2026-04-01T00:00:00Z",
    "provider": "anthropic",
    "model": "claude-opus-4-6",
    "anthropic_version": "2023-06-01"
  },
  "interactions": [
    {
      "request_hash": "a3f2...",
      "request": { ... },
      "response": { ... },
      "latency_ms": 1830
    }
  ]
}
```

### 11.2 Cassette 匹配规则

- 按 `request_hash = sha256(canonical_json(request))` 匹配
- canonical_json 要排除非确定性字段（`metadata.trace_id` / 时间戳）
- 不匹配 → 测试失败（不允许回退到真调用）

### 11.3 Cassette 维护

- Cassette 文件存放 `tests/cassettes/`
- 模型升级 → 重新 RECORD → diff 审查 → 合并
- Cassette 脱敏：写入前跑一次 secret redactor

### 11.4 为什么不直接 mock LLM

因为 LLM 的**风格漂移**是真实存在的风险：同一个 prompt 在 claude-opus-4.5 → 4.6 的输出风格可能差异大。Mock 是开发者手写的"我以为 LLM 会这样回"，Cassette 是真实历史。Replay 测试能抓住"LLM 升级后我们的 Agent 循环不再能收敛"这类问题。

### 12.1 基准测试 (bench)

关键路径 **MUST** 有 Go bench：

- `BenchmarkFrameParse` - 协议帧解析
- `BenchmarkFrameWrite` - 协议帧写入

- `BenchmarkHashChainAppend` - oplog append

每次 PR 对比 baseline，退化 > 10% → CI 警告（不阻断）。

### 12.2 负载测试

在 staging 每周跑一次：

- 100 个并发 Run
- 10 个大脑，每个 10 Run
- 每个 Run 平均 30 Turn
- 持续 1 小时
- 监控：P99 延迟 / 错误率 / 内存 / FD / goroutine 数

验收标准：

- Run 成功率 ≥ 95%
- 主进程内存稳定（无泄漏）
- CPU 利用率 < 80%
- 无 sidecar zombie 残留

### 13.2 流水线 MUST

- Stage 1 ~ 4 **MUST** 在每次 PR 全部通过才允许合并

- Flaky 测试 **MUST** 在 1 周内修复或删除
- **MUST NOT** 使用 `@Retry` 掩盖 flaky
- Flaky 率 **MUST** < 1%

```go
// admin-go/app/mvp/internal/workflow/executor/brainkernel/compliance/fake_sidecar.go

// FakeSidecar 实现一个最小的 sidecar，用于 Unit 和 Contract 测试。
// 它按 script 执行，不依赖真 LLM，不依赖真 tool 执行。
type FakeSidecar struct {
    Script      []SidecarStep  // 预设的行为序列
    cursor      int
    stdin       io.Reader
    stdout      io.Writer
    protocol    *protocol.Session
}

type SidecarStep interface {
    Apply(s *FakeSidecar) error
}

type ReplyStep struct {
    RequestID string
    Result    any
    Error     *errmodel.BrainError
}

type ReverseRPCStep struct {
    Method string
    Params any
}

type DelayStep struct {
    Duration time.Duration
}

type CrashStep struct {
    ExitCode int
}

type SendMalformedFrameStep struct {
    Raw []byte
}

func (f *FakeSidecar) Run(ctx context.Context) error {
    for _, step := range f.Script {
        if err := step.Apply(f); err != nil {
            return err
        }
        if ctx.Err() != nil {
            return ctx.Err()
        }
    }
    return nil
}
```
