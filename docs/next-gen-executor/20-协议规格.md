# 20 · BrainKernel stdio 协议规格 v1（RFC 风格）

> **定位**：这是 BrainKernel 主进程与大脑 sidecar 之间的**传输层 + 会话层**正式规格文档，是 [02-BrainKernel设计.md](./02-BrainKernel设计.md) §12.5.4 的下位规格，相当于一份内部 RFC。任何第三方大脑 SDK 实现者读完这一份即可独立写出协议栈实现，不需要反推 Go 源码。
>
> **规格性质**：本文档使用 RFC 2119 的关键词：**MUST** / **MUST NOT** / **SHOULD** / **SHOULD NOT** / **MAY**，对应中文"必须 / 禁止 / 应当 / 不应 / 可以"。任何实现偏离 MUST 条款即为**不合规实现**，主进程有权拒绝加载。
>
> **适用范围**：本规格适用于通过 `SubprocessRunner` 启动的所有 BrainKernel sidecar。`MCPAdapterRunner` 消费的外部 MCP server 不走本规格，而走 Anthropic MCP stdio transport 规格（详见 02 §12.5.0）。
>
> **版本**：协议版本 `v1`。v1 冻结后任何方法的参数只能加字段不能改/删；返回值同理。打破 v1 契约需起 v2，两版并存一段时间后淘汰 v1。

## 1. 为什么要把协议规格独立成文档

02 是内核宪法，讲的是"为什么分层、为什么全 sidecar、为什么 fault_policy 可配"。协议规格是另一类文档——它要回答**比特层面的问题**：

- 一个帧的第 0 字节到第 N 字节是什么？
- 主进程读到半截 JSON 时该怎么办？
- 双向 RPC 的 id 怎么分配才不会撞车？
- stdin 管道被写满之后写方 goroutine 会不会死锁？
- sidecar 怎么区分"主进程让我停"和"主进程死了"？

这些问题的答案必须**精确到字节和状态机**，而且要对所有语言实现（Go / Rust / Python / TypeScript 写的大脑 SDK）同样适用。把这些内容混进 02 会让宪法失焦，而且跨语言实现者读宪法会淹没在业务语义里找不到协议规格。所以本文档独立成篇。

本规格对应 02 的决策追溯锚点：**决策 5**（全 sidecar）和**决策 6**（Runner 三方法）共同定义了 sidecar 的存在方式，本规格定义这些 sidecar 如何与主进程对话的**比特级契约**。

## 2. 传输层：帧格式

### 2.1 历史错误与修正

**02 §12.5.4 最初的描述**："基于 JSON-RPC 2.0，每行一条 JSON 消息，以换行分隔（换行分隔流式解析，不需要 Content-Length 头）"。

**这个描述是错的**，本规格正式替代之。换行分隔（NDJSON）方案有三个致命问题：

1. **大 payload 不可靠**：一次 `llm.complete` 的 messages 数组可能 800 KB；一次 `artifact.put` 的 base64 截图可能 300 KB；序列化成单行 JSON 对 writer 的严谨性要求极高，任何一次不规范的 pretty-print 或隐藏换行就会让 reader 解帧错位。
2. **DoS 攻击面**：没有帧大小上限的 NDJSON reader 等于"读到换行为止"，恶意或 bug sidecar 可以吐一行 4 GB 的数据把主进程打爆内存。
3. **与生态不一致**：LSP、Anthropic MCP stdio transport、Microsoft DAP 全部采用 HTTP 风格的 `Content-Length` 头部帧。NDJSON 在整个 stdio JSON-RPC 生态里是少数派，第三方开发者认知成本高。

**v1 正式采用 LSP 风格的 Content-Length 头部帧**。下面的 §2.2 ~ §2.5 是 MUST 级规格。

### 2.2 帧结构

一个帧由**三段**组成，严格顺序：

```
┌─────────────────────────────────────────────────────────────────┐
│ Header Block（ASCII 文本，以 \r\n\r\n 结束）                      │
├─────────────────────────────────────────────────────────────────┤
│ Body（UTF-8 编码的 JSON，长度由 Header 的 Content-Length 字段声明） │
└─────────────────────────────────────────────────────────────────┘
```

**Header Block** 由一行或多行 `Key: Value` 组成，每行以 `\r\n`（CRLF）结束，整个 Header Block 以空行（即多一个 `\r\n`）结束。格式严格对齐 HTTP/1.1 的 header 语法。

**Body** 是 UTF-8 编码的 JSON 文本，**MUST NOT** 带 BOM（字节序标记）。Body 里的 JSON 可以自由包含换行、空格、任意 Unicode 字符，因为帧边界由 Content-Length 决定，不依赖内容字符。

### 2.3 必填 Header 字段

| Header | 含义 | 约束 |
|---|---|---|
| `Content-Length` | Body 的字节长度（十进制整数） | **MUST** 出现，**MUST** 是 Body 的精确 UTF-8 字节数（不是字符数），取值范围 `[0, 16777216]` = `[0, 16 MiB]` |
| `Content-Type` | Body 的媒体类型 | **MUST** 为 `application/vscode-jsonrpc; charset=utf-8`。省略时主进程和 sidecar **MAY** 假设等同于该值，但 v1 合规实现 **SHOULD** 显式写出以便 tcpdump 调试时一眼能认 |

**禁止的 Header**：任何非上表中列出的 Header 字段都是 **non-normative**。v1 实现接收到未知 Header **MUST** 忽略（不是拒绝），以便未来扩展；v1 实现发送 **MUST NOT** 主动加未知 Header。

### 2.4 帧大小上限（DoS 防线）

| 方向 | 单帧 Body 最大值 | 超限行为 |
|---|---|---|
| 主进程 → sidecar | 16 MiB | 主进程写出前自检，超限 **MUST** 拒绝写出并把请求转换为本地错误（`frame_too_large`） |
| sidecar → 主进程 | 16 MiB | 主进程 reader 解析 Header 后发现 Content-Length > 16 MiB，**MUST** 立即：(1) 停止读该 Body；(2) 对该 sidecar 实例发 `SIGTERM`；(3) 写审计日志 `sidecar_frame_too_large`；(4) 将该 sidecar 标记为 `quarantined` |

**为什么是 16 MiB**：JSON-RPC 消息本身应该是小控制消息；大数据（截图、视频、大文件）**MUST** 走 `content_ref` 间接引用（详见 §8），不内联进帧。16 MiB 留出了系统 prompt + 完整 messages + 大 tool_result 的合理上限，同时对单帧解析内存友好。

### 2.5 字节级示例

一个最小的 `health.ping` 请求（主进程发）在 wire 上的完整字节序列：

```
Content-Length: 61\r\n
Content-Type: application/vscode-jsonrpc; charset=utf-8\r\n
\r\n
{"jsonrpc":"2.0","id":"k:1","method":"health.ping","params":{}}
```

Body 部分是精确的 61 字节 UTF-8（不含任何前后空白或换行）。`\r\n` 字面量对应 ASCII 0x0D 0x0A。

sidecar 的响应同样格式：

```
Content-Length: 116\r\n
Content-Type: application/vscode-jsonrpc; charset=utf-8\r\n
\r\n
{"jsonrpc":"2.0","id":"k:1","result":{"uptime_seconds":42,"active_runs":0,"total_runs":17,"memory_rss":52428800}}
```

### 2.6 Parser 状态机

合规的 reader 必须实现如下状态机：

```
     ┌─────────────────┐
     │ READ_HEADER_LINE│ ←─┐
     └────────┬────────┘   │
              │             │
       读到一行（到 \r\n）  │
              │             │
              ▼             │
     ┌─────────────────┐    │
     │ 判断是否空行    │    │
     └────┬────────┬───┘    │
          │        │         │
       非空行     空行       │
          │        │         │
          │        ▼         │
          │  ┌──────────┐    │
          │  │ 验证必填 │    │
          │  │ Header   │    │
          │  └────┬─────┘    │
          │       │          │
          │       ▼          │
          │  ┌──────────┐    │
          │  │READ_BODY │    │
          │  │(精确读取 │    │
          │  │ N 字节) │     │
          │  └────┬─────┘    │
          │       │          │
          │       ▼          │
          │  ┌──────────┐    │
          │  │ 解析 JSON│    │
          │  │ 派发     │    │
          │  └────┬─────┘    │
          │       │          │
          └───────┴──────────┘
                  │
                  │(循环)
                  ▼
             READ_HEADER_LINE
```

**状态机约束**（全部 MUST 级）：

1. `READ_BODY` 阶段 **MUST** 按 Content-Length 精确读 N 字节，不能用"读到下一个空行"之类的启发式。
2. 读 Body 过程中遇到 EOF（对端关闭 stdout）**MUST** 判定为 `peer_closed`，立即走半关闭状态机（§6），**MUST NOT** 尝试解析已读到的部分 Body。
3. 读 Body 超时 **MUST** 由主进程侧的 `read_timeout`（默认 5 秒，仅对单次 Body 读取计时）控制，超时同样走半关闭状态机。
4. `READ_HEADER_LINE` 阶段每行长度 **MUST** 限制在 8192 字节以内，超限直接 kill 对端（防止恶意 Header 打爆解析器）。

### 2.7 Writer 的串行化要求（关键）

**writer 侧 MUST 单例化**：每个 stdio 方向（主进程 → sidecar 的 stdin、sidecar → 主进程的 stdout）**MUST** 有且只有一个 writer goroutine（或等价的 writer task）。所有出站消息必须先进一个内部 channel（或 queue），由 writer goroutine 串行写出。

**为什么**：如果允许多个 goroutine 并发写同一个 pipe，帧与帧之间会**交织**（一个帧的 Header 被另一个帧的 Body 插入），解帧立即崩溃。这是 stdio 协议最经典的坑。MCP / LSP 的所有合规实现都遵守这条规则。

**Go 侧推荐实现**：

```go
type frameWriter struct {
    w       io.Writer
    ch      chan []byte // buffered, capacity = MaxInFlight * 2
    stopCh  chan struct{}
    doneCh  chan struct{}
}

func (fw *frameWriter) loop() {
    defer close(fw.doneCh)
    for {
        select {
        case frame := <-fw.ch:
            if _, err := fw.w.Write(frame); err != nil {
                // log + break
                return
            }
        case <-fw.stopCh:
            return
        }
    }
}
```

**多个生产者**（处理不同请求的业务 goroutine）只能把编码好的完整帧字节数组丢进 `fw.ch`，**MUST NOT** 直接 `fw.w.Write`。

## 3. 会话层：JSON-RPC 2.0 细化

### 3.1 基础约束

本协议基于 **JSON-RPC 2.0**（https://www.jsonrpc.org/specification）。v1 对 JSON-RPC 2.0 的细化：

1. **版本字段**：每个消息 **MUST** 带 `"jsonrpc":"2.0"`，缺失即视为不合规帧，对端 **MUST** 回 `-32600 Invalid Request` 或直接 kill（取决于严重程度）。
2. **消息类型**：四种——Request / Response / Notification / BatchRequest。v1 **MUST NOT** 使用 BatchRequest（MCP 和 LSP 都禁用了批量请求，理由是跨进程背压和错误定位复杂度太高）。
3. **Notification**：`id` 字段缺失或显式为 `null` 的消息是 notification，对端 **MUST NOT** 回响应。v1 中 `trace.emit` 和 `log.emit` 可使用 notification 形式。

### 3.2 方法名命名空间

所有方法名采用 `namespace.verb` 结构，namespace 固定为以下 8 个之一：

| namespace | 用途 | 方向 |
|---|---|---|
| `initialize` | 握手（特例，无 namespace，直接方法名） | 主 → 子 |
| `brain` | 大脑生命周期（run / cancel / describe） | 主 → 子 |
| `health` | 心跳 | 主 → 子 |
| `shutdown` | 优雅停机（特例，无 namespace） | 主 → 子 |
| `tool` | 工具调用 | 子 → 主（反向） |
| `llm` | LLM 代理 | 子 → 主（反向） |
| `plan` | BrainPlan 持久化 | 子 → 主（反向） |
| `subtask` | 子任务委派 | 子 → 主（反向，仅 central_brain） |
| `artifact` | 产物存储 | 子 → 主（反向） |
| `trace` | 审计追踪 | 子 → 主（反向） |
| `log` | 结构化日志 | 子 → 主（反向） |
| `$` | 协议元方法（`$/ping` / `$/progress` / `$/cancelRequest`） | 双向 |

**扩展规则**：v1 冻结后新增 namespace **MUST** 先经文档 PR 评审，**MUST NOT** 在 wire 上私自扩展。未知 namespace 的请求对端 **MUST** 回 `-32601 Method Not Found`。

### 3.3 保留前缀：`$/` 协议元方法

借鉴 LSP 的约定，`$/` 前缀的方法是**协议层内部通信**，对业务层透明。v1 预定义：

- `$/ping` / `$/pong`（notification + notification）：协议层探活，独立于业务层的 `health.ping`。探活无响应触发半关闭状态机，但**不**触发业务层熔断器。
- `$/cancelRequest`（notification）：取消某个特定 in-flight 请求。参数 `{id}`。
- `$/progress`（notification）：长 running 请求的进度推进。用于流式响应（详见 22-Agent-Loop规格.md 的 ChatEvent 定义）。

## 4. 双向 RPC 与 id 命名空间

### 4.1 问题陈述

标准 JSON-RPC 2.0 假设通信是**单向**的：客户端发请求，服务端回响应。但 BrainKernel 的 stdio 协议是**全双工双向**的：

- 主进程向 sidecar 发 `brain.run`（主是 client，子是 server）
- 同时 sidecar 向主进程反向发 `llm.complete`（子是 client，主是 server）

两个方向的请求各自需要 `id` 字段来匹配响应。**如果两个方向共用同一个 id 命名空间，就会撞车**：主进程分配了 `id=1` 给 `brain.run`，sidecar 也分配了 `id=1` 给 `llm.complete`，主进程 reader 收到一个 `id=1` 的响应时，不知道该派发给 `brain.run` 的 waiter 还是 `llm.complete` 的 waiter。

MCP 规范对这个问题含糊其辞，各家实现都各自发明解法。本规格**精确定义** id 命名空间的分配规则。

### 4.2 id 格式：string-prefix tuple

v1 协议规定：**所有 id MUST 是字符串类型**，**MUST** 带前缀，格式为 `"<origin>:<seq>"`：

| 发起方 | 前缀 | 示例 |
|---|---|---|
| 主进程（Kernel） | `k` | `"k:1"`, `"k:2"`, `"k:37"` |
| sidecar | `s` | `"s:1"`, `"s:2"`, `"s:88"` |

**为什么用字符串而不是数字**：JSON-RPC 2.0 允许 id 是 string / number / null 三种类型。数字在 JSON 里序列化成 `1` 或 `2`，跨语言时浮点精度会变成隐形陷阱（JavaScript 没有 int64，大数字会丢精度）。string 绝对安全，而且 tcpdump/jq 调试时可以一眼看出消息源头。

**seq 分配规则**：每个方向独立维护自己的单调递增计数器，从 `1` 开始。seq 是 64 位整数，不会在一次 sidecar 生命周期内耗尽。sidecar 重启后 seq 归零（新的 initialize 会话是独立上下文）。

### 4.3 响应匹配规则

收到响应时，对端 **MUST** 按以下规则派发：

1. 读 id 的前缀
2. 如果前缀 = 自己的前缀（`k` 对主进程，`s` 对 sidecar），说明这是**对自己发出的请求的响应**，派发给对应 waiter
3. 如果前缀 = 对端的前缀，说明这是**对端发来的新请求**，路由到 server handler
4. 如果前缀未知，**MUST** 回 `-32600 Invalid Request` 并写审计

### 4.4 Notification 的 id 约定

Notification 的 id **MUST** 缺失或为 JSON `null`。**MUST NOT** 给 notification 分配带前缀的 id（即使对端忽略响应，分配 id 会让 seq 计数器失步）。

### 4.5 迟到响应处理

场景：主进程对 sidecar 发了 `id=k:42` 的请求，设置 30 秒超时。30 秒后主进程认为超时，本地清理 waiter，把对应的 request 标记为 `abandoned`。但 35 秒后 sidecar 才回 `id=k:42` 的响应。

**规定行为**（MUST）：

1. 主进程收到 `k:42` 的响应后，查 waiter 表——没有，说明已 abandoned
2. **MUST** 丢弃响应内容（不回调任何业务逻辑）
3. **MUST** 写 `trace.emit level=warn event=stale_response_received id=k:42 latency=35s`
4. **MUST** 累加 sidecar 实例的 `stale_response_count`
5. 连续 stale > 阈值（默认 10 次 / 分钟）**MUST** 触发熔断：该 sidecar 实例状态转 `degraded`，不再派发新 Run 但允许当前 Run 跑完

同样规则对称适用于 sidecar 侧（sidecar 对主进程发请求、主进程迟到响应）。

### 4.6 in-flight 窗口

每个方向的 in-flight（已发送未响应）请求数有硬上限：

| 方向 | 默认上限 | 触发上限时的行为 |
|---|---|---|
| 主 → 子 | 由 `SubprocessSpec.MaxConcurrentRun × 8` 决定，默认 32 | 主进程端本地 block，writer channel 满则等待 |
| 子 → 主 | 固定 64 | sidecar SDK **MUST** 本地排队；超过排队上限（默认 256）**MUST** 返回 `brain_overloaded` 给业务代码 |

**理由**：反向 RPC 的 64 窗口足够覆盖"一个 Agent Loop 步里并发发起几十个 tool.invoke + 几个 trace.emit"的合理场景，同时防止恶意或失控 sidecar 用无限并发打爆主进程 goroutine 池。

### 4.7 `$/cancelRequest` 的精准取消

对方发来的请求如果还没来得及处理，接收方可以发 `$/cancelRequest {id: "k:42"}` 通知要求对方取消。接收 `$/cancelRequest` 的规则（MUST）：

1. 查 waiter 表，如果 id 存在 → 调对应 handler 的 ctx.Cancel
2. 如果 id 不存在（已完成或从未见过）→ 静默忽略
3. **MUST** 在后续的响应（如果还会发响应）里把 `error.code` 设为 `-32800`（保留给 cancelled）并在 `error.message` 写 `"request cancelled by peer"`

## 5. 背压与流控

### 5.1 pipe buffer 饱和问题

Linux 管道的默认 buffer 是 64 KiB（`/proc/sys/fs/pipe-max-size` 可调）。当主进程向 sidecar 的 stdin 写入速度 > sidecar 读取速度时，stdin buffer 会满，`write()` 系统调用会 block。此时如果主进程有多个 goroutine 都试图写 stdin，会**全部卡死**，形成死锁链。

### 5.2 解决方案：writer channel + 超时丢弃

§2.7 已规定 writer 必须单例化。本节补充超时策略：

- writer goroutine 从 channel 取出帧后调用 `w.Write(frame)`，**MUST** 在外层包一个 `SetWriteDeadline`（如果底层实现支持）或 `select-with-timeout`
- 默认 write_timeout = 10 秒
- 超时 → writer goroutine 判定 sidecar **hung**，立即走以下序列：
  1. 关 writer channel（拒绝新帧入队）
  2. 对所有 pending waiter 返回 `sidecar_hung` 错误
  3. 对 SidecarInstance 发 SIGTERM
  4. 走半关闭状态机（§6）

### 5.3 反向 RPC 的内部排队

sidecar SDK 实现反向 RPC 时（见 02 §12.5.6 brain-sdk-go），**MUST** 提供两级队列：

1. **In-flight 表**（64 容量，见 §4.6）：已发送未响应的请求
2. **Pending 队列**（256 容量）：等待进入 in-flight 的请求

生产者（大脑业务代码）调 `sdk.llm.Complete(ctx, req)` 时：

- 如果 in-flight < 64 → 直接发，加入 in-flight 表
- 如果 in-flight = 64 → 进 pending 队列，等待某个 in-flight 完成后晋升
- 如果 pending = 256 → **MUST** 立即返回错误 `brain_overloaded`（不 block 业务 goroutine，让业务代码决定降级）

这条规则防止业务代码无限 spawn goroutine 把 SDK 内部打爆。

### 5.4 pipe buffer 扩容（可选优化）

Linux 上 sidecar 和主进程启动后 **MAY** 调 `fcntl(fd, F_SETPIPE_SZ, 1048576)` 把 pipe buffer 扩到 1 MiB。扩容能大幅降低高吞吐场景的 syscall 次数。扩容失败（权限不足或 kernel 不支持）**MUST NOT** 报错，静默回落到默认 64 KiB。

## 6. 半关闭与生命周期状态机

### 6.1 问题陈述

stdio 协议的关闭远比看起来复杂：

- stdin 的 EOF 和 stdout 的 EOF 是**两个独立事件**
- 进程 exit 和 fd 关闭是**两个独立事件**，可以任意顺序发生
- 主进程 kill sidecar 时可能留下 zombie（`os/exec.Cmd` 必须 `Wait()` 才能回收）
- in-flight 请求在关闭期间必须有明确的归宿（成功 / 失败 / abandoned）

v1 规定一个精确的 **5 态状态机**覆盖所有关闭路径。

### 6.2 状态枚举

```go
type SidecarState int

const (
    StateStarting SidecarState = iota  // spawn 后、initialize 成功前
    StateRunning                        // initialize 完成，正常服务
    StateDraining                       // 收到 shutdown 或 cancel，等 in-flight 清空
    StateClosed                         // stdin 已关闭，不再发送新帧
    StateWaited                         // 子进程已 exit，等待 OS 回收
    StateReaped                         // os.Process.Wait() 成功返回，完全清理
)
```

### 6.3 状态转换图

```
              spawn
                │
                ▼
         ┌──────────┐
         │ Starting │
         └────┬─────┘
              │ initialize 成功
              ▼
         ┌──────────┐
         │ Running  │ ──┐
         └────┬─────┘   │
              │          │
 shutdown /   │          │ stdin/stdout 意外 EOF
 idle_timeout │          │ 或 frame_too_large
 或熔断       │          │ 或 write_timeout
              │          │
              ▼          │
         ┌──────────┐    │
         │ Draining │ ◄──┘
         └────┬─────┘
              │
      in-flight 为 0
      或 drain_timeout
              │
              ▼
         ┌──────────┐
         │  Closed  │  ← stdin 关闭
         └────┬─────┘
              │
      process exit
      或 SIGTERM 超时 → SIGKILL
              │
              ▼
         ┌──────────┐
         │  Waited  │
         └────┬─────┘
              │
      os.Process.Wait() 返回
              │
              ▼
         ┌──────────┐
         │  Reaped  │
         └──────────┘
```

### 6.4 各状态的规则

#### Starting
- 允许接受 `initialize` 的响应
- **MUST NOT** 发送任何业务请求
- 超时上限：30 秒。超时 → 直接 SIGTERM 跳到 Waited

#### Running
- 正常双向 RPC
- 所有业务请求都允许
- `health.ping` 每 30 秒由主进程发送

#### Draining
- **MUST NOT** 再派发新 `brain.run`（返回 `-32099 shutting_down` 给上层调度器）
- 已有 `brain.run` 允许跑完
- 主进程停止向 sidecar 发心跳（不需要了）
- sidecar 的反向 RPC（`llm.complete` / `tool.invoke` / 等）**仍然可用**，主进程照常服务
- 超时上限：`drain_timeout`，默认 60 秒。超时 → 对所有 in-flight `brain.run` 发 `brain.cancel`，进入 Closed

#### Closed
- 主进程调 `stdin.Close()`，sidecar 读到 EOF
- sidecar SDK **MUST** 在收到 EOF 后立即进入自身的 drain → exit 流程
- 主进程 reader goroutine 继续读 stdout，直到：
  - 读到 EOF（sidecar 进程 exit） → 进入 Waited
  - 超时 5 秒 → 发 SIGTERM，继续等
  - 再超时 5 秒 → 发 SIGKILL，进入 Waited

#### Waited
- 主进程调 `cmd.Wait()` 回收 zombie，释放 fd
- **MUST** 在这一步清理所有资源：
  - 关闭 reader/writer goroutine
  - 关闭 stdin/stdout/stderr 三个 fd
  - 清空 waiter 表，对任何残留 waiter 返回 `sidecar_dead` 错误
- 进入 Reaped

#### Reaped
- 终态。SidecarInstance 可以从 SidecarPool 移除

### 6.5 并发安全的状态转换

SidecarInstance 的状态字段 **MUST** 用原子 CAS 或 mutex 保护。状态转换函数签名建议：

```go
func (s *SidecarInstance) transitionTo(to SidecarState) error {
    s.mu.Lock()
    defer s.mu.Unlock()
    if !isValidTransition(s.state, to) {
        return fmt.Errorf("invalid transition: %v → %v", s.state, to)
    }
    old := s.state
    s.state = to
    s.stateChangedAt = time.Now()
    s.emitStateEvent(old, to)
    return nil
}
```

非法转换（如 `Reaped → Running`）**MUST** 返回错误且 **MUST NOT** 修改状态。这是一条硬护栏，防止并发 bug 让状态机进入不可能的状态。

### 6.6 Zombie 回收的 MUST 条款

**每一个 spawn 出来的 sidecar 进程 MUST 最终经过 `os.Process.Wait()`**，没有例外。Go 的 `exec.Cmd` 如果不调 `Wait()`，子进程会变成 zombie 直到父进程退出，Linux 下 PID 表会累计泄漏。

推荐实现：

```go
func (s *SidecarInstance) cleanup() {
    close(s.writeCh)              // 1) 关闭 writer channel
    s.stdin.Close()                // 2) 关闭 stdin，触发 sidecar EOF
    s.readerDone.Wait()            // 3) 等 reader goroutine 退出
    s.cmd.Wait()                   // 4) 必须：回收 zombie
    metrics.ProcessReaped.Inc()
    s.transitionTo(StateReaped)
}
```

### 6.7 孤儿进程防护

主进程自身崩溃时，所有 sidecar 成为孤儿进程（parent 变成 init/systemd）。为了避免孤儿进程继续烧 LLM token 或占 CPU，sidecar 侧 **MUST** 实现"父进程监控"：

**Linux 实现**：sidecar SDK 在 initialize 后立即调 `prctl(PR_SET_PDEATHSIG, SIGTERM)`。当父进程 exit 时，内核会对 sidecar 自动发 SIGTERM，sidecar SDK 捕获后走自杀流程。

**macOS 实现**：没有 PR_SET_PDEATHSIG，sidecar **MUST** 开一个 heartbeat goroutine，每 2 秒 `kill(getppid(), 0)` 探活；返回 ESRCH 或 ppid 变成 1 → 父进程死了 → 自杀。

**Windows 实现**：v1 不支持（见 02 §12.5.1 代价表），v2.1 通过 Job Object 实现。

## 7. 心跳与 keep-alive

### 7.1 两层心跳

v1 区分**协议层心跳**和**业务层心跳**：

| 层 | 方法 | 频率 | 失败阈值 | 失败后果 |
|---|---|---|---|---|
| 协议层 | `$/ping` (notification) | 10 秒 | 3 次连续无响应 | 判定 sidecar hung，进入 Draining |
| 业务层 | `health.ping` (request) | 30 秒 | 3 次连续无响应 | 判定 sidecar dead，触发重启 |

**为什么要两层**：`$/ping` 是轻量 notification，即使 sidecar 正在处理复杂业务也应该有独立的协议层 goroutine 回 `$/pong`。如果 `$/ping` 也挂了，说明进程真的 hung（死锁 / 无限循环 / kernel blocking）。`health.ping` 是业务层，要求返回 `active_runs` / `memory_rss` / `last_error` 等业务字段，比 `$/ping` 重。

### 7.2 `$/ping` 格式

主进程发：
```json
{"jsonrpc":"2.0","method":"$/ping","params":{"seq":42}}
```

sidecar 回：
```json
{"jsonrpc":"2.0","method":"$/pong","params":{"seq":42}}
```

`seq` 字段用于主进程识别响应匹配，因为 notification 没有 id。sidecar **MUST** 原样回带 seq。

### 7.3 健康降级

v1 引入**三态健康**（对技术审阅报告 A.2 的回应）：

- `healthy`：`$/ping` 和 `health.ping` 都正常
- `degraded`：`$/ping` 正常但 `health.ping` 超时 1-2 次，或 stale_response_count 超阈值；**不再派发新 Run**，但 in-flight Run 继续
- `quarantined`：连续 3 次 `health.ping` 失败，或 `$/ping` 超时，或 frame_too_large；立即 Draining

## 8. 二进制与大 payload 传输

### 8.1 硬规则：大对象 MUST 走 content_ref

Body 上限 16 MiB（§2.4）意味着**任何单个对象超过 1 MiB（留出 JSON 序列化 + 其他字段的余量）MUST 走 content_ref 间接引用**，不允许内联 base64。

典型场景：
- 截图（一张 PNG 可能 300 KB ~ 3 MiB）
- 视频片段（几 MB ~ 几十 MB）
- 大 diff（超过 1 MiB 的多文件 diff）
- 大 tool_result 文本（超过 1 MiB 的文件内容）

### 8.2 内联阈值

| payload | 推荐内联（base64） | 必须走 content_ref |
|---|---|---|
| 小截图（缩略图） | < 64 KiB | ≥ 64 KiB |
| tool_result 文本 | < 256 KiB | ≥ 256 KiB |
| diff | < 128 KiB | ≥ 128 KiB |
| 任何二进制 | < 64 KiB | ≥ 64 KiB |

**硬上限**：任何字段 **MUST NOT** 内联超过 1 MiB 的 base64。超限由主进程在解析时拒绝（`frame_payload_too_large` 错误）。

### 8.3 Upload Session 协议

走 content_ref 的流程通过**三段式 upload session** 完成：

```
sidecar                             main process
   │                                     │
   │  artifact.beginUpload               │
   │  {run_id, kind, mime, size,         │
   │   sha256_hex}                       │
   │────────────────────────────────────►│
   │                                     │
   │                      {upload_id,    │
   │                       write_path}   │
   │◄────────────────────────────────────│
   │                                     │
   │  (sidecar 直接写 write_path         │
   │   用 os.WriteFile 或流式)           │
   │                                     │
   │  artifact.commitUpload              │
   │  {upload_id}                        │
   │────────────────────────────────────►│
   │                                     │
   │                 (主进程校验 sha256  │
   │                  + 移入最终位置)    │
   │                                     │
   │                      {artifact_ref} │
   │◄────────────────────────────────────│
```

**MUST 条款**：

1. `beginUpload.params` **MUST** 包含预计算的 `sha256_hex`（hex 编码的 sha256 摘要）
2. 主进程返回的 `write_path` **MUST** 在主进程控制的临时目录下（`~/.easymvp-exec/uploads/<run_id>/<upload_id>/blob`）
3. sidecar 写完后 **MUST** 发 `commitUpload`，**MUST NOT** 遗留 pending upload
4. 主进程在 `commitUpload` 收到时 **MUST** 重算 sha256 并与 `beginUpload` 提供的值对比；不匹配 **MUST** 拒绝 commit 并触发 Guardrail 审计
5. 主进程对未 commit 的 upload session 有 5 分钟 GC 超时，超时自动清理 write_path

**Atlas 模式扩展**：多机部署时 `write_path` 替换为 `s3_presigned_url` 或 `minio_presigned_url`，sidecar 直接 PUT 到对象存储，commit 时主进程校验 sha256（从对象存储下载头部验证）。

### 8.4 下载（反向：主进程 → sidecar）

有些场景需要主进程把大 artifact 发给 sidecar（例如 verifier_brain 要读 code_brain 产出的 diff）。协议对称：

```
sidecar                             main process
   │                                     │
   │  artifact.requestRead               │
   │  {artifact_id}                      │
   │────────────────────────────────────►│
   │                                     │
   │               {read_path,           │
   │                sha256_hex, size}    │
   │◄────────────────────────────────────│
   │                                     │
   │  (sidecar 从 read_path 读取         │
   │   并自检 sha256)                    │
```

主进程 **MUST** 确保 `read_path` 在 sidecar 生命周期内可访问（不会被 GC 回收）。sidecar 读完后 **SHOULD** 发 `artifact.releaseRead {artifact_id}` 通知主进程可以清理临时文件。

## 9. 错误帧编码

### 9.1 JSON-RPC 2.0 error object 的限制

标准 JSON-RPC 2.0 的 error 对象是：

```json
{"code": -32601, "message": "Method Not Found", "data": <any>}
```

v1 对 `data` 字段做强约束，使其携带 BrainKernel 错误模型（详见 21-错误模型.md）的完整信息：

```json
{
  "code": -32000,
  "message": "tool_execution_failed: code.run_tests exit 1",
  "data": {
    "class": "Transient",
    "retryable": true,
    "error_code": "tool_execution_failed",
    "fingerprint": "a3f2...",
    "trace_id": "01HQ...",
    "span_id": "5e2b...",
    "cause": {
      "class": "Permanent",
      "error_code": "test_failure",
      "message": "3 tests failed in auth_test.go"
    },
    "suggestions": [
      "Check that all auth_test.go dependencies are installed",
      "Run `go test ./auth -v` locally to reproduce"
    ]
  }
}
```

字段规格详见 [21-错误模型.md](./21-错误模型.md) §3。本文档只约束 **wire format 层面** 的 MUST 条款：

1. `data` **MUST** 是 JSON object（不是 string / number / null）
2. `data.class` **MUST** 是 21-错误模型 §3.2 定义的四分类之一
3. `data.retryable` **MUST** 是 boolean
4. `data.error_code` **MUST** 是 snake_case ASCII 字符串
5. `data.fingerprint` **SHOULD** 是 16 位 hex 字符串（用于日志聚合）
6. `data.trace_id` **SHOULD** 遵循 W3C Trace Context 格式

### 9.2 预定义 code 范围

| code 范围 | 用途 |
|---|---|
| `-32700 ~ -32600` | JSON-RPC 2.0 标准错误（Parse error / Invalid Request / Method not found / Invalid params / Internal error） |
| `-32099 ~ -32000` | BrainKernel v1 保留（`-32099` = `shutting_down`，`-32098` = `frame_too_large`，`-32097` = `sidecar_hung`，`-32096` = `brain_overloaded`，`-32000` = 通用业务错误，细分看 `data.error_code`） |
| `-32800` | 请求被取消（对应 `$/cancelRequest`） |

新增保留 code 需更新本文档并在 [21-错误模型.md](./21-错误模型.md) 同步，**MUST NOT** 在 wire 上私自使用新 code。

## 附录 A · 帧格式 BNF

```bnf
frame           = header-block  CRLF  body
header-block    = header-line *( header-line )
header-line     = header-name ":" OWS header-value OWS CRLF
header-name     = "Content-Length" / "Content-Type"
header-value    = token / quoted-string
body            = *OCTET  ; exactly Content-Length bytes, UTF-8 JSON

CRLF            = %x0D.0A
OWS             = *( SP / HTAB )
SP              = %x20
HTAB            = %x09
OCTET           = %x00-FF
```

**说明**：
- `header-block` 以独立的 CRLF 结尾（即连续两个 CRLF 分隔 header 和 body）
- `body` 是精确 `Content-Length` 字节，不多不少
- header-line 数量不设上限，但总 header-block 大小 MUST < 16 KiB

## 附录 B · 与 LSP / MCP / DAP 的对照表

| 维度 | BrainKernel v1 | LSP 3.17 | MCP 2024-11 | DAP |
|---|---|---|---|---|
| 帧格式 | Content-Length 头部帧 | Content-Length 头部帧 | Content-Length 头部帧 | Content-Length 头部帧 |
| Content-Type | `application/vscode-jsonrpc; charset=utf-8` | 同左 | 同左 | 同左 |
| 帧上限 | 16 MiB | 未强制 | 未强制 | 未强制 |
| 协议基础 | JSON-RPC 2.0 | JSON-RPC 2.0 | JSON-RPC 2.0 | 自定义（类 JSON-RPC） |
| Batch | 禁用 | 禁用 | 禁用 | N/A |
| id 类型 | string-prefix tuple | int / string | int / string | int |
| id 命名空间 | 主 `k:*` / 子 `s:*` | 混用单命名空间 | 混用单命名空间 | 单向 client→server |
| 反向 RPC | 双向全双工 | 双向全双工 | 有限（sampling / roots） | 有限（event） |
| `$/` 元方法 | MUST 实现 `$/ping` `$/cancelRequest` `$/progress` | MUST 实现 `$/cancelRequest` `$/progress` | MAY | N/A |
| 状态机 | 5 态（Starting / Running / Draining / Closed / Waited / Reaped） | 未强制 | 未强制 | 未强制 |
| 大 payload | upload session + sha256 | resource URI | resource URI | 不涉及 |
| 心跳 | 两层（`$/ping` + `health.ping`） | 无 | `ping` | 无 |
| 合规测试 | §11 Contract Tests（v1 MUST） | 官方 TCK | 官方 TCK | 官方 adapter tests |

**关键差异**：v1 相比 LSP / MCP 更严格地规定了帧上限（16 MiB）、id 命名空间、状态机、两层心跳、upload session 协议。这些差异都是为了支持**长 running AI agent 任务**和**第三方大脑跨语言实现**的特殊需求。
