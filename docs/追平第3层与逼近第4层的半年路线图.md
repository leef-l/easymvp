# 追平第3层与逼近第4层的半年路线图

## 一、文档目标

本文档定义 EasyMVP 从当前阶段出发，在未来 6 个月内：

1. 追平“第 3 层：半自治项目执行层”
2. 在局部能力上逼近“第 4 层：自治项目管理层”

文档重点不是写愿景，而是给出可施工、可验收、可取舍的推进路线。

---

## 二、分层目标回顾

### 2.1 第 3 层：半自治项目执行层

标志：

1. 方案确认后可自动推进大部分项目
2. 自动审核、自动执行、自动返工
3. 人主要在关键节点介入
4. 具备稳定控制面和执行治理

### 2.2 第 4 层：自治项目管理层

标志：

1. 系统不只会执行，还会做重规划
2. 能基于失败、风险、资源状况自动调整路线
3. 自动判断何时返工、何时升级、何时暂停
4. 人主要负责目标、预算、战略裁决

---

## 三、当前起点判断

截至 2026-04-05，当前代码库已经具备：

1. WorkflowRun 分阶段主链
2. Review 阶段任务化
3. Execute 阶段迁移
4. 基础 runtime 控制面
5. V2 Pause/Resume/Retry/Skip 主链
6. Dashboard 与 review workspace 的基础前端

当前尚未补齐：

1. Execution Console 完整控制面
2. watchdog v2
3. rework / complete 阶段闭环
4. 任务级环境隔离
5. 多执行器统一抽象
6. 自动验收 / 自动重规划

因此当前整体大致处于：

- 第 2 层完成
- 第 3 层初步成型

---

## 四、半年总策略

总体策略：

1. 先补底座
2. 再补执行治理
3. 再补返工与验收
4. 最后做自治管理

不建议的推进方式：

1. 同时做所有执行器
2. 同时做权限、前端、隔离、自治 PM
3. 把“平台化”优先于“主链稳定”

建议优先级：

1. WorkflowRun 主链完全替代旧链
2. git worktree 隔离
3. 执行器统一抽象
4. rework / complete / event stream
5. 自动验收
6. 自动重规划

---

## 五、6 个月路线图总览

### 第 1 个月：WorkflowRun 主链彻底收口

目标：

1. 完成 M4 收口
2. 让 `workflow_v2` 达到可稳定替代旧链的门槛

核心任务：

1. Execution Console 最小版
2. watchdog v2 与 `domain_task` 对齐
3. `review -> execute` 失败一致性继续收紧
4. 项目列表 / Dashboard / Review / Execute 状态统一
5. 新项目默认走 V2 的灰度条件定义

验收标准：

1. 新项目不再依赖旧 `mvp_task` 主链
2. 执行阶段可视化可用
3. 失败、暂停、恢复、重试行为稳定

### 第 2 个月：任务级环境隔离落地

目标：

1. 引入 `git worktree` 隔离
2. 先服务 `workflow_v2 + aider`

核心任务：

1. `mvp_task_workspace`
2. `workspace/` 模块
3. worktree create / cleanup
4. Aider 切到隔离工作区
5. 失败保留 / 成功清理策略

验收标准：

1. Aider 写仓任务不直接污染主工作区
2. 每个任务可追踪独立 workspace
3. 清理和异常处理可用

### 第 3 个月：执行器平台化

目标：

1. 从“多分支 if/switch 执行”升级到统一执行器平台

核心任务：

1. `ExecutorRegistry`
2. `ExecutorAdapter`
3. Aider 迁入统一抽象
4. OpenHands 正式接入
5. Claude Code / Codex CLI / Gemini CLI 预留标准接口

验收标准：

1. 新执行器接入不再改调度器主逻辑
2. 执行器日志、失败、取消模型统一
3. 至少 Aider + OpenHands 两个正式执行器可用

### 第 4 个月：返工链与完成阶段闭环

目标：

1. 从“执行完就停”升级到真正闭环

核心任务：

1. ReworkStageService
2. bug analysis / failure analysis
3. handoff record
4. complete stage
5. timeline / event feed

验收标准：

1. 返工不再依赖临时命名规则
2. 执行失败可自动进入返工链
3. 项目完成阶段可输出总结

### 第 5 个月：自动验收能力

目标：

1. 让系统不只会执行，还能判断“做得对不对”

核心任务：

1. 自动验收规则模型
2. 项目级 acceptance checklist
3. 代码任务自动检查
4. 文档/创作/分析类任务自动验收模板
5. 验收失败自动进入返工链

验收标准：

1. 任务完成不再只看状态
2. 项目进入 complete 前必须经过验收
3. 验收失败有结构化原因

### 第 6 个月：自治项目管理初版

目标：

1. 在局部能力上逼近第 4 层

核心任务：

1. 自动重规划
2. 风险升级策略
3. 资源与执行器选择策略
4. 项目自治汇报机制
5. 异常场景自动暂停 / 自动建议

验收标准：

1. 系统能根据执行结果调整后续路线
2. 系统能自动输出项目级周报/阶段报告
3. 人只在关键裁决点介入

## 七、每阶段的风险与克制点

### 7.1 第 1-2 个月风险

风险：

1. 主链还没稳就叠隔离层
2. 前端和后端推进不同步

克制点：

1. 不要同时接多执行器
2. 先只服务 Aider

### 7.2 第 3-4 个月风险

风险：

1. 执行器抽象过度设计
2. OpenHands 接入拖慢主线

克制点：

1. 先保证 Aider 迁移
2. OpenHands 先能跑，不追求一步到位平台化

### 7.3 第 5-6 个月风险

风险：

1. 自动验收规则过于理想化
2. 自治管理层过早做成“黑盒决策”

克制点：

1. 验收先做显式规则
2. 重规划先做建议型，再做自动执行型

### 8.1 至少需要的角色

1. 后端主架构负责人
2. 执行与调度负责人
3. 前端控制台负责人

1. 一人负责 WorkflowRun 主链

## 九、里程碑判断标准

### 9.1 追平第 3 层的标准

达到以下条件，才算真正追平第 3 层：

1. 新项目主链稳定运行
2. 自动审核 + 自动执行 + 自动返工初步可用
3. 任务隔离已接入
4. 多执行器已有统一抽象
5. 执行过程可观察、可暂停、可恢复、可重试

### 9.2 逼近第 4 层的标准

达到以下条件，才算开始逼近第 4 层：

1. 具备自动验收
2. 具备自动重规划
3. 能根据失败与风险自动调整路线
4. 人类只在关键策略点介入

---

## 十、成功概率判断

### 10.1 有机会做到的部分

1. 6 个月追平第 3 层
2. 6 个月在局部能力上逼近第 4 层

### 10.2 6 个月内不现实的部分

1. 完整达到第 5 层
2. 任意开放型项目完全无人监管
3. 全自动商业化稳定交付所有类型项目

---

## 十一、结论

这条路线真正重要的不是“多接几个 Agent”，而是完成三次升级：

1. 从工作流系统升级成稳定执行底座
2. 从稳定执行底座升级成多执行器平台
3. 从多执行器平台升级成自治项目系统

未来 6 个月最现实、也最有胜率的目标是：

1. 先把 WorkflowRun 做成稳定可替代旧链的底座
2. 用 worktree 和执行器抽象补齐执行治理
3. 再通过自动验收和自动重规划，开始逼近第 4 层

这条路线是可实现的，但前提是始终围绕主线推进，不被“什么都想一起做”拖散。
