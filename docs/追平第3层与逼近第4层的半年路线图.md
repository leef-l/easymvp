# 追平第3层与逼近第4层的半年路线图

## 一、文档目标

本文档定义 EasyMVP 从当前阶段出发，在未来 6 个月内：

1. 追平“第 3 层：半自治项目执行层”
2. 在局部能力上逼近“第 4 层：自治项目管理层”

文档重点不是写愿景，而是给出可施工、可验收、可取舍的推进路线。

---

## 二、分层目标回顾

### 2.1 第 3 层：半自治项目执行层

标志：

1. 方案确认后可自动推进大部分项目
2. 自动审核、自动执行、自动返工
3. 人主要在关键节点介入
4. 具备稳定控制面和执行治理

### 2.2 第 4 层：自治项目管理层

标志：

1. 系统不只会执行，还会做重规划
2. 能基于失败、风险、资源状况自动调整路线
3. 自动判断何时返工、何时升级、何时暂停
4. 人主要负责目标、预算、战略裁决

## 四、半年总策略

总体策略：

1. 先补底座
2. 再补执行治理
3. 再补返工与验收
4. 最后做自治管理

不建议的推进方式：

1. 同时做所有执行器
2. 同时做权限、前端、隔离、自治 PM
3. 把“平台化”优先于“主链稳定”

建议优先级：

1. WorkflowRun 成为新项目默认主链（legacy 保持兼容）
2. git worktree 隔离
3. 执行器统一抽象
4. rework / complete / event stream
5. 自动验收
6. 自动重规划

## 七、每阶段的风险与克制点

### 7.1 第 1-2 个月风险

风险：

1. 主链还没稳就叠隔离层
2. 前端和后端推进不同步

克制点：

1. 不要同时接多执行器
2. 先只服务 Aider

### 7.2 第 3-4 个月风险

风险：

1. 执行器抽象过度设计
2. OpenHands 接入拖慢主线

克制点：

1. 先保证 Aider 迁移
2. OpenHands 先能跑，不追求一步到位平台化

### 7.3 第 5-6 个月风险

风险：

1. 自动验收规则过于理想化
2. 自治管理层过早做成“黑盒决策”

克制点：

1. 验收先做显式规则
2. 重规划先做建议型，再做自动执行型

## 十一、结论

这条路线真正重要的不是“多接几个 Agent”，而是完成三次升级：

1. 从工作流系统升级成稳定执行底座
2. 从稳定执行底座升级成多执行器平台
3. 从多执行器平台升级成自治项目系统

未来 6 个月最现实、也最有胜率的目标是：

1. 先把 WorkflowRun 做成新项目默认主链（legacy 保持兼容运行）
2. 用 worktree 和执行器抽象补齐执行治理
3. 再通过自动验收和自动重规划，开始逼近第 4 层

这条路线是可实现的，但前提是始终围绕主线推进，不被“什么都想一起做”拖散。
