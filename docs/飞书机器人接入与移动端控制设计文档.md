# 飞书机器人接入与移动端控制设计文档

> 更新日期：2026-04-08
>
> 说明：虽然文件名保留为“飞书”，但当前实现已经同时覆盖飞书和 Telegram 协作能力。

## 1. 定位

当前协作层的职责很明确：

- 把工作流事件推送到 IM 平台
- 把人工干预入口带到移动端
- 把平台用户与系统用户做绑定
- 让审批、测试消息、菜单配置等能力有统一后台入口

业务主逻辑仍在 EasyMVP 后端，不迁移到 IM 平台里执行。

## 2. 当前已落地能力

### 2.1 飞书

当前已经有：

- 飞书配置查询与保存
- 飞书用户绑定 / 解绑
- 测试消息
- 机器人菜单配置
- 群菜单创建、查询、删除
- Webhook / WebSocket 两种接入模式

### 2.2 Telegram

当前已经有：

- Telegram Bot 配置查询与保存
- Telegram 用户绑定 / 解绑
- 测试消息
- Bot 命令菜单配置
- Polling 运行状态

### 2.3 工作流通知

当前通知器会覆盖这类事件：

- AI 回复推送
- 任务失败通知
- 审核待处理
- 验收待处理
- 检查点通知
- 项目完成通知

## 3. 后端结构

当前相关代码主要分布在：

- `admin-go/app/mvp/internal/collab/adapter`
- `admin-go/app/mvp/internal/collab/notifier`
- `admin-go/app/mvp/internal/collab/service`
- `admin-go/app/mvp/internal/controller/chat/workflow_feishu.go`
- `admin-go/app/mvp/internal/controller/chat/workflow_telegram.go`
- `admin-go/app/mvp/internal/controller/chat/feishu_callback.go`
- `admin-go/app/mvp/internal/controller/chat/tg_polling.go`

其中：

- 飞书适配器负责发消息、菜单、回调验证和长连接
- Telegram 适配器负责消息、命令和轮询
- 通知服务负责把工作流事件转成面向用户的消息

当前用户绑定落在：

- `mvp_user_collab_binding`

## 4. 前端入口

当前前端已经提供独立协作页：

- `views/mvp/workflow/feishu.vue`
- `views/mvp/workflow/telegram.vue`

工作流仪表盘也直接提供跳转入口，方便从项目运行态进入协作配置。

## 5. 运行方式

### 5.1 飞书

当前飞书支持两种接入模式：

- `webhook`
- `websocket`

配置字段包括：

- `appId`
- `appSecret`
- `verificationToken`
- `encryptKey`
- `defaultNotifyUserIds`
- `connectionMode`

### 5.2 Telegram

当前 Telegram 以 Bot Token + Polling 为主，配置字段包括：

- `enabled`
- `botToken`
- `botRunning`

## 6. 与工作流的关系

协作层现在已经和 Workflow V2 打通，但定位仍然是“控制面延伸”：

- 它可以发通知
- 它可以承接人工动作
- 它可以暴露快捷菜单

但它不会替代：

- 工作流状态机
- 执行器调度
- 审核/验收核心逻辑

换句话说，IM 平台是入口和触达层，真正的业务状态推进仍然回到 MVP 后端。

## 7. 当前边界

更新本文档时确认到的边界如下：

- 当前一等支持的平台是飞书和 Telegram
- 文档不再保留“钉钉 / 企业微信 / Slack 未来接入”的方案段落
- 协作层的职责是通知、绑定、菜单和人工入口，不承担主工作流执行内核
