# 飞书机器人接入与移动端控制设计文档

## 一、文档目标

本文档定义 EasyMVP 将项目运行态能力接入飞书，并通过手机 App 完成通知、审批、人工干预、阶段报告查看的完整落地方案。

本文档不是“只做飞书适配”的一次性方案，而是要求：

1. 第一阶段以飞书为正式落地平台
2. 架构层预留钉钉、企业微信等同类平台扩展口
3. 业务主逻辑仍保留在 EasyMVP 内部
4. 飞书仅承担移动端入口、通知触达、审批交互和轻操作控制

## 二、范围与边界

### 2.1 第一阶段范围

第一阶段接入飞书后，支持以下能力：

1. 项目状态通知
2. 审核/验收审批
3. 异常告警
4. 返工/重试/重验触发
5. 阶段报告 / 项目周报推送
6. 手机端查看项目关键状态

### 2.2 不在第一阶段范围内

1. 不将 WorkflowRun 主逻辑迁入飞书
2. 不将执行器调度托管给飞书
3. 不依赖飞书 AI 托管能力作为主执行引擎
4. 不同时接入多个 IM 平台

### 2.3 核心原则

1. EasyMVP 是执行内核
2. 飞书是移动端控制面入口
3. 平台适配层必须可扩展
4. 所有审批和控制动作都必须回到 EasyMVP 后端落库

## 三、为什么第一阶段选飞书

按当前项目形态，飞书优于钉钉和企业微信的原因：

1. 移动端交互体验更适合项目机器人
2. 卡片消息能力更适合状态展示和审批
3. 机器人/应用接入能力成熟
4. 更适合作为“项目控制台的移动延伸”

但架构上不能把飞书写死，因此要抽象出统一“外部协作平台”接口。

## 四、总体方案

总体架构如下：

1. EasyMVP 内部事件源
2. 平台通知编排层
3. 平台适配器层
4. 飞书应用 / 机器人
5. 手机端飞书 App

工作方式：

1. WorkflowRun / Accept / Rework / Complete 产生业务事件
2. 事件进入统一的 `CollabPlatformService`
3. `CollabPlatformService` 根据订阅配置选择平台
4. 飞书适配器将内部事件转换为飞书卡片/消息
5. 用户在飞书手机端点击动作
6. 飞书回调 EasyMVP 后端
7. EasyMVP 校验权限并执行真实业务动作

## 五、平台抽象设计

这是本文档最关键的部分。

不能直接在业务层写：

- `sendFeishuMessage()`
- `handleFeishuApprove()`

必须统一抽象为“协作平台接入层”。

### 5.1 统一平台枚举

建议平台枚举：

- `feishu`
- `dingtalk`
- `wecom`

后续如需扩展可继续增加：

- `telegram`
- `slack`

### 5.2 平台能力模型

统一定义平台能力：

1. `notify`
   - 纯消息推送
2. `interactive_card`
   - 支持卡片和按钮交互
3. `approval_action`
   - 支持审批型动作回调
4. `deep_link`
   - 支持跳转到 Web 控制台
5. `user_mapping`
   - 支持外部用户与系统用户映射

### 5.3 核心接口

建议新增：

```go
type CollabPlatformAdapter interface {
    PlatformCode() string
    SendMessage(ctx context.Context, req *SendMessageRequest) (*SendMessageResult, error)
    SendCard(ctx context.Context, req *SendCardRequest) (*SendMessageResult, error)
    VerifyCallback(ctx context.Context, req *VerifyCallbackRequest) (*VerifyCallbackResult, error)
    ParseAction(ctx context.Context, req *ParseActionRequest) (*PlatformAction, error)
}
```

这样业务层只依赖抽象，不依赖飞书 SDK 细节。

## 八、飞书侧接入方式

第一阶段建议使用“飞书开放平台应用 + 机器人消息 + 交互卡片 + 回调”的组合。

不建议第一阶段使用飞书 AI 智能伙伴托管业务逻辑。

### 8.1 飞书应用形态

建议：

1. 创建企业自建应用
2. 开启机器人能力
3. 开启卡片交互回调
4. 配置事件订阅 / 回调地址

### 8.2 飞书中承担的职责

1. 消息触达
2. 卡片审批
3. 手机端操作入口
4. Web 控制台深链跳转

### 8.3 EasyMVP 中承担的职责

1. 真实业务状态机推进
2. 权限校验
3. 审批/返工/重试落库
4. 审计日志

### 8.4 飞书后台必须获取的参数

飞书应用创建完成后，第一阶段至少需要拿到以下参数：

1. `App ID`
2. `App Secret`
3. `Verification Token`
4. `Encrypt Key`
5. 回调地址 URL

建议后台获取路径如下：

1. `凭证与基础信息`
   - 获取 `App ID`
   - 获取 `App Secret`
2. `事件与回调 / 事件订阅`
   - 获取 `Verification Token`
   - 获取 `Encrypt Key`
   - 配置回调地址
3. `机器人能力 / 权限管理`
   - 开启机器人消息能力
   - 开启交互卡片动作回调

参数用途如下：

1. `App ID / App Secret`
   - 服务端换取 `tenant_access_token`
   - 用于发消息、查用户、调飞书开放接口
2. `Verification Token`
   - 校验事件回调来源
3. `Encrypt Key`
   - 事件订阅开启加密后用于解密请求
4. 回调地址 URL
   - 接收飞书事件、卡片点击动作、测试联通性

### 8.5 第一阶段后台配置清单

为了避免接入时反复返工，第一阶段建议按下面清单配置：

1. 创建企业自建应用
2. 开启机器人能力
3. 开启事件订阅
4. 配置卡片回调地址
5. 配置可见范围为内部测试用户/测试群
6. 发布到企业内部使用范围

第一阶段至少需要的能力范围如下：

1. 发消息
   - 支持机器人向用户或群发送通知
2. 交互卡片
   - 支持按钮审批、重试、返工、重验
3. 用户信息获取
   - 至少能完成飞书用户到 EasyMVP 用户的绑定映射
4. 事件订阅
   - 支持回调校验、动作回调、必要的事件通知

说明：

1. 飞书后台的具体权限名称可能随版本调整
2. 第一阶段不强依赖“全量组织通讯录权限”
3. 以“能完成消息、回调、用户映射”这三类能力为最小标准
4. 如无明确需要，不要一开始申请过宽权限

### 8.6 推荐的接入顺序

建议按下面顺序接，不要一上来全开：

1. 先创建应用并拿到 `App ID / App Secret`
2. 开启机器人能力并完成发消息测试
3. 开启事件订阅并完成 URL 校验
4. 开启交互卡片并联通审批回调
5. 再补用户绑定、订阅管理、审计日志

## 十一、动作回调设计

所有飞书点击动作都必须回调 EasyMVP 后端，再由后端执行业务动作。

### 11.1 回调路径

建议新增：

- `POST /api/collab/feishu/callback`

### 11.2 回调处理流程

1. 验证飞书签名
2. 解析动作类型
3. 解析业务 payload
4. 映射飞书用户到系统用户
5. 校验权限
6. 调用 Workflow / Accept / Review 对应服务
7. 返回更新后的卡片或结果消息

### 11.3 禁止事项

1. 禁止飞书回调直接改数据库
2. 禁止跳过 EasyMVP 原有服务层
3. 禁止把飞书用户 ID 当作系统用户 ID 直接使用

## 二十一、平台扩展口设计

为了后续接钉钉、企业微信，必须冻结以下扩展规则。

### 21.1 不允许飞书字段侵入业务层

业务层不允许出现：

1. 飞书卡片 JSON 结构
2. 飞书签名验证逻辑
3. 飞书用户 ID

这些都必须留在 `adapter/feishu/` 中。

### 21.2 不允许业务事件直接发平台消息

业务层只发标准事件，例如：

- `accept.pending`
- `workflow.failed`

平台格式转换必须由适配层处理。

### 21.3 用户映射必须平台无关

不能只建：

- `feishu_user_binding`

必须建通用表：

- `mvp_collab_user_binding`

## 二十四、灰度与回滚

### 24.1 灰度策略

1. 第一阶段只接飞书
2. 只对内部测试项目开启
3. 只开放通知和审批能力

### 24.2 开关建议

建议新增配置：

1. `collab.feishu.enabled`
2. `collab.feishu.callback_enabled`
3. `collab.feishu.approval_enabled`
4. `collab.platform.default`

### 24.3 回滚策略

若飞书接入异常：

1. 关闭 `collab.feishu.enabled`
2. 停止外发消息
3. 保留内部主链运行
4. 飞书回调入口降级为只记录日志

## 二十五、与现有模块的关系

### 25.1 与 WorkflowRun

飞书只消费 WorkflowRun 事件，不改变 WorkflowRun 主设计。

### 25.2 与 Accept Stage

飞书是 Accept 的移动端审批入口，不承担 Accept 裁决逻辑。

### 25.3 与数据权限补强

飞书接入必须遵守：

- [WorkflowRun已实现模块数据权限补强清单.md](./WorkflowRun已实现模块数据权限补强清单.md)

不能因外部平台接入而绕过数据权限治理方向。

## 二十六、冻结前检查项

在进入实施前，必须冻结：

1. 飞书是否为第一阶段唯一平台
2. 平台抽象接口
3. 用户绑定表设计
4. 订阅表设计
5. 回调鉴权方案
6. 审计日志方案
7. 第一阶段消息模板范围

## 二十七、结论

对 EasyMVP 来说，飞书最适合承担“手机端移动控制面”的角色，但业务执行内核必须保留在 EasyMVP 内部。

因此正确落地方式不是“把项目机器人托管给飞书”，而是：

1. EasyMVP 继续作为项目执行内核
2. 飞书作为通知、审批、轻操作入口
3. 平台接入层保持抽象，为钉钉、企业微信预留扩展口

这套方案可以直接作为第一阶段飞书接入的正式实施基线。
