# L3.5到L4自动化与少人值守演进方案

## 1. 目标

当前系统已经具备：

1. 阶段化主链：`design -> review -> execute -> accept -> complete`
2. 自动审核
3. 自动执行
4. 自动返工
5. 自动验收
6. 自治初版：风险评估、熔断、重规划建议、执行器选择、阶段/项目汇报

这说明系统已经接近第 3 层后段，但还没有真正进入“少人值守”和“项目级自治”。

本方案的目标是：

1. 让系统从“可自动跑主链”升级到“默认自动推进”
2. 让人从“持续盯盘操作”降级为“异常裁决与策略维护”
3. 让项目运行逐步逼近第 4 层，而不是继续堆局部功能

## 2. 总体定位

本方案不追求一步到位“完全无人”。

目标定位是三段式演进：

1. `L3.5`：少人值守
2. `L3.8`：局部无人
3. `L4-初版`：项目级自治

核心原则：

`默认自动推进，风险命中即停，人工只处理少数高风险决策。`

## 3. 自动化原则

### 3.1 自动化不等于放任执行

系统不能因为“能自动做”就一律自动做。

自动化必须建立在：

1. 状态机清晰
2. 风险边界明确
3. 决策可追溯
4. 人工可接管

### 3.2 三层能力模型

后续所有自动化能力，统一落到三层：

1. `感知层`
   收集状态、失败、质量、成本、权限、执行器反馈
2. `决策层`
   根据规则产出下一步动作
3. `执行层`
   真正执行 retry / rework / rerun accept / pause / replan / notify

禁止继续把自动化逻辑分散在各 stage service 的零散 `if/else` 中无限膨胀。

## 4. 决策分级模型

### 4.1 目标

把所有系统动作分成三类：

1. `A级：系统自动执行`
2. `B级：系统给建议，人确认`
3. `C级：必须人工决定`

### 4.2 建议分级

#### A级动作

1. `design -> review`
2. `review -> execute`
3. `execute -> accept`
4. `accept passed -> complete`
5. 低风险 `retry`
6. 低风险 `rerun accept`
7. 明确规则驱动的 `accept failed -> rework`
8. 同执行器、同范围内的轻量重试

#### B级动作

1. `accept -> manual_review`
2. `rework -> execute` 与 `rework -> accept` 的边界路径
3. 熔断后的重规划执行
4. 执行器切换建议
5. 质量不足但非 blocker 的放行建议

#### C级动作

1. 跨权限域资源变更
2. 高成本执行器切换
3. 大范围重规划
4. 高风险项目直接完成/终止
5. 无证据支撑的强制放行

### 4.3 决策数据模型

建议后续将自治决策统一抽象为：

```text
decision_type
decision_level
trigger_source
trigger_context
recommendation
auto_executable
human_required
final_action
```

## 5. 风险闸门模型

### 5.1 目标

让系统不是“全部自动”，而是“低风险自动，高风险停下”。

### 5.2 四类闸门

#### 权限闸门

当动作超出项目或数据作用域时，必须停止自动化。

触发条件示例：

1. 涉及跨部门项目
2. 涉及敏感资源
3. 涉及跨项目引用

#### 质量闸门

当 `accept` 不满足质量标准时，禁止进入 `complete`。

触发条件示例：

1. blocker issue
2. error 超阈值
3. 证据缺失
4. LLM 结论为 `failed`

#### 成本闸门

当动作可能引发明显成本上升时，系统只能建议，不能自动执行。

触发条件示例：

1. 切换到高成本模型
2. 切换到高成本执行器
3. 触发大量重试

#### 运行风险闸门

当系统已出现异常累积，必须暂停主链并等待裁决。

触发条件示例：

1. 连续失败
2. 熔断
3. 超时批量扩大
4. rework 轮次过多

## 6. 自治中台设计

### 6.1 目标

把现在分散在：

1. `watchdog`
2. `autonomy`
3. `accept`
4. `rework`
5. `scheduler`

里的自动动作收口到统一中台。

### 6.2 核心模块

建议新增三层中台：

1. `Decision Center`
2. `Policy Engine`
3. `Action Dispatcher`

### 6.3 职责划分

#### Decision Center

负责：

1. 收集触发事件
2. 组装上下文
3. 调用策略引擎
4. 保存决策记录

#### Policy Engine

负责：

1. 风险判断
2. 自动等级判断
3. 是否可自动执行
4. 推荐动作生成

#### Action Dispatcher

负责：

1. retry
2. rework
3. rerun accept
4. pause
5. replan
6. notify human

### 6.4 事件输入

统一输入事件建议包括：

1. `task.failed`
2. `task.timeout`
3. `accept.failed`
4. `accept.manual_review`
5. `workflow.circuit_break`
6. `rework.completed`
7. `replan.suggested`
8. `manual.override`

### 6.5 动作输出

统一输出动作建议包括：

1. `retry_task`
2. `trigger_rework`
3. `rerun_accept`
4. `pause_workflow`
5. `switch_executor`
6. `replan_workflow`
7. `notify_human`
8. `auto_complete`

## 7. 状态机自动推进策略

### 7.1 主链自动推进

系统应逐步形成默认自动推进：

1. `design -> review`
2. `review -> execute`
3. `execute -> accept`
4. `accept -> complete`

前提是：

1. 风险闸门未命中
2. 权限闸门未命中
3. 质量闸门通过

### 7.2 旁路自动推进

系统应支持默认自动处理这些旁路：

1. `execute failed -> rework`
2. `accept failed -> rework`
3. `rework completed -> execute`
4. `rework completed -> accept`

### 7.3 人工停驻点

以下情况必须进入人工节点：

1. `manual_review`
2. 熔断后高风险重规划
3. 跨权限域动作
4. 成本超阈值动作

## 8. 无人值守运行模型

### 8.1 第一步：少人值守

目标：

1. 正常项目不需要人工持续看板
2. 人只在异常时被动拉起

系统特征：

1. 主链自动推进
2. 异常自动告警
3. 人工审批入口就绪

### 8.2 第二步：局部无人

目标：

1. 轻故障可自恢复
2. 轻返工可自动闭环

系统特征：

1. 自动 retry
2. 自动 rework
3. 自动 rerun accept
4. 熔断后自动给出重规划建议

### 8.3 第三步：项目级自治

目标：

1. 系统能根据运行结果自动调整路线
2. 系统能控制成本、风险、质量
3. 人只处理少数关键决策

系统特征：

1. 自动执行器选择
2. 自动批次/路线调整
3. 自动阶段汇报
4. 自动暂停与自动建议

## 9. 协作平台定位

### 9.1 目标

协作平台不是“又一个控制台”，而是：

`少人值守模式下的人类兜底入口`

### 9.2 第一阶段职责

建议飞书第一期只承接：

1. 异常提醒
2. 人工审批
3. 人工驳回
4. 一键重试/返工/放行
5. 项目周报/阶段报告

### 9.3 原则

1. 正常推进不依赖人工进入后台
2. 只有 B/C 级决策才拉人
3. 飞书是入口，不是执行内核

## 10. 数据权限要求

自动化和无人值守必须建立在权限安全边界上。

冻结要求：

1. 自治决策不能越过项目作用域
2. 所有自动动作都要记录触发原因和作用域
3. 协作平台只暴露允许用户可见的项目与动作
4. 跨部门/跨项目动作必须进入 B/C 级

## 11. 关键实施项

### 11.1 第一批：少人值守底座

1. 决策分级表
2. 风险闸门规则
3. Decision Center 骨架
4. Action Dispatcher 骨架
5. 飞书审批与告警最小闭环

### 11.2 第二批：局部无人

1. 自动 retry 政策
2. 自动 rework 回流
3. 自动 rerun accept
4. 熔断后自动建议
5. 成本闸门策略

### 11.3 第三批：项目级自治

1. 自动执行器选择
2. 自动批次调整
3. 自动重规划执行
4. 项目级自治汇报

## 12. 需要新增的核心对象

建议后续新增或强化：

1. `policy_rule`
2. `risk_gate_rule`
3. `decision_action`
4. `human_checkpoint`

如果不想新增很多表，第一版也至少要把这些结构体冻结在自治模块里。

## 13. 验收标准

### 13.1 L3.5 验收

1. 正常项目主链可自动跑通
2. 人工只在异常时介入
3. 自动动作都有审计记录

### 13.2 L3.8 验收

1. 常见失败可自动恢复
2. 返工和重验可自动闭环
3. 大部分项目可少人值守

### 13.3 L4 初版验收

1. 系统能根据结果自动调整路线
2. 系统能控制风险与成本
3. 人只在少数关键裁决点介入

## 14. 结论

往自动化、无人方向走，不能继续靠增加零散功能点，而应收口为：

1. `决策分级`
2. `风险闸门`
3. `自治中台`
4. `协作平台兜底入口`

最终目标不是“让系统什么都能自动做”，而是：

`让系统在明确边界内默认自动推进，在高风险场景下可控地停下来。`

## 16. 自治中台模块设计

### 16.1 目录建议

```text
admin-go/app/mvp/internal/workflow/autonomy/
  decision_center.go
  policy_engine.go
  risk_gate.go
  action_dispatcher.go
  human_checkpoint.go
  model.go
```

### 16.2 模块职责

#### `decision_center.go`

职责：

1. 接事件
2. 组装上下文
3. 调 `PolicyEngine`
4. 写 `mvp_decision_action`
5. 调 `ActionDispatcher`

#### `policy_engine.go`

职责：

1. 决策分级
2. 风险闸门判断
3. 决定是否可自动执行
4. 生成推荐动作

#### `risk_gate.go`

职责：

1. 统一评估权限闸门
2. 统一评估质量闸门
3. 统一评估成本闸门
4. 统一评估运行风险闸门

#### `action_dispatcher.go`

职责：

1. 调用现有 `retry/rework/rerun_accept/pause/replan/notify`
2. 统一落执行结果
3. 统一生成失败回执

#### `human_checkpoint.go`

职责：

1. 创建人工介入任务
2. 对接飞书/协作平台审批
3. 支持后台手动处理

## 18. 与现有模块的接线点

### 18.1 Watchdog

接入点：

1. `task.failed`
2. `task.timeout`
3. `workflow.circuit_break`

要求：

1. Watchdog 不再直接决定所有后续动作
2. 统一发事件给 `DecisionCenter`

### 18.2 Accept

接入点：

1. `accept.failed`
2. `accept.manual_review`
3. `accept.passed`

要求：

1. `accept` 自己只做裁决
2. 是否自动返工、是否人工放行、是否自动完成，交给自治中台

### 18.3 Rework

接入点：

1. `rework.completed`

要求：

1. `rework` 不自己决定所有回流路径
2. 回流目标由策略引擎给出

### 18.4 Autonomy 现有模块

现有：

1. `risk_assessor`
2. `circuit_breaker`
3. `replanner`
4. `reporter`
5. `engine_selector`

调整建议：

1. 这些模块保留为能力组件
2. 决策入口收口到 `DecisionCenter`

## 19. 飞书审批与告警事件流

### 19.1 第一阶段能力

飞书第一期建议只承接：

1. `waiting_human` 的人工节点通知
2. `approve/reject` 审批回调
3. 项目阶段报告推送
4. 熔断与高风险异常告警

### 19.2 飞书事件流

建议流程：

1. `DecisionCenter` 生成 `mvp_human_checkpoint`
2. 协作平台适配层推送飞书卡片
3. 用户在飞书点击 `批准/驳回/暂停/返工`
4. 回调网关写入 `decision_action + human_checkpoint`
5. `ActionDispatcher` 执行最终动作

### 19.3 权限要求

飞书审批必须校验：

1. 用户与系统账号绑定关系
2. 项目访问权限
3. 当前 checkpoint 是否允许该用户处理

## 23. 风险与边界

### 23.1 不建议第一版做的事情

1. 不建议一开始就做“完全无人”
2. 不建议让 LLM 直接跳过风险闸门
3. 不建议让飞书变成主控制台
4. 不建议把所有自治策略写死在代码里

### 23.2 必须保留的人类兜底

1. 高成本动作
2. 跨权限域动作
3. 大规模重规划
4. 无证据支撑的质量放行

## 24. 开发启动条件

进入开发前，至少需要冻结：

1. 决策等级枚举

3. 动作枚举
4. `decision_action` 表结构

## 25. 最终建议

继续往自动化、少人值守推进时，最优先的不是再补单点页面，而是先把：

1. `决策分级`
2. `风险闸门`
3. `自治中台`
4. `飞书人工节点`

四件事做成统一底座。

这样系统才能从“能自动跑流程”，升级成“能可控地自动管理项目”。

## 26. 实施任务拆解

### 26.1 第一批：决策与闸门底座

1. 新建 `mvp_policy_rule / mvp_risk_gate_rule / mvp_decision_action / mvp_human_checkpoint`
2. 新增 `autonomy` 中台骨架：`decision_center / policy_engine / risk_gate / action_dispatcher`
3. 冻结 A/B/C 决策等级枚举
4. 冻结风险闸门枚举与默认规则
5. 将 `watchdog / accept / rework` 的自动动作入口统一改为走 `DecisionCenter`
6. 打通 `decision_action` 审计记录

交付标准：

1. 自动动作不再分散在多处 service 中直接拍板
2. 所有自动动作都有统一审计记录

### 26.2 第二批：人工节点与后台控制面

1. 新增 `human_checkpoint` 管理逻辑
2. 新增后台审批接口
3. 新增 `Autonomy Console` 页面
4. 新增 `Human Checkpoint Panel`
5. 新增 `Risk Gate Panel`
6. 接通人工批准/驳回后的动作派发

交付标准：

1. B/C 级动作都能停驻到人工节点
2. 人工处理后可以恢复系统自动推进

### 26.3 第三批：飞书少人值守入口

1. 飞书告警消息推送
2. 飞书审批卡片
3. 飞书回调网关
4. 飞书用户绑定与权限校验
5. 飞书阶段报告/项目报告推送

交付标准：

1. 人不需要持续盯后台
2. 关键审批可在飞书完成

### 26.4 第四批：项目级自治能力

1. 自动重规划从“建议”升级到“可配置自动执行”
2. 执行器选择策略接入 `DecisionCenter`
3. 批次/路线调整策略接入 `PolicyEngine`
4. 高风险动作统一过闸门

交付标准：

1. 项目可在边界内自动调整运行路线
2. 人只介入少数关键决策

## 28. 联调顺序

建议联调顺序：

1. 先联调 `decision_action` 和 `risk_gate_rule`
2. 再联调 `watchdog/accept/rework -> DecisionCenter`
3. 再联调后台人工节点
4. 最后接飞书审批入口

原则：

1. 不先接飞书再补自治中台
2. 不先做页面再补决策审计

## 29. 测试设计

### 29.1 用例层次

1. 单元测试：决策等级与闸门规则
2. 集成测试：`accept failed -> rework -> execute/accept`
3. 流程测试：熔断 -> 人工审批 -> 恢复执行
4. 端到端测试：飞书审批卡片回调

### 29.2 核心用例

1. blocker issue 触发自动返工
2. error/warn 触发人工审核
3. 熔断触发自动暂停
4. 高成本执行器切换被闸门拦截
5. 飞书审批通过后系统恢复推进

## 30. 灰度与回滚

### 30.1 灰度开关建议

1. `workflow.autonomy.enabled`
2. `workflow.autonomy.policy_engine_enabled`
3. `workflow.autonomy.risk_gate_enabled`
4. `workflow.autonomy.human_checkpoint_enabled`
5. `workflow.collab.feishu_enabled`

### 30.2 回滚原则

1. 关闭 `workflow.autonomy.enabled` 后，系统退回现有主链自动推进逻辑
2. 关闭 `workflow.collab.feishu_enabled` 后，只关闭移动端入口，不影响后台人工审批
3. 任何新自动动作都必须支持回退到“仅建议，不自动执行”
