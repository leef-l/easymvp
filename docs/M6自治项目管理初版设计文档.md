# M6 自治项目管理初版设计文档

## 一、目标

在月5自动验收全面交付的基础上，让系统从"会执行、会验收"升级到"会判断、会调整"。

月6完成后，系统应具备：

1. **自动重规划**：执行过程中发现方案不可行时，系统能自主发起重规划
2. **风险升级策略**：基于失败模式自动判断应该重试、返工还是暂停等人
3. **执行器智能选择**：根据任务特征自动匹配最佳执行器
4. **项目自治汇报**：自动生成阶段报告，人只看结论不追过程
5. **异常自动暂停**：连续失败、资源异常等场景下自动保护性暂停

核心原则：**先做建议型，再做自动执行型**。所有自治决策先走"建议→确认→执行"路径，验证稳定后再开放全自动。

---

## 二、当前能力边界

| 能力 | 现状 | 月6目标 |
|------|------|---------|
| 失败处理 | watchdog 重试 ≤3 次 → escalate → rework（架构师分析→回写→重执行） | 增加"重规划"路径：当 rework 也失败时，自动触发方案级重规划 |
| 风险判断 | 无，所有失败等价处理 | 引入风险分级：transient / structural / fatal，不同级别走不同路径 |
| 执行器选择 | 按 `ai_engine_config.engine_type` 固定配置 | 增加自动推荐：根据任务类型/复杂度/历史成功率选择 |
| 汇报机制 | complete 阶段有总结 JSON | 增加阶段级汇报 + 项目周报/日报 |
| 异常保护 | watchdog 单任务级别 | 增加项目级异常检测（连续失败率、资源耗尽、成本超限） |

## 四、核心能力设计

### 4.1 自动重规划（Replanner）

#### 触发条件

重规划不是随便触发的，只在以下明确场景启动：

1. **Rework 失败**：rework 阶段的 failure_analysis 任务本身失败（架构师也无法修复）
2. **批次全军覆没**：某个批次所有任务都失败或升级
3. **验收连续不过**：accept 阶段连续 N 轮（默认 3 轮）decision=failed
4. **人工触发**：用户在控制台点击"重规划"按钮

#### 执行流程

```
触发条件命中
    ↓
RiskAssessor 评估风险级别
    ↓ structural 或 fatal
创建 autonomy_decision（decision_type=replan, mode=suggest）
    ↓
调用 LLM（architect 角色）生成重规划建议
    - 输入：原方案摘要 + 失败任务列表 + 失败原因 + 验收问题
    - 输出：JSON { action: "replan_partial|replan_full|abort", affected_tasks: [...], new_plan_summary: "..." }
    ↓
写入 recommendation → 等待人工确认（suggest 模式）
    ↓ 人工 approved
创建新 plan_version → 更新/替换受影响的 domain_task → 重启调度
```

#### 重规划级别

| 级别 | 含义 | 动作 |
|------|------|------|
| `replan_partial` | 部分任务需要调整 | 只更新受影响任务的描述和依赖 |
| `replan_full` | 整体方案需要重做 | 创建全新 plan_version，重新生成全部 domain_task |
| `abort` | 项目不可行，建议终止 | 暂停项目，等待人工决策 |

#### Replanner 接口

```go
type Replanner struct {
    decisionRepo *repo.AutonomyDecisionRepo
}

// Evaluate 评估是否需要重规划，返回建议。
func (r *Replanner) Evaluate(ctx context.Context, input *ReplanInput) (*ReplanRecommendation, error)

// Execute 执行已批准的重规划决策。
func (r *Replanner) Execute(ctx context.Context, decisionID int64) error
```

### 4.2 风险评估器（RiskAssessor）

#### 风险分级

```go
const (
    RiskTransient  = "transient"   // 瞬态错误：网络超时、API 限流、临时资源不可用
    RiskStructural = "structural"  // 结构性错误：方案缺陷、依赖缺失、接口不匹配
    RiskFatal      = "fatal"       // 致命错误：工作目录损坏、Git 冲突不可解、权限错误
)
```

#### 分级策略

```
错误信息 + 失败模式 → 规则匹配 + LLM 辅助判断 → 风险级别
    ↓
transient  → 重试（现有 watchdog 路径）
structural → rework 或 replan（视 rework 次数判断）
fatal      → 自动暂停 + 通知人工
```

#### 规则匹配示例

| 错误模式 | 风险级别 | 依据 |
|----------|----------|------|
| `timeout`、`rate_limit`、`connection refused` | transient | 瞬态网络问题 |
| `import not found`、`undefined reference`、`type mismatch` | structural | 方案级缺陷 |
| `permission denied`、`git conflict`、`disk full` | fatal | 环境级问题 |
| 同一任务重试 ≥3 次仍失败 | structural→fatal 升级 | 重试无效说明不是瞬态 |
| 同批次 ≥50% 任务失败 | structural | 批次级系统性问题 |

#### RiskAssessor 接口

```go
type RiskAssessor struct{}

// Assess 评估任务失败的风险级别。
func (a *RiskAssessor) Assess(ctx context.Context, input *RiskInput) (*RiskResult, error)

// AssessBatch 评估批次级别风险。
func (a *RiskAssessor) AssessBatch(ctx context.Context, workflowRunID int64, batchNo int) (*RiskResult, error)
```

### 4.3 执行器智能选择（EngineSelector）

#### 当前现状

任务的执行器由 `mvp_project_role.execution_mode` 或 `ai_engine_config.engine_type` 固定配置，所有任务用同一个执行器。

#### 目标

根据任务特征自动推荐执行器，提升成功率：

| 任务特征 | 推荐执行器 | 理由 |
|----------|-----------|------|
| 多文件大范围重构 | claude_code / codex_cli | 全局理解能力强 |
| 单文件精确编辑 | aider | 专注、快速、成本低 |
| 需要运行测试/部署 | openhands | 有完整沙箱环境 |
| 文档/分析/审计类 | chat | 不需要代码编辑能力 |
| 前端 UI 实现 | gemini_cli | 多模态理解 UI 设计稿 |

#### 选择策略

```
任务描述 + affected_resources + role_type + 项目类型
    ↓
规则引擎匹配（优先）
    ↓ 无明确匹配
LLM 辅助推荐（可选）
    ↓
返回推荐执行器 + 置信度 + 理由
```

#### 实施方式

**第一阶段（建议型）**：在调度任务时记录推荐引擎到 `autonomy_decision`，但仍使用配置引擎执行。积累数据后分析推荐准确率。

**第二阶段（自动型）**：推荐置信度 ≥0.8 时自动切换执行器。

#### EngineSelector 接口

```go
type EngineSelector struct{}

// Recommend 为任务推荐最佳执行器。
func (s *EngineSelector) Recommend(ctx context.Context, task *DomainTaskInfo) (*EngineRecommendation, error)
```

### 4.4 项目自治汇报（Reporter）

#### 汇报类型

| 类型 | 触发时机 | 内容 |
|------|----------|------|
| `stage` | 每个阶段完成时自动生成 | 阶段耗时、任务统计、问题摘要、关键决策 |
| `daily` | 每日定时（项目 running 期间） | 当日进展、新增问题、风险预警、明日计划 |
| `weekly` | 每周定时 | 周进展总结、趋势分析、里程碑达成率 |
| `summary` | 项目完成时 | 全周期回顾（复用 complete stage 总结 + 增强） |

#### 生成方式

阶段报告 = 结构化指标 + LLM 生成自然语言摘要

```
收集指标快照（任务统计、耗时、失败率、验收评分等）
    ↓
构建 prompt（指标 + 近期事件流 + 已有问题）
    ↓
LLM 生成 Markdown 报告
    ↓
存入 mvp_project_report
```

#### Reporter 接口

```go
type Reporter struct {
    reportRepo *repo.ProjectReportRepo
}

// GenerateStageReport 生成阶段报告。
func (r *Reporter) GenerateStageReport(ctx context.Context, workflowRunID int64, stageType string) error

// GenerateDailyReport 生成日报。
func (r *Reporter) GenerateDailyReport(ctx context.Context, workflowRunID int64) error

// GetReports 获取项目报告列表。
func (r *Reporter) GetReports(ctx context.Context, projectID int64, reportType string) ([]ProjectReport, error)
```

### 4.5 熔断器（CircuitBreaker）

#### 熔断条件

| 条件 | 阈值 | 动作 |
|------|------|------|
| 连续失败任务数 | ≥5 | 暂停项目 + 生成诊断报告 |
| 批次失败率 | ≥60% | 暂停项目 + 触发风险评估 |
| Rework 轮次 | ≥3（同一原始任务） | 暂停该任务 + 升级人工 |
| Accept 连续不过 | ≥3 轮 | 暂停 + 建议重规划 |
| API 成本超预算 | 可配置 | 暂停 + 通知 |

#### 实现方式

熔断器嵌入 watchdog 的检查循环，作为附加检查项：

```go
type CircuitBreaker struct {
    decisionRepo *repo.AutonomyDecisionRepo
}

// Check 检查项目是否应该触发熔断。
func (cb *CircuitBreaker) Check(ctx context.Context, workflowRunID int64) (*CircuitBreakResult, error)
```

每次 watchdog 循环时调用 `Check`，如果触发熔断：
1. 暂停项目（调用 workflow service 的 Pause）
2. 创建 `autonomy_decision`（type=circuit_break）
3. 生成诊断报告（调用 Reporter）
4. 发射事件到 EventBus（前端 timeline 可见）

## 十、风险与克制

1. **重规划不能无限循环**：同一项目最多允许 2 次全量重规划，超过则强制暂停等人
2. **建议型优先**：所有自治决策默认走 suggest 模式，经过足够验证后再开放 auto
3. **LLM 调用成本控制**：风险评估优先用规则匹配，LLM 只在规则无法判断时介入
4. **不与现有流程冲突**：自治模块是附加层，关闭后系统回退到原有行为
5. **渐进式上线**：Phase 1-2 先上（低风险），Phase 3-5 逐步开放

---

## 十一、验收标准

1. 连续 5 个任务失败时系统自动暂停并生成诊断报告
2. Rework 失败时系统生成重规划建议（人工确认后执行）
3. 每个阶段完成后自动生成阶段报告
4. 自治决策全部有审计记录，可在控制台查看
5. 关闭 autonomy 配置后系统行为不变（向后兼容）
