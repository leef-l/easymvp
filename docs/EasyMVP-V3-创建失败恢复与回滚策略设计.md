# EasyMVP V3 创建失败恢复与回滚策略设计

> 更新时间：2026-04-19
> 上游文档：[EasyMVP-V3-创建项目流程与页面设计](./EasyMVP-V3-创建项目流程与页面设计.md)
> 关联文档：[EasyMVP-V3-创建初始化事件流接口设计](./EasyMVP-V3-创建初始化事件流接口设计.md)
> 关联文档：[EasyMVP-V3-本地目录与项目工作区规范](./EasyMVP-V3-本地目录与项目工作区规范.md)
> 目标：定义创建项目过程中失败场景的分类、恢复策略、回滚边界和用户可见行为。

## 1. 设计结论

创建失败不能只分成“成功/失败”。

V3 必须明确：

1. 哪类失败可重试
2. 哪类失败必须修改输入
3. 哪类失败需要回滚
4. 回滚后哪些痕迹保留

## 2. 失败分类

建议至少分 5 类：

1. `input_invalid`
2. `workspace_conflict`
3. `runtime_not_ready`
4. `profile_binding_failed`
5. `plan_bootstrap_failed`

## 3. 恢复原则

建议遵守：

1. 能局部重试就不要整链重建
2. 能保留已完成结果就不要全回滚
3. 用户必须知道当前保留了什么
4. 前端必须能明确渲染是否可继续

## 4. 回滚边界

### 4.1 必须回滚

1. 错误创建的目录占位
2. 无效的临时绑定记录
3. 不完整的初始化缓存

### 4.2 可保留

1. `project_id`
2. 已通过的路径检查结果
3. 已成功初始化的工作区目录

### 4.3 原则

如果 `project_id` 已发给前端，优先保留项目主记录，而不是重新生成。

## 5. 用户可见结果

失败后前端必须能展示：

1. 失败步骤
2. 当前错误原因
3. 是否可重试
4. 哪些步骤已完成
5. 是否已执行部分回滚

## 6. 重试策略

### 6.1 当前步骤重试

适用于：

1. `runtime_not_ready`
2. `profile_binding_failed`
3. `plan_bootstrap_failed`

### 6.2 修改输入后重建

适用于：

1. `workspace_conflict`
2. `input_invalid`

## 7. 不该怎么做

不应该：

1. 一失败就删干净所有记录
2. 一失败就保留所有脏状态
3. 前端不知道是否还能继续

## 8. 后续细分专题

1. 创建失败错误码表
2. 回滚状态事件 schema
3. 重试接口设计
