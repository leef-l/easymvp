# 执行器接入架构设计文档

> 更新日期：2026-04-08

本文只描述当前代码里已经存在的执行器抽象和接入链路。

## 1. 定位

在 Workflow V2 中，执行器不再直接耦合到调度器。调度主链现在是：

```text
mvp_domain_task.execution_mode
  -> executor.Registry
  -> 具体 Executor
  -> 任务结果 / 工作空间收尾 / 调度回调
```

也就是说：

- `DomainTaskScheduler` 负责挑任务、控并发、看依赖
- 执行器负责“如何跑”
- `workspace` 负责写仓任务隔离

## 2. 当前已注册执行器

当前注册发生在 `admin-go/app/mvp/internal/workflow/orchestrator/registry.go`。

| 模式 | 是否需要工作空间 | 当前定位 |
|------|------------------|----------|
| `aider` | 是 | CLI 写仓主力模式 |
| `chat` | 否 | 对话型执行和兜底模式 |
| `openhands` | 是 | 多步 Agent 执行 |
| `claude_code` | 是 | CLI 写仓执行器 |
| `codex_cli` | 是 | CLI 写仓执行器 |
| `gemini_cli` | 是 | CLI 写仓执行器 |
| `auto` | 是 | 按配置自动选执行器 |

`auto` 模式当前优先级为：

```text
claude_code -> aider -> codex_cli -> gemini_cli -> openhands -> chat
```

它会先检查 `ai_engine_config` 中哪些执行器被启用，再按顺序选择第一个可用执行器。

## 3. 统一接口

当前执行器接口已经落在 `internal/workflow/executor/executor.go`：

```go
type Executor interface {
    Name() string
    Execute(ctx context.Context, req *Request) *Result
    NeedsWorkspace() bool
}
```

统一请求 `Request` 包含：

- `ProjectID`
- `WorkflowRunID`
- `TaskID`
- `TaskRecord`
- `ModelInfo`
- `Workspace`

统一结果 `Result` 包含：

- `Success`
- `Output`
- `Error`

这意味着新增执行器时，不需要再改调度器主逻辑，只需要：

1. 实现 `Executor`
2. 决定是否需要 `workspace`
3. 在 `orchestrator/registry.go` 中注册

## 4. 当前调用链

执行阶段的真实调用顺序如下：

1. `stage/execute/service.go` 查询待执行的 `mvp_domain_task`
2. 通过 `execution_mode` 从 `Registry` 取执行器
3. 调用 `engine.ResolveProjectModelInfo` 解析模型信息
4. 如果执行器需要隔离，则先准备 `git worktree`
5. 调用 `exec.Execute(...)`
6. 根据结果更新任务状态，并通知调度器继续推进

其中：

- 成功路径会走 `handleSuccess`
- 失败路径会走 `handleFailure`
- `failure_analysis` 任务成功后还会触发返工回调

## 5. 配置来源

执行器运行时会组合使用几类配置：

### 5.1 模型来源

- `mvp_project_role`
- `mvp_role_preset`
- `ai_model`
- `ai_provider`
- `ai_plan`

### 5.2 引擎来源

- `ai_engine`
- `ai_engine_config`

当前引擎配置会影响：

- 超时
- 默认模型
- 命令模板
- `workspace_root`
- 额外 JSON 配置

### 5.3 系统检查

`workflow/system-check` 已经把这些依赖暴露到系统概览页，包括：

- Aider 引擎配置
- OpenHands 引擎配置
- Aider 可执行环境
- OpenHands 可执行环境

## 6. AI 模块与 MVP 的关系

需要区分两条链路：

### 6.1 AI 模块手工任务

`admin-go/app/ai` 里的 `task` 运行时目前优先支持：

- `aider`
- `openhands`

这条链路适合做：

- 引擎连通性验证
- 命令模板验证
- 单次手工执行

### 6.2 MVP 工作流任务

`admin-go/app/mvp` 里的执行器注册表支持的范围更大，包括：

- `claude_code`
- `codex_cli`
- `gemini_cli`
- `auto`

也就是说，AI 模块和 MVP 工作流共享引擎/模型配置数据，但不是同一条运行入口。

## 7. 当前实现边界

更新本文档时确认的边界如下：

- 当前最成熟的配置检查和手工运行链路仍然集中在 `Aider` / `OpenHands`
- `chat` 是明确的无工作空间兜底模式
- `auto` 会跳过依赖 Docker 命令模板但当前不可用的执行器
- 文档不再宣称所有执行器成熟度完全一致；是否可生产使用，应以对应执行器实现和环境准备情况为准
3. 把 OpenHands 从“预留/回退”改成正式适配器

## 十、实施边界

第一阶段目标不是“一次接完所有执行器”，而是：

1. 统一抽象先落地
2. Aider 迁移进抽象层
3. OpenHands 接通
4. 再扩 Claude Code / Codex CLI / Gemini CLI

---

## 十一、验收标准

1. 新执行器不需要改调度器主逻辑即可接入
2. 新执行器可被角色配置选择
3. 执行器运行结果能统一回写到 `DomainTask`
4. 支持暂停、取消、超时、失败回写
5. 前端可区分任务由哪个执行器执行

---

## 十三、结论

执行器接入的核心不是”多加几个命令行工具”，而是把 EasyMVP 从”单执行器调度系统”升级为”统一执行器平台”。

这套设计的关键是：

1. 业务层与执行器解耦
2. 调度层与执行器解耦
3. 配置、日志、失败、并发统一治理
