# 执行器接入架构设计文档

## 一、文档目标

本文档定义 EasyMVP 在 WorkflowRun 架构下的执行器接入总方案。

目标：

1. 将 Aider、OpenHands、Claude Code、Codex CLI、Gemini CLI 等执行器纳入统一运行框架。
2. 保持 `WorkflowRun -> StageRun -> DomainTask` 主链不被单一执行器耦合。
3. 让执行器接入具备一致的配置、调度、日志、失败处理和回写能力。

---

## 二、总体原则

1. `DomainTask` 是统一任务载体，执行器只是执行后端。
2. 执行器能力通过 `execution_mode` 和 `engine_code` 表达，不直接写死在调度器里。
3. 调度器只负责“何时执行”，执行器适配层只负责“如何执行”。
4. 所有执行器都必须支持最小公共协议：
   - 输入任务
   - 启动执行
   - 接收流式输出
   - 返回最终结果
   - 失败分类
   - 可取消

---

## 三、分层架构

### 3.1 业务层

- `WorkflowRun`
- `StageRun`
- `PlanVersion`
- `TaskBlueprint`
- `DomainTask`

该层只关心任务语义，不关心具体由哪个执行器完成。

### 3.2 调度层

- `StageScheduler`
- `DomainTaskScheduler`

职责：

1. 选择待执行任务
2. 检查批次、依赖、资源锁
3. 分配给执行器运行

### 3.3 执行器抽象层

新增统一抽象：

- `ExecutorAdapter`
- `ExecutorRegistry`
- `ExecutionSession`

职责：

1. 根据 `engine_code/execution_mode` 选择具体执行器
2. 统一生命周期
3. 统一结果结构

### 3.4 执行器实现层

- `AiderAdapter`
- `OpenHandsAdapter`
- `ClaudeCodeAdapter`
- `CodexCliAdapter`
- `GeminiCliAdapter`
- `ChatAdapter`

---

## 四、核心抽象

### 4.1 ExecutorAdapter

建议接口：

```go
type ExecutorAdapter interface {
    Code() string
    Capabilities() ExecutorCapabilities
    ValidateConfig(ctx context.Context, cfg ExecutorConfig) error
    Start(ctx context.Context, req *ExecutionRequest) (*ExecutionSession, error)
    Cancel(ctx context.Context, sessionID string) error
}
```

### 4.2 ExecutionRequest

统一输入建议包含：

- `workflowRunID`
- `stageRunID`
- `taskID`
- `projectID`
- `engineCode`
- `executionMode`
- `workDir`
- `taskName`
- `taskPrompt`
- `writeFiles`
- `readFiles`
- `modelInfo`
- `timeout`
- `maxSteps`

### 4.3 ExecutionSession

统一运行态建议包含：

- `sessionID`
- `engineCode`
- `startedAt`
- `status`
- `stream channel`
- `artifacts`
- `final result`

## 五、执行器分类

### 5.1 CLI 型执行器

代表：

- Aider
- Claude Code
- Codex CLI
- Gemini CLI

特点：

- 本地命令执行
- 适合文件编辑任务
- 与当前架构最兼容

### 5.2 Agent/Sandbox 型执行器

代表：

- OpenHands
- SWE-agent

特点：

- 多步自治执行
- 运行开销更高
- 更适合复杂任务

### 5.3 Chat/API 型执行器

代表：

- Chat 模式
- 未来 HTTP Agent

特点：

- 轻量
- 吞吐高
- 更适合非文件重写类任务

---

## 六、推荐接入顺序

1. OpenHands
2. Claude Code
3. Codex CLI
4. Gemini CLI
5. 远程 HTTP Agent

原因：

1. OpenHands 当前数据库和配置已经预留。
2. Claude Code / Codex CLI / Gemini CLI 与 Aider 的接入模式最接近。
3. 这些执行器可以复用现有工作目录、文件列表、任务提示词模型。

---

## 七、关键设计决策

### 7.1 不把执行器逻辑写进调度器

调度器只决定：

- 是否可调度
- 何时调度
- 调度给哪个任务

不决定：

- 命令行参数怎么拼
- 流输出怎么解析
- 失败怎么归因

### 7.2 不把执行器和项目分类强耦合

项目分类只影响默认推荐，不应强绑定。

例如：

- 软件开发默认 `aider/claude_code/codex_cli`
- 创作类默认 `chat`
- 复杂多步任务可切 `openhands`

### 7.3 执行器必须可观测

最少输出：

- 启动日志
- 流式输出
- 结束状态
- 失败分类
- 耗时

---

## 八、数据库落地方向

现有基础：

- `mvp_domain_task.execution_mode`
- `ai_engine`
- `ai_engine_config`

建议后续补充：

- `mvp_execution_session`
- `mvp_execution_event`
- `mvp_execution_artifact`

目的：

1. 记录执行器运行态
2. 记录中间流输出
3. 记录产物文件、日志、报告

---

## 九、与现有代码的衔接点

当前已存在：

- `execution_mode`
- `AiderRunner`
- `DomainTaskScheduler`
- `ExecuteStageService`

建议改造点：

1. 从 `switch execution_mode` 升级为 `ExecutorRegistry`
2. 把 `execute/service.go` 中的执行分发改为统一适配器调用
3. 把 OpenHands 从“预留/回退”改成正式适配器

---

## 十、实施边界

第一阶段目标不是“一次接完所有执行器”，而是：

1. 统一抽象先落地
2. Aider 迁移进抽象层
3. OpenHands 接通
4. 再扩 Claude Code / Codex CLI / Gemini CLI

---

## 十一、验收标准

1. 新执行器不需要改调度器主逻辑即可接入
2. 新执行器可被角色配置选择
3. 执行器运行结果能统一回写到 `DomainTask`
4. 支持暂停、取消、超时、失败回写
5. 前端可区分任务由哪个执行器执行

---

## 十三、结论

执行器接入的核心不是”多加几个命令行工具”，而是把 EasyMVP 从”单执行器调度系统”升级为”统一执行器平台”。

这套设计的关键是：

1. 业务层与执行器解耦
2. 调度层与执行器解耦
3. 配置、日志、失败、并发统一治理
