# L4 项目级自治能力设计方案

> 第四批实施 · 从"少人值守"到"项目级自治"

## 1. 设计目标

### 1.1 核心定位

L3.5 三批交付了：决策底座 + 后台控制面 + 飞书少人值守入口。
系统已能"默认自动推进，异常时停下拉人"。

第四批的目标是让系统从**被动响应异常**升级到**主动优化运行路线**：

```
L3.5 被动模式：出问题 → 策略匹配 → 决策 → 执行/拉人
L4   主动模式：运行中持续感知 → 预判风险 → 自动调整 → 事后复盘
```

### 1.2 验收标准

1. 项目可在边界内自动调整运行路线（执行器/批次/方案）
2. 人只介入少数关键决策（C 级动作）
3. 系统可从历史运行中学习，自动优化策略参数
4. 所有自治决策可审计、可回溯、可回滚

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **意图驱动** | 策略不再是"if-else 规则"，而是"目标函数 + 约束条件" |
| **反馈闭环** | 每个自治动作的结果都会反馈回策略引擎，形成学习环 |
| **渐进式信任** | 新策略默认 audit_only，积累信任后逐步放权 |
| **可解释性** | 每个自治决策都能回答"为什么这么做"和"依据是什么" |
| **可回滚性** | 任何自治动作都可以被人工撤销，系统恢复到动作前状态 |

## 2. 架构演进：从规则引擎到自治内核

### 2.1 当前架构（L3.5）

```
事件 → PolicyEngine.Match(trigger_source) → 静态规则匹配 → 固定动作
```

**局限：**
- 规则是静态的：`trigger_source=task.failed → retry_task`，不考虑上下文
- 闸门是阈值的：`retry_count >= 3 → 降级 C 级`，不考虑趋势
- 重规划是被动的：熔断后才触发，不预判
- 执行器是固定的：项目角色配置写死，不随任务特征动态选择

### 2.2 目标架构（L4）

```
┌─────────────────────────────────────────────────────────────────┐
│                    自治内核 (Autonomy Kernel)                    │
│                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐    │
│  │  Sensor   │   │ Planner  │   │ Actuator │   │ Learner  │    │
│  │  感知器   │──>│  规划器  │──>│  执行器  │──>│  学习器  │    │
│  │          │   │          │   │          │   │          │    │
│  │ 收集指标 │   │ 评估态势 │   │ 派发动作 │   │ 反馈优化 │    │
│  │ 检测异常 │   │ 生成方案 │   │ 跟踪效果 │   │ 调整参数 │    │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘   └────┬─────┘    │
│       │              │              │              │           │
│       └──────────────┴──────────────┴──────────────┘           │
│                          反馈环                                 │
└─────────────────────────────────────────────────────────────────┘
         ↑                                          ↓
    事件总线 (EventBus)                      动作派发 (ActionDispatcher)
         ↑                                          ↓
┌────────┴────────────────────────────────────┴──────────────────┐
│              工作流运行时 (Workflow Runtime)                     │
│  Scheduler · Executor · Watchdog · Stage Services              │
└────────────────────────────────────────────────────────────────┘
```

**四个核心组件的职责：**

| 组件 | 职责 | 输入 | 输出 |
|------|------|------|------|
| Sensor（感知器） | 持续采集运行指标，检测异常模式 | 事件流 + DB 状态 | 态势快照（Situation） |
| Planner（规划器） | 基于态势评估，生成最优动作方案 | Situation + 策略库 | ActionPlan |
| Actuator（执行器） | 派发并跟踪动作效果 | ActionPlan | ActionResult |
| Learner（学习器） | 从动作结果中学习，优化策略参数 | ActionResult | 策略调整建议 |

### 2.3 与现有架构的关系

**不是推翻重来，而是在现有中台上套壳升级：**

```
现有组件               L4 新组件                 关系
────────────────────────────────────────────────────
PolicyEngine    →   Planner 的子模块          规则匹配仍保留，作为 Planner 的快速路径
RiskGate        →   Sensor 的子模块           闸门检查变为"持续感知"的一部分
ActionDispatcher →   Actuator 的子模块        动作派发逻辑不变，增加效果跟踪
DecisionCenter  →   自治内核的外壳            Decide() 方法内部路由到 Sensor→Planner→Actuator
CircuitBreaker  →   Sensor 的熔断检测器       不变，作为异常检测的一种
Replanner       →   Planner 的重规划子模块    不变，作为规划器的一种策略
EngineSelector  →   Planner 的执行器选择子模块 从未使用 → 正式接入
(新增)          →   Learner                   全新模块
```

## 3. Sensor（感知器）—— 从"事件触发"到"持续感知"

### 3.1 设计理念

当前系统是**事件驱动**的：任务失败了才触发决策。
L4 要实现**持续感知**：运行过程中不断评估态势，提前预判风险。

### 3.2 态势模型（Situation）

```go
// Situation 是某一时刻工作流运行状态的完整快照。
// 它是 Planner 做决策的唯一输入，确保决策可解释、可复现。
type Situation struct {
    // 基础标识
    WorkflowRunID int64
    ProjectID     int64
    ProjectFamily string  // coding / creative / analysis
    CategoryCode  string
    
    // 进度指标
    Progress      *ProgressMetrics
    // 健康指标
    Health        *HealthMetrics
    // 资源指标
    Resource      *ResourceMetrics
    // 趋势指标
    Trend         *TrendMetrics
    // 上下文
    ActiveStage   string    // 当前阶段
    SnapshotAt    time.Time // 快照时间
}

type ProgressMetrics struct {
    TotalTasks       int     // 总任务数
    CompletedTasks   int     // 已完成
    RunningTasks     int     // 运行中
    FailedTasks      int     // 失败
    PendingTasks     int     // 等待中
    CompletionRate   float64 // 完成率
    CurrentBatchNo   int     // 当前批次
    TotalBatches     int     // 总批次数
    BatchProgress    float64 // 当前批次完成率
}

type HealthMetrics struct {
    ConsecutiveFailures   int     // 连续失败数
    RecentFailureRate     float64 // 最近 N 个任务的失败率
    AvgTaskDuration       int64   // 平均任务耗时(秒)
    MedianTaskDuration    int64   // 中位数任务耗时
    P95TaskDuration       int64   // P95 任务耗时
    RetryCount            int     // 总重试次数
    EscalationCount       int     // 总升级次数
    ReworkRounds          int     // 返工轮数
    AcceptRounds          int     // 验收轮数
    StaleTaskCount        int     // 卡死任务数
    HeartbeatMissCount    int     // 心跳丢失数
}

type ResourceMetrics struct {
    RunningConcurrency    int     // 当前并发数
    MaxConcurrency        int     // 最大并发数
    ResourceUtilization   float64 // 资源利用率
    LockedResourceCount   int     // 已锁定资源数
    ConflictCount         int     // 资源冲突次数（累计）
    TokensConsumed        int64   // Token 消耗总量
    EstimatedTokensLeft   int64   // 预估剩余 Token 额度
}

type TrendMetrics struct {
    FailureRateTrend      string  // rising / stable / falling
    DurationTrend         string  // rising / stable / falling
    ThroughputTrend       string  // rising / stable / falling
    // 滑动窗口指标（最近 10 个任务 vs 之前 10 个）
    RecentAvgDuration     int64
    PreviousAvgDuration   int64
    RecentFailureRate     float64
    PreviousFailureRate   float64
}
```

### 3.3 感知器实现

```go
type Sensor struct {
    circuitBreaker *CircuitBreaker  // 复用现有熔断器
    riskGate       *RiskGate        // 复用现有风险闸门
}

// Perceive 生成当前态势快照。
// 被 Planner 按需调用（事件触发时 + 定时巡检时）。
func (s *Sensor) Perceive(ctx context.Context, workflowRunID int64) (*Situation, error)

// DetectAnomalies 基于态势快照检测异常模式。
// 返回零个或多个异常信号，每个信号可能触发一个决策流程。
func (s *Sensor) DetectAnomalies(sit *Situation) []AnomalySignal
```

**异常信号类型：**

```go
type AnomalySignal struct {
    Type       string  // 异常类型
    Severity   string  // info / warning / critical
    Message    string  // 可读描述
    Metrics    g.Map   // 相关指标快照
    Confidence float64 // 置信度 0-1
}

// 预定义异常类型
const (
    AnomalyFailureSpike      = "failure_spike"       // 失败率突增
    AnomalyDurationDrift     = "duration_drift"       // 任务耗时漂移
    AnomalyThroughputDrop    = "throughput_drop"       // 吞吐量骤降
    AnomalyResourceExhaustion = "resource_exhaustion"  // 资源耗尽趋势
    AnomalyBatchStall        = "batch_stall"           // 批次停滞
    AnomalyCircuitBreak      = "circuit_break"         // 熔断（复用现有）
    AnomalyTokenBudgetLow    = "token_budget_low"      // Token 预算不足
)
```

### 3.4 定时巡检（Patrol）

除了事件触发，Sensor 还提供**定时巡检**能力：

```go
// StartPatrol 启动定时巡检协程。
// 每隔 interval 秒对所有运行中的工作流做一次态势感知。
// 发现异常信号后发布事件，由 Planner 接手。
func (s *Sensor) StartPatrol(ctx context.Context, interval int)
```

**巡检周期建议：** 与 watchdog 错开，默认 60 秒。
**巡检范围：** 所有 status=running 的 workflow_run。

## 4. Planner（规划器）—— 从"规则匹配"到"态势规划"

### 4.1 设计理念

当前 PolicyEngine 是**规则匹配器**：给定 trigger_source，查表返回固定动作。
L4 的 Planner 是**态势规划器**：综合当前态势，生成最优动作方案。

**关键区别：**

| 维度 | PolicyEngine（L3.5） | Planner（L4） |
|------|---------------------|---------------|
| 输入 | trigger_source（单事件） | Situation（全局态势） |
| 逻辑 | 规则表查找 | 多策略评估 + 最优选择 |
| 输出 | 单个动作 | ActionPlan（可含多步） |
| 学习 | 无 | 从历史结果中调整参数 |
| 解释 | "匹配了规则 X" | "因为 A+B+C，所以建议 D" |

### 4.2 动作方案模型（ActionPlan）

```go
// ActionPlan 是 Planner 输出的动作方案。
// 可包含一个或多个步骤，支持条件分支和回滚预案。
type ActionPlan struct {
    PlanID        string           // 唯一标识
    Trigger       string           // 触发源
    Situation     *Situation       // 决策时的态势快照
    Steps         []ActionStep     // 动作步骤（顺序执行）
    Reasoning     string           // 决策理由（可读文本）
    Confidence    float64          // 方案置信度 0-1
    RollbackPlan  *ActionPlan      // 回滚预案（可选）
    ExpectedOutcome string         // 预期效果描述
    DecisionLevel string           // A/B/C
}

type ActionStep struct {
    StepNo       int
    ActionType   string           // retry_task / switch_executor / adjust_batch / ...
    TargetID     int64            // 目标实体ID（taskID / workflowRunID）
    Parameters   map[string]interface{} // 动作参数
    Condition    string           // 前置条件表达式（可选）
    Timeout      int              // 超时秒数
}
```

### 4.3 策略体系重构

PolicyEngine 的静态规则仍然保留，作为**快速路径（Fast Path）**。
在此之上新增**策略函数（Strategy Functions）**，支持动态决策。

```go
// Strategy 策略函数接口。
// 每个策略函数负责一个决策域（执行器选择、批次调整、重规划等）。
type Strategy interface {
    // Name 策略名称
    Name() string
    // Evaluate 评估当前态势，返回动作方案候选
    Evaluate(ctx context.Context, sit *Situation, trigger string) *ActionPlan
    // Applicable 是否适用于当前态势
    Applicable(sit *Situation, trigger string) bool
    // Priority 优先级（数值越小越优先）
    Priority() int
}
```

**内置策略函数：**

| 策略 | 名称 | 决策域 | 说明 |
|------|------|--------|------|
| `EngineSelectionStrategy` | 执行器动态选择 | 任务执行前 | 根据任务特征 + 历史成功率选择最优执行器 |
| `BatchAdjustStrategy` | 批次动态调整 | 批次推进时 | 根据资源利用率和失败趋势调整并发度/批次编排 |
| `AdaptiveRetryStrategy` | 自适应重试 | 任务失败时 | 根据失败模式选择重试策略（立即/退避/换执行器/升级） |
| `ProactiveReplanStrategy` | 主动重规划 | 异常检测时 | 不等熔断，提前预判并建议调整 |
| `CostGuardStrategy` | 成本守卫 | 所有动作前 | 评估动作的 Token/时间成本，超阈值降级 |
| `QualityGateStrategy` | 质量关口 | 验收决策时 | 基于历史验收通过率动态调整质量标准 |

### 4.4 Planner 决策流程

```
事件/巡检触发
  ↓
Sensor.Perceive() → Situation
  ↓
Sensor.DetectAnomalies() → []AnomalySignal（如果是巡检）
  ↓
Planner.Plan(ctx, situation, trigger)
  │
  ├─ [快速路径] PolicyEngine.Match(trigger)
  │   ├─ 命中 → 检查是否需要策略函数增强
  │   │   ├─ 规则标记 strategy_override=false → 直接使用规则结果
  │   │   └─ 规则标记 strategy_override=true → 继续评估策略函数
  │   └─ 未命中 → 进入策略函数评估
  │
  ├─ [策略评估] 遍历所有 Applicable 的策略函数
  │   ├─ 按 Priority 排序
  │   ├─ 逐个 Evaluate
  │   └─ 选择 Confidence 最高的 ActionPlan
  │
  ├─ [风险检查] RiskGate.Check(plan, situation)
  │   ├─ 未阻断 → 返回 plan
  │   └─ 阻断 → 降级 + fallback
  │
  └─ 输出 ActionPlan
```

### 4.5 执行器动态选择（EngineSelectionStrategy）

**这是第四批最核心的能力之一。**

当前 `execution_mode` 在项目角色配置中写死。L4 要实现：
- 任务实例化时，根据任务特征推荐最优执行器
- 任务失败重试时，自动切换到更强的执行器
- 基于历史数据，学习哪种执行器对哪类任务效果最好

```go
type EngineSelectionStrategy struct {
    selector  *EngineSelector     // 复用现有规则
    statsRepo *EngineStatsRepo    // 历史统计仓储
}

func (s *EngineSelectionStrategy) Evaluate(
    ctx context.Context, sit *Situation, trigger string,
) *ActionPlan {
    // 1. 获取任务特征
    taskInfo := s.getTaskInfo(ctx, sit)
    
    // 2. 规则推荐（现有 EngineSelector）
    ruleRec := s.selector.Recommend(taskInfo)
    
    // 3. 历史数据修正
    stats := s.statsRepo.GetStats(ctx, taskInfo.ProjectFamily, taskInfo.RoleType)
    // 如果历史数据显示某执行器对此类任务成功率更高，优先选择
    historyRec := s.adjustByHistory(ruleRec, stats)
    
    // 4. 失败升级（如果是重试场景）
    if trigger == TriggerTaskFailed {
        currentEngine := taskInfo.ExecutionMode
        historyRec = s.escalateEngine(currentEngine, historyRec, sit.Health)
    }
    
    return &ActionPlan{
        Steps: []ActionStep{{
            ActionType: ActionTypeSwitchExecutor,
            Parameters: map[string]interface{}{
                "engine_type":  historyRec.EngineType,
                "confidence":   historyRec.Confidence,
                "reason":       historyRec.Reason,
            },
        }},
        Reasoning:  historyRec.Reason,
        Confidence: historyRec.Confidence,
    }
}
```

**执行器升级链：**

```
chat → aider → claude_code → openhands
  ↑      ↑         ↑            ↑
轻量   精准编辑  全局理解    沙箱环境
低成本  中成本    高成本      最高成本
```

**升级规则：**
1. 首次失败：同执行器重试（退避策略）
2. 同执行器重试 2 次仍失败：升级到链上下一个执行器
3. 已是最高级执行器：升级给架构师（escalate）

### 4.6 批次动态调整（BatchAdjustStrategy）

**目标：** 根据运行态势动态调整批次编排和并发度。

```go
type BatchAdjustStrategy struct{}

func (s *BatchAdjustStrategy) Evaluate(
    ctx context.Context, sit *Situation, trigger string,
) *ActionPlan {
    var steps []ActionStep
    
    // 1. 并发度调整
    if sit.Health.RecentFailureRate > 0.3 && sit.Resource.RunningConcurrency > 5 {
        // 失败率高 → 降低并发度，让系统稳定
        newConcurrency := max(sit.Resource.RunningConcurrency / 2, 3)
        steps = append(steps, ActionStep{
            ActionType: ActionTypeAdjustConcurrency,
            Parameters: g.Map{"max_concurrent": newConcurrency},
        })
    } else if sit.Health.RecentFailureRate < 0.05 && 
              sit.Resource.ResourceUtilization < 0.5 {
        // 一切顺利且资源空闲 → 提高并发度
        newConcurrency := min(sit.Resource.RunningConcurrency * 2, 40)
        steps = append(steps, ActionStep{
            ActionType: ActionTypeAdjustConcurrency,
            Parameters: g.Map{"max_concurrent": newConcurrency},
        })
    }
    
    // 2. 批次合并/拆分
    if sit.Trend.ThroughputTrend == "falling" && sit.Progress.BatchProgress < 0.3 {
        // 吞吐量下降且当前批次完成度低 → 可能存在瓶颈
        // 建议拆分大批次为小批次，提前推进无依赖的任务
        steps = append(steps, ActionStep{
            ActionType: ActionTypeSplitBatch,
            Parameters: g.Map{"batch_no": sit.Progress.CurrentBatchNo},
        })
    }
    
    return &ActionPlan{Steps: steps, ...}
}
```

### 4.7 自适应重试策略（AdaptiveRetryStrategy）

**取代现有的固定重试逻辑，根据失败模式选择最优重试策略：**

```go
type AdaptiveRetryStrategy struct{}

func (s *AdaptiveRetryStrategy) Evaluate(
    ctx context.Context, sit *Situation, trigger string,
) *ActionPlan {
    taskID := extractTaskID(trigger)
    failurePattern := s.classifyFailure(ctx, taskID)
    
    switch failurePattern {
    case PatternTransient:
        // 瞬态错误（网络/限流） → 指数退避重试
        return &ActionPlan{
            Steps: []ActionStep{{
                ActionType: ActionTypeRetryTask,
                Parameters: g.Map{
                    "backoff": "exponential",
                    "delay_ms": s.calcBackoff(sit.Health.RetryCount),
                },
            }},
            DecisionLevel: "A",
        }
        
    case PatternResourceConflict:
        // 资源冲突 → 等待 + 重新调度
        return &ActionPlan{
            Steps: []ActionStep{{
                ActionType: ActionTypeRescheduleTask,
                Parameters: g.Map{"delay_seconds": 30},
            }},
            DecisionLevel: "A",
        }
        
    case PatternCapabilityMismatch:
        // 能力不匹配 → 切换执行器
        return &ActionPlan{
            Steps: []ActionStep{{
                ActionType: ActionTypeSwitchExecutor,
                Parameters: g.Map{"escalate": true},
            }},
            DecisionLevel: "B",
        }
        
    case PatternStructural:
        // 结构性错误 → 触发返工
        return &ActionPlan{
            Steps: []ActionStep{{
                ActionType: ActionTypeTriggerRework,
            }},
            DecisionLevel: "B",
        }
        
    case PatternFatal:
        // 致命错误 → 暂停 + 拉人
        return &ActionPlan{
            Steps: []ActionStep{{
                ActionType: ActionTypePauseWorkflow,
            }},
            DecisionLevel: "C",
        }
    }
}
```

**失败模式分类（增强版）：**

```go
const (
    PatternTransient          = "transient"           // 网络/限流/超时
    PatternResourceConflict   = "resource_conflict"   // 文件锁/合并冲突
    PatternCapabilityMismatch = "capability_mismatch" // 执行器能力不足
    PatternStructural         = "structural"          // 方案/依赖/架构问题
    PatternFatal              = "fatal"               // 环境/权限/不可恢复
)
```

### 4.8 主动重规划策略（ProactiveReplanStrategy）

**不等熔断，提前预判并建议调整。**

```go
type ProactiveReplanStrategy struct {
    replanner *Replanner  // 复用现有重规划器
}

func (s *ProactiveReplanStrategy) Applicable(sit *Situation, trigger string) bool {
    // 仅在以下条件下触发主动重规划评估：
    // 1. 批次失败率持续上升
    // 2. 任务耗时持续膨胀
    // 3. 已完成超过 30% 的任务（有足够数据）
    return sit.Progress.CompletionRate > 0.3 && (
        sit.Trend.FailureRateTrend == "rising" ||
        sit.Trend.DurationTrend == "rising")
}

func (s *ProactiveReplanStrategy) Evaluate(
    ctx context.Context, sit *Situation, trigger string,
) *ActionPlan {
    // 1. 评估是否值得重规划
    costBenefit := s.estimateCostBenefit(sit)
    if costBenefit.NetBenefit < 0 {
        return nil  // 重规划代价大于收益，不建议
    }
    
    // 2. 调用 LLM 生成重规划建议
    rec := s.replanner.Evaluate(ctx, &ReplanInput{
        WorkflowRunID: sit.WorkflowRunID,
        ProjectID:     sit.ProjectID,
        TriggerSource: "proactive_analysis",
    })
    
    // 3. 主动重规划默认 B 级（需人工确认）
    return &ActionPlan{
        Steps: []ActionStep{{
            ActionType: ActionTypeReplanWorkflow,
            Parameters: g.Map{"recommendation": rec},
        }},
        DecisionLevel: "B",
        Reasoning: fmt.Sprintf("检测到%s趋势，建议提前调整", 
            describeTrend(sit.Trend)),
    }
}
```

## 5. Actuator（执行器）—— 从"单步执行"到"效果跟踪"

### 5.1 设计理念

当前 ActionDispatcher 执行动作后就结束了。
L4 的 Actuator 要**跟踪动作效果**，为 Learner 提供反馈数据。

### 5.2 动作效果跟踪

```go
// ActionResult 动作执行结果，包含效果评估。
type ActionResult struct {
    ActionID      int64
    PlanID        string
    ActionType    string
    Success       bool
    Error         string
    
    // 效果评估
    Outcome       *ActionOutcome
}

type ActionOutcome struct {
    // 动作前后的关键指标对比
    BeforeSituation  *Situation  // 动作前态势
    AfterSituation   *Situation  // 动作后态势（延迟采集）
    
    // 效果评分
    EffectivenessScore float64   // -1.0 ~ 1.0（负=恶化，正=改善）
    
    // 具体指标变化
    FailureRateDelta   float64   // 失败率变化
    DurationDelta      int64     // 耗时变化（秒）
    ThroughputDelta    float64   // 吞吐量变化
}
```

### 5.3 延迟效果评估

动作效果不是即时可见的。例如切换执行器后，需要等几个任务完成才能评估效果。

```go

// 在动作执行后 N 秒/N 个任务完成后，自动采集 AfterSituation 并计算效果。
func (a *Actuator) ScheduleOutcomeEvaluation(
    actionID int64,

    evalCondition EvalCondition,  // {AfterSeconds: 120} 或 {AfterTaskCount: 5}
```

## 6. Learner（学习器）—— 从"固定参数"到"自适应优化"

### 6.1 设计理念

这是 L4 最具前瞻性的组件。目标是让系统从运行历史中学习，自动优化策略参数。

**不是机器学习，是规则参数自适应调整。** 保持可解释性和确定性。

### 6.2 学习目标

| 学习维度 | 输入 | 输出 | 说明 |
|----------|------|------|------|
| 执行器亲和性 | 任务特征 + 执行器 + 成功/失败 | 执行器推荐权重 | "aider 对前端任务成功率 82%，claude_code 93%" |
| 重试策略参数 | 失败模式 + 重试结果 | 退避参数/升级阈值 | "瞬态错误 3 秒后重试成功率 90%，1 秒只有 40%" |
| 并发度最优值 | 项目类型 + 并发数 + 吞吐量/失败率 | 最优并发数 | "软件开发项目最优并发数 12，超过 15 失败率骤增" |
| 阈值自适应 | 闸门阈值 + 触发/误报次数 | 调整后阈值 | "连续失败阈值从 5 调到 4，能更早发现问题" |
| 决策级别校准 | 动作级别 + 人工干预率 | 级别调整建议 | "retry_task 从未被人工驳回，建议从 B 降为 A" |

### 6.4 学习算法（指数移动平均）

使用**指数移动平均（EMA）**更新参数，确保：
- 新数据权重更高（适应变化）
- 旧数据逐渐衰减（避免历史包袱）
- 简单、确定性、可解释

```go
// 更新公式：new_value = α * observed + (1 - α) * old_value
// α (学习率) 由 sample_count 动态调整：
//   样本少时 α 大（快速学习）
//   样本多时 α 小（稳定输出）
func calcLearningRate(sampleCount int) float64 {
    if sampleCount < 10 {
        return 0.3  // 快速学习期
    } else if sampleCount < 50 {
        return 0.15 // 稳定学习期
    }
    return 0.05     // 微调期
}
```

### 6.5 学习保护机制

| 保护 | 说明 |
|------|------|
| **最小样本量** | 参数需 ≥ 10 个样本才会被策略函数使用 |
| **置信度门槛** | confidence < 0.5 的参数不会影响决策 |
| **异常值过滤** | 单次观测值偏离均值 3σ 以上时丢弃 |
| **人工锁定** | 管理员可锁定某个参数值，学习器不再更新 |
| **衰减机制** | 30 天未更新的参数 confidence 自动衰减 |
| **回滚能力** | 每次更新保留前值，异常时可回滚 |

```
Layer 5：patrol_enabled=0                       不巡检（仅事件触发）
  ↓ 开启
Layer 6：全部开启                                完整 L4 能力
```

**每一层都可以独立回滚到上一层，无任何副作用。**

## 12. 后续迭代方向（L4.5 → L5）

本方案预留了以下扩展点，无需重构即可迭代：

### 12.1 LLM 增强策略函数

当前策略函数是纯规则的。后续可让策略函数内部调用 LLM：

```go
// 未来：LLM 增强的执行器选择
func (s *EngineSelectionStrategy) EvaluateWithLLM(ctx, sit, trigger) *ActionPlan {
    // 1. 规则初筛
    candidates := s.ruleBasedFilter(sit)
    // 2. LLM 精排（传入任务描述 + 候选列表 + 历史统计）
    best := s.llmRank(ctx, candidates, sit)
    return best
}
```

### 12.2 跨项目学习

当前学习是按 `project_family:category_code` 隔离的。后续可实现：
- 全局学习层：跨项目的通用规律
- 项目学习层：项目特有的偏好
- 合并策略：全局参数 + 项目偏移

### 12.3 策略市场

将策略函数做成可插拔的"策略包"：
- 用户可从策略市场安装/卸载策略
- 策略包包含：策略代码 + 默认参数 + 适用场景描述
- 支持用户自定义策略函数（脚本化/DSL）

### 12.4 自治等级自动升降

基于学习器的置信度，自动调整策略的决策级别：

- 共享资源池（Token 配额、执行器实例）
- 跨项目优先级调度
- 全局成本优化

## 13. 关键设计决策 FAQ

### Q1: 为什么不用机器学习？

**答：** 可解释性 + 确定性。每个自治决策都必须能回答"为什么"。EMA 参数调整是确定性的、可追溯的、可回滚的。未来可在 LLM 增强策略函数中引入更复杂的推理，但底层参数管理仍保持确定性。

### Q2: 策略函数和 PolicyEngine 规则什么关系？

**答：** PolicyEngine 规则是**快速路径**。如果规则能直接决策且不需要态势分析，就不走策略函数（性能优先）。规则的 `strategy_override=1` 字段控制是否允许策略函数覆盖规则结果。

### Q3: 学习器会不会失控？

**答：** 五重保护：最小样本量、置信度门槛、异常值过滤、人工锁定、衰减机制。而且 `learner_enabled=0` 可随时关闭。

### Q4: 定时巡检会不会影响性能？

**答：** 巡检只读 DB 聚合查询，不写入业务表。60 秒间隔，每次查询 < 100ms。如果 running 的 workflow_run 超过 50 个，自动降低巡检频率。

### Q5: 向后兼容怎么保证？

**答：** 所有新能力默认关闭（配置项默认 0）。`strategy_enabled=0` 时 Planner 完全退化为 PolicyEngine.Match()，行为与 L3.5 完全一致。
