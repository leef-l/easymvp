# 系统机制类问题彻底根治计划

## 1. 当前结论

这份文档关注的不是模型偶发输出错误，而是 EasyMVP 自身的系统机制是否会反复制造同类事故。

截至 `2026-04-10`，这项工作已经不是“从零规划”，而是已经形成了明确的实现基线：

- `Track A` 路径基准、guard 前置、syncBack 硬闸门：主体已落地
- `Track B` detached finalize、假 `running` 修复、orphan 对账：主体已落地
- `Track C` watchdog lease 收紧与观测：已落地
- `Track D` 事件快路径、Redis Stream、降级：主体已落地，completion 恢复闭环、运行态观测、durable idempotency、Redis 降级/恢复/reclaim 集成验证与真实 workflow 回放验收均已补齐

因此，当前最重要的任务不是继续把已完成的治理项写成“待实现”，而是把真正剩余的问题收缩到少数几个闭环点上。

## 2. 根治目标不变

系统机制层面，仍然必须满足以下底线：

- 不允许重复嵌套路径、标题型垃圾文件、越界文件污染主仓库
- 不允许 `syncBack` 绕过硬校验把垃圾内容带回主仓库
- 不允许执行器超时、进程被 kill、workspace finalize 失败后长期留下假 `running`
- 不允许 orphan worktree 长期堆积且不可见
- 不允许显式失败、显式完成、静默卡死三类恢复链路混在一起，导致 workflow 卡住

### 4.6 已补齐：syncBack display path 误判

本轮修复：

- `admin-go/utility/worktreeguard/guard.go`
- `admin-go/app/mvp/internal/workspace/git_worktree.go`
- `admin-go/utility/worktreeguard/guard_test.go`
- `admin-go/app/mvp/internal/workspace/git_worktree_test.go`

当前行为：

- 路径归一化已支持去掉树形目录前缀、双空格说明段、`#` 行尾注释、常见 display-path 包装
- `syncBack` 二次校验会先归一化 changed path，再做 suspicious / invalid 判定
- 真正越界的文件仍会被拦，只有“展示形态污染”不再造成假阳性

结论：

- 这类 `.gitignore / scripts/ / frontend/...` 的 display path 误判已经根治，不应再靠人工 rework 才能绕过去

3. `syncBack` 的 cherry-pick 和 copy fallback 都有 repo-root 硬闸门
4. 执行器被 kill 后，不再频繁留下假 `running`
5. orphan worktree 有系统级对账、修复和 system-check 展示
6. 显式失败可秒级进入 `retry / rework`
7. 静默卡死可由 watchdog 在目标时间内检测并补偿
8. 最后一个任务完成后即使进程崩溃，重启后仍能推进阶段
9. Redis 不可用时主链可继续，恢复后 backlog 可继续消费
10. 至少一条真实 workflow 回放验证通过

## 9. 验证矩阵

| 场景 | 触发方式 | 预期结果 | 通过信号 |
| --- | --- | --- | --- |
| 深层子目录路径污染 | 在深层子目录任务中制造重复嵌套路径 | duplicate path 在 syncBack 前被清理或阻断 | 主仓库无污染，guard / syncBack 日志可见 |
| 标题型垃圾文件 | 执行器生成 `运行方式：` 等文件 | suspicious file 在 guard / syncBack 前被清理或阻断 | 主仓库无该文件，任务失败原因可见 |
| 假 `running` 修复 | finalize 中断、执行器被 kill | workspace / domain_task 状态被修正 | DB 中不残留长期 `running` 假状态 |
| orphan 对账 | 制造磁盘 orphan / DB orphan / running mismatch | orphan sweep 能识别并修复或报警 | system-check 可见对账结果 |
| 显式失败快路径 | 任务显式失败 | 秒级进入 `retry / rework` | 事件表、日志、状态迁移一致 |
| 静默卡死慢路径 | 停止心跳但不写失败 | watchdog 在目标时间内判死并补偿 | watchdog snapshot 与任务状态一致 |
| completion crash recovery | 最后一个任务完成后立刻杀进程 | 重启后仍推进阶段 | workflow 不停留在“任务全完成但阶段不推进” |
| Redis 降级与恢复 | Redis 不可用后恢复 | 主链不中断，恢复后 backlog 继续消费 | system-check 显示 degraded -> healthy，pending 可见 |

## 10. 交付物定义

每个收口项都必须同时交付以下内容，才算“完成”：

- 代码改动
- 单测或集成测试
- system-check 或日志可观测项
- 一条对应的回放/演练记录
- 文档回填

如果只有代码，没有观测或回放证据，则只能算“已实现”，不能算“已收口”。

## 11. 最终交付定义

当以下条件全部成立时，才可判定“系统机制类问题已进入根治完成态”：

- 路径契约与 syncBack 硬闸门稳定生效
- detached finalize / cleanup / 假 `running` 修复已稳定生效
- orphan sweep 与 system-check 可持续对账
- watchdog 双保险对显式失败、静默卡死、进程重启均形成闭环
- completion 恢复链补齐，不再存在“全任务完成但阶段不推进”的卡态
- 关键崩溃场景有自动化测试
- 真实项目回放验证通过
