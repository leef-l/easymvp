# 系统机制类问题彻底根治计划

## 1. 目标

这份文档针对的不是“模型会不会偶发写错代码”，而是 EasyMVP 自身的系统机制问题。  
目标是把那些会反复出现、会污染主仓库、会让 workflow 卡死、会导致假状态的底层机制问题一次性收口。

核心判断标准：

- 不允许再出现同类问题换个目录、换个任务名后重新冒出来
- 不允许垃圾文件、越界文件、重复嵌套路径落入主仓库
- 不允许因为执行器超时或进程被 kill 就留下假 `running`
- 不允许因为恢复链路慢或丢事件，让工作流卡很久才被拉起来

## 2. 范围

本计划只覆盖系统机制类问题，不覆盖业务需求正确性。

覆盖范围：

- 路径基准与资源边界
- worktree guard
- syncBack / cherry-pick / copy fallback
- cleanup / finalize / stale workspace
- watchdog / 事件驱动恢复
- system-check / 观测 / 降级

不覆盖范围：

- 模型输出内容质量
- 项目业务逻辑对错
- 生成项目是否“产品设计合理”

## 3. 问题分类

### 3.1 路径基准混用

当前系统里最反复的一类问题，是同一套路径同时承担了两种语义：

- 对执行器来说，需要“相对当前执行目录”的路径
- 对 guard / syncBack 来说，需要“相对仓库根目录”的路径

这会直接导致：

- `backend/backend/...`
- `frontend/src/components/frontend/src/components/...`
- 深层子目录看不见根目录 stray file

### 3.2 标题型垃圾文件泄漏

模型有时会把自然语言标题当成文件名，例如：

- `运行方式：`
- `验证方式：`
- `说明：`

这类文件如果没有被 guard 及时发现和清理，就会导致任务失败，甚至可能污染主仓库。

### 3.3 syncBack 回写不够硬

当前 worktree 同步回主仓库时：

- cherry-pick 路径缺少二次 guard
- copy fallback 更危险，因为它绕开 Git 冲突语义，容易把脏文件直接复制回主仓库

### 3.4 cleanup / finalize 绑定原始上下文

执行器一旦超时、被 kill、或上层 ctx 被取消：

- cleanup 可能不执行
- finalize 可能失败
- 任务状态可能停留在假 `running`
- worktree 可能残留

### 3.5 orphan worktree 无系统级清扫

当前更多是“某个任务重置时顺手清理自己的 worktree”，还缺独立的 orphan 扫描与系统检查。

### 3.6 恢复机制慢且依赖轮询

当前显式失败、显式完成、静默卡死三类恢复都没有彻底分层：

- 显式状态变化应该走事件快路径
- 静默异常应该走 watchdog 慢路径

## 4. 彻底根治原则

### 原则 1：路径语义彻底拆分

必须把下面两类路径彻底分开：

- `executor-relative paths`
- `repo-root-relative paths`

谁都不能同时承担两种语义。

### 原则 2：主仓库入口必须有硬闸门

任何变更想回到主仓库之前，都必须再次经过：

- suspicious path 清理
- duplicate path 清理
- repo-root 级越界校验

### 原则 3：收尾逻辑不能依赖任务主上下文

cleanup / finalize / status repair 必须使用 detached context。  
否则执行器一旦被杀，收尾就会跟着死。

### 原则 4：显式失败和静默故障分层处理

- 显式失败/完成：事件驱动快路径
- 静默卡死：watchdog 慢路径

### 原则 5：任何临时目录都必须可对账

磁盘 worktree、DB 记录、任务状态必须能互相核对；不允许出现“磁盘还有、DB 没了”这种长期 orphan。

## 5. 总体方案

## 5.1 路径与 guard 统一

目标：彻底消灭路径基准混用。

要做的事：

1. 执行器提示词、文件列表、命令输入统一使用当前 `workDir` 相对路径
2. guard / syncBack / status 校验统一使用 repo-root 相对路径
3. `Capture/Validate` 一律以 repo root 为采样基准
4. 深层子目录任务必须补集成测试

对应代码：

- `admin-go/app/mvp/internal/workflow/executor/resource_targets.go`
- `admin-go/app/mvp/internal/workflow/executor/aider.go`
- `admin-go/utility/worktreeguard/guard.go`

验收标准：

- 不再出现 `backend/backend/...`
- 不再出现 `frontend/src/components/frontend/src/components/...`
- 从深层子目录执行时也能发现根目录 stray file

## 5.2 suspicious / duplicate 清理前置化

目标：把明显垃圾文件在进入失败路径前就自动清掉。

要做的事：

1. duplicate path 清理前置
2. suspicious title file 清理前置
3. 清理后必须重新校验一次
4. 清理日志可见

对应代码：

- `admin-go/utility/worktreeguard/guard.go`
- `admin-go/app/mvp/internal/engine/aider_runner.go`

验收标准：

- 标题型垃圾文件不会再直接导致主仓库污染
- duplicate path 不会再靠人工排查删除

## 5.3 syncBack 前硬校验

目标：不管任务 worktree 里出现什么，主仓库都不能被污染。

要做的事：

1. `syncWorktreeCommit()` 进入 cherry-pick 前做 repo-root 级 guard
2. `syncChangedFilesToMain()` copy fallback 前再做一层硬校验
3. 若发现 suspicious / invalid path：
   - 直接阻断同步
   - 任务保持失败/待修复
   - 不允许落主仓库

对应代码：

- `admin-go/app/mvp/internal/workspace/git_worktree.go`

验收标准：

- 即使执行器生成垃圾文件，也不会回写到主仓库
- copy fallback 不再是无保护路径

## 5.4 detached cleanup / finalize

目标：执行器失败不再拖着状态修复一起死。

要做的事：

1. Aider/CLI 执行完后的 cleanup、guard、finalize 使用短生命周期 detached context
2. finalize 失败时写 warning，但仍保证任务状态可修复
3. 增加假 `running` 修复链

对应代码：

- `admin-go/app/mvp/internal/engine/aider_runner.go`
- `admin-go/app/mvp/internal/workflow/executor/cli_helpers.go`
- 其他 executor 的 finalize 调用点

验收标准：

- 执行器进程被 kill 时，不再频繁留下假 `running`
- cleanup/finalize 的失败率明显下降

## 5.5 orphan worktree 扫描

目标：磁盘、DB、任务状态可对账。

要做的事：

1. 新增 orphan sweeper
2. 定期扫描 `.mvp-worktrees/`
3. 对账：
   - 磁盘存在但 DB 无记录
   - DB 已删除但磁盘仍存在
   - 任务终态但 worktree 未清理
4. system-check 展示 orphan 数量

对应代码：

- `admin-go/app/mvp/internal/workspace/`
- `admin-go/app/mvp/internal/controller/chat/workflow_system_check.go`

验收标准：

- orphan worktree 不再长期堆积
- 系统检查可见

## 5.6 watchdog 双保险

目标：恢复机制彻底分层。

本项单独计划已拆文档：

- [Watchdog双保险实施计划.md](./Watchdog双保险实施计划.md)

本计划与其关系：

- 本文关注“系统机制总方案”
- `Watchdog双保险实施计划` 关注“恢复链专项实施”

## 6. 优先级

### P0：必须立即收口

- repo-root 路径基准统一
- suspicious / duplicate 前置清理
- syncBack 前硬校验

### P1：必须尽快收口

- detached cleanup / finalize
- 假 `running` 修复链

### P2：系统稳定化

- orphan worktree 扫描
- system-check 扩展
- watchdog 双保险

## 7. 可并行拆分

### Track A：路径与 guard

负责：

- repo-root 基准统一
- duplicate / suspicious 清理
- 深层子目录测试

主要文件：

- `admin-go/utility/worktreeguard/*`
- `admin-go/app/mvp/internal/workflow/executor/*`
- `admin-go/app/mvp/internal/engine/aider_runner.go`

### Track B：syncBack 安全化

负责：

- syncBack 前二次 guard
- copy fallback 安全闸门
- 回写失败策略

主要文件：

- `admin-go/app/mvp/internal/workspace/git_worktree.go`
- `admin-go/app/mvp/internal/workspace/git_worktree_test.go`

### Track C：cleanup / finalize / stale repair

负责：

- detached cleanup
- detached finalize
- 假 running 修复
- orphan sweeper

主要文件：

- `admin-go/app/mvp/internal/workflow/executor/cli_helpers.go`
- `admin-go/app/mvp/internal/engine/aider_runner.go`
- `admin-go/app/mvp/internal/workspace/*`
- `admin-go/app/mvp/internal/controller/chat/workflow_system_check.go`

### Track D：恢复链

负责：

- watchdog 收紧
- 事件快路径
- Redis Stream / 降级

主要文件：

- `admin-go/app/mvp/internal/workflow/watchdog/*`
- `admin-go/app/mvp/internal/workflow/scheduler/*`
- `admin-go/app/mvp/internal/workflow/orchestrator/*`

## 8. 验收标准

必须同时满足：

1. 深层子目录任务不再产生重复嵌套路径污染
2. 根目录 stray file 能从子目录任务中被发现并清理
3. 主仓库回写前存在硬闸门，垃圾文件不能落库
4. 执行器被 kill 时不再频繁留下假 `running`
5. orphan worktree 有系统级检查和清理
6. 显式失败/完成恢复速度显著提升
7. 至少一条真实项目从失败到恢复再到完成的链路回放通过

## 9. 明确不能承诺的部分

以下问题不能“彻底不再发生”，只能尽量收敛：

- 模型偶发写错业务代码
- 模型对需求理解偏差
- 第三方 CLI / 模型服务超时或异常

系统机制真正能承诺的是：

- 不让这些错误越界污染主仓库
- 不让这些错误把 workflow 长时间卡死
- 出错后能自动发现、修复、重试、降级

## 10. 交付定义

当以下条件全部成立时，才可判定“系统机制类问题已进入根治完成态”：

- 路径基准已统一
- syncBack 有硬校验
- detached cleanup / finalize 已上线
- orphan worktree 有检查与清理
- watchdog 双保险已落地
- 真实项目回放验证通过
- system-check 可直接看到关键状态
