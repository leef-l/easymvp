# 流水线架构重构设计文档

> 实施完成后删除此文档

## 一、现状问题

### 1.1 核心缺陷：单行道架构

当前四个角色是**功能标签**，不是**流程阶段**：

```
架构师输出任务 → 无人审核 → 直接执行 → 审计员只审代码 → Bug回到架构师
```

导致：
- 架构师输出的任务描述乱码/不可执行 → 实施员盲目执行 → 批量失败
- affected_resources 错误 → Aider 找不到文件 → 重试 3 次 → 升级 → 可能输出同样的错误 → 死循环
- 协调员角色完全未使用（代码中零引用）
- running 状态下架构师仍可被触发（batch_no=0 紧急任务），与正在执行的任务冲突

### 1.2 审查发现的 Bug（一并修复）

| # | 问题 | 文件 |
|---|------|------|
| 1 | Entity/DAO 缺少 active_batch_no、locked_resources、heartbeat_at | entity/do/dao |
| 2 | task_parser 清理旧任务不在事务中 | task_parser.go |
| 3 | state_machine.go 的 ValidateTransition() 从未被调用 | 全局 |
| 4 | mergeIntoGlobalContext 无并发控制 | context_compressor.go |
| 5 | workDir 前端必填 vs 后端非编码类自动生成矛盾 | form.vue + workflow.go |
| 6 | SSE 事件类型前端不区分，error 事件被吞 | chat/index.vue |
| 7 | mvp_role_preset collation 不一致 | 数据库 |
| 8 | 9 个重复索引 | 数据库 |

## 二、流水线架构设计

### 2.1 核心理念

将四个角色从"功能标签"变为"流水线阶段"，每个阶段有明确的**输入、输出、质量门**。

### 2.2 流水线阶段

```
Stage 0: 设计阶段（designing）
  ├─ 用户与架构师 AI 对话
  ├─ 架构师输出任务清单（JSON）
  └─ 任务解析为 draft 状态

Stage 1: 方案审核（reviewing）  ← 新增阶段
  ├─ 系统自动预检（格式、路径、依赖、冲突）
  ├─ 审计员 AI 审核任务质量
  ├─ 协调员 AI 优化调度计划
  ├─ 通过 → draft → pending
  └─ 不通过 → 反馈架构师修正 → 回到 Stage 0

Stage 2: 执行阶段（running）
  ├─ 调度器按批次分发
  ├─ 实施员执行（Aider/Chat）
  └─ 每个任务完成后 → Stage 3

Stage 3: 验收阶段（auditing）  ← 增强
  ├─ 审计员审查实施结果
  ├─ 通过 → completed
  ├─ 不通过 → Bug 闭环修复
  └─ 闭环超限 → 暂停项目

Stage 4: 集成验证（可选，远期）
```

### 2.3 项目状态机变更

```
旧：designing → running → paused / completed
新：designing → reviewing → running → paused / completed
                  ↑                     │
                  └─────────────────────┘ (review_failed 回退)
```

**数据库变更**：mvp_project.status 新增 `reviewing` 值。

### 2.4 方案审核阶段详细设计

#### 2.4.1 触发时机

用户点击"确认方案" → 不再直接 draft→pending，而是进入 `reviewing` 阶段。

#### 2.4.2 三步审核流程

**Step 1：系统预检（零 AI 消耗，毫秒级）**

| 检查项 | 规则 | 级别 |
|--------|------|------|
| 任务名非空 | len(name) > 0 | error |
| 任务描述质量 | len(desc) >= 10 且含中/英文字符 | error |
| affected_resources 格式 | 路径格式合法（无乱码/特殊字符） | error |
| affected_resources 存在性 | 编码类项目：文件或目录必须存在 | warning |
| depends_on 有效性 | 引用的任务名在当前列表中 | error（自动修正） |
| batch_no 一致性 | 有依赖的任务 batch_no > 被依赖任务的 batch_no | warning（自动修正） |
| 资源冲突预检 | 同 batch_no 内不能两个任务改同一文件 | error |
| role_level 覆盖 | 项目角色配置中有对应的 role_type + role_level | warning |

预检结果：error 数量 > 0 → 阻止进入下一步；warning 附加到任务描述中。

**Step 2：审计员 AI 审核（质量门）**

创建一个审核任务（role_type=auditor，batch_no=0），输入为所有 draft 任务的 JSON 概要。

审计员审核维度：
- 任务描述是否明确可执行
- 任务粒度是否合理（不能太大也不能太碎）
- 依赖关系是否合理
- 是否遗漏了关键模块
- 角色分配是否合理

审核输出格式：
```json
{
  "approved": true/false,
  "issues": [
    {"task_name": "xxx", "severity": "error/warning", "message": "问题描述"}
  ],
  "suggestions": "整体建议"
}
```

**Step 3：协调员 AI 优化（首次激活协调员）**

如果审核通过，创建协调员优化任务（role_type=coordinator，batch_no=0）：
- 资源冲突精细检测
- 批次顺序优化
- 并行度建议
- 预估执行时间

协调员输出格式：
```json
{
  "optimized_batches": {
    "任务名": {"batch_no": 1, "reason": "调整原因"}
  },
  "parallelism_score": 0.8,
  "estimated_duration": "约2小时",
  "warnings": []
}
```

#### 2.4.3 审核失败处理

审核不通过时：
1. 将 issues 列表转为架构师修正任务（batch_no=0）
2. 项目状态退回 `designing`
3. 在架构师对话中发送通知消息，附带审核结果
4. 用户可以和架构师讨论修改后重新确认

#### 2.4.4 审核超时保护

审核总时长上限 5 分钟（可配置）。超时自动跳过 AI 审核，仅保留系统预检结果。

## 四、阶段隔离强化

### 4.1 当前问题

running 状态下 bugloop 可以创建 batch_no=0 的架构师任务，这些任务的 AI 输出可能和正在执行的实施员任务冲突（修改同一文件）。

### 4.2 解决方案

引入**阶段锁**：不同阶段只允许特定角色执行。

| 项目状态 | 允许执行的角色 | 禁止的操作 |
|----------|--------------|-----------|
| designing | architect | 不能创建实施/审计任务 |
| reviewing | auditor, coordinator | 不能创建实施任务 |
| running | implementer, auditor | 架构师任务仅限 Bug分析/失败分析（名称前缀限制） |
| paused | 无 | 全部暂停 |

实现方式：在 `scheduleOnce` 中增加角色过滤。

## 五、状态机强制应用

### 5.1 当前状态

`state_machine.go` 存在但 `ValidateTransition()` 从未被调用。

### 5.2 改造

创建 `updateTaskStatus()` 函数封装所有任务状态变更，内部强制调用 `ValidateTransition()`：

```go
func updateTaskStatus(ctx context.Context, taskID int64, fromStatus, toStatus string, extra g.Map) error {
    if err := ValidateTransition(fromStatus, toStatus); err != nil {
        return err
    }
    data := g.Map{"status": toStatus, "updated_at": gtime.Now()}
    for k, v := range extra {
        data[k] = v
    }
    _, err := g.DB().Model("mvp_task").
        Where("id", taskID).
        Where("status", fromStatus). // CAS
        Update(data)
    return err
}
```

全部状态变更都通过此函数，替代散落各处的裸 `Update(g.Map{"status": xxx})`。

-- （MySQL VARCHAR 无需 ALTER，只需更新注释）
ALTER TABLE mvp_project MODIFY COLUMN status varchar(20) NOT NULL DEFAULT 'designing'
  COMMENT '项目状态: designing/reviewing/running/paused/completed';

-- 2. 任务状态补充 draft
ALTER TABLE mvp_task MODIFY COLUMN status varchar(20) NOT NULL DEFAULT 'pending'

ALTER TABLE mvp_task DROP INDEX idx_project;

ALTER TABLE mvp_task DROP INDEX idx_mvp_task_project_batch;
ALTER TABLE mvp_task DROP INDEX idx_mvp_task_conversation;
ALTER TABLE mvp_conversation DROP INDEX idx_mvp_conversation_project;

## 七、task_parser 事务化

### 7.1 问题

ParseAndCreateTasks 分 4 步清理旧数据（日志→依赖→任务），中间 DB 断开会数据不一致。

### 7.2 方案

用 GoFrame 的事务包裹清理 + 创建操作：

```go
err := g.DB().Transaction(ctx, func(ctx context.Context, tx gdb.TX) error {
    // 所有清理和创建在同一事务中
    return nil
})
```

## 八、全局上下文并发控制

### 8.1 问题

mergeIntoGlobalContext 无锁，多批次同时完成时互相覆盖。

### 8.2 方案

在 ContextCompressor 中为每个项目维护一个 sync.Mutex：

```go
type ContextCompressor struct {
    projectLocks sync.Map // projectID → *sync.Mutex
}
```
