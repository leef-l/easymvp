# EasyMVP 自治宪法操作系统终版方案

作者：架构师B

> 文档性质：超前理念终版
> 与 [EasyMVP自治系统终极基线：七层自治模型.md](/www/wwwroot/project/easymvp/docs/EasyMVP自治系统终极基线：七层自治模型.md) 的关系：
> 本文档不替代实施基线，而是给出我对 EasyMVP 最终形态的定稿立场。
> 工程实施以七层基线为准，理念边界以本文档为准。

---

## 1. 最终判断

如果只允许我给 EasyMVP 定一个终极方向，我的答案是：

`EasyMVP 不是多角色系统，不是更强 Planner，不是更聪明的 Agent 平台。`

`EasyMVP 的终态应该是：自治宪法操作系统。`

它治理的核心不是：

- 谁来做
- 用哪个模型
- 哪个执行器更强

而是：

- 当前什么状态变化被允许发生
- 这些变化基于什么证据
- 在什么目标边界下允许自动发生
- 当不确定时系统如何自动收紧自己

一句话定义：

`EasyMVP 是一个对任务、资源、风险、证据、状态变化进行统一治理的自治控制系统。`

---

## 2. 为什么这才是终态

真正高级的自治系统，不是更像人，而是更像一个受约束的操作系统。

传统 AI 系统的重心通常是：

1. 提升模型能力
2. 增加角色数量
3. 提高规划复杂度
4. 让系统更敢自动做事

这些都不是真正的终态。

真正的终态是：

1. 知道什么可以自动做
2. 知道什么绝不能自动做
3. 知道证据不足时必须停
4. 知道高层组件绝不能直接改世界
5. 知道学习机制只能让系统更稳，不能让系统更野

所以最终竞争力不来自“更聪明”，而来自：

- 可控
- 可证明
- 可回滚
- 可审计
- 可长期无人化运行而不失控

---

## 3. 终态主线

我认为 EasyMVP 的终态主线应固定为：

`Intent -> Contract -> Execute -> Evidence -> StateChange`

### 3.1 Intent

定义：

- 为什么现在做
- 这件事服务于什么目标
- 在什么预算、时限、风险边界下做

### 3.2 Contract

定义：

- 需要什么能力
- 成功必须产出什么
- 如何验证合格
- 允许什么副作用

### 3.3 Execute

定义：

- 真正改世界的唯一阶段
- 执行器、工具、节点、沙箱、云资源都属于这一层

### 3.4 Evidence

定义：

- 实际做了什么
- 产出了什么
- 结果是否可信
- 证据是否充分

### 3.5 StateChange

定义：

- 是否允许继续推进
- 是否允许自动推进
- 是否应转入 `rework / pause / manual_review / rollback`

最终要治理的不是 Role，也不是 Capability，而是：

`StateChange 的合法性。`

## 4. 最终架构观

我认可当前仓库中的七层实施基线，但从理念终态看，我主张用下面这组对象来理解系统：

1. `Constitution`
2. `Goal`
3. `Strategy`
4. `Situation`
5. `Contract`
6. `Capability & Routing`
7. `Execution`
8. `Evidence & Governance`
9. `Learning & Meta-Cognition`

如果压缩回工程语境，可以等价映射为现有七层：

| 终态对象 | 工程落点 |
|----------|----------|
| Constitution | 渗透所有层的不可突破规则 |
| Goal | L6 目标层 |
| Strategy + Situation | L5 策略层 |
| Contract | L4 契约层 |
| Capability & Routing | L3 能力层 |
| Execution | L2 执行层 |
| Evidence & Governance | L1 证据与治理层 |
| Learning & Meta-Cognition | L7 元认知层 + 学习闭环 |

关键点不是层数，而是：

`高层只约束，低层才执行。`

---

## 5. 我坚持的四个核心对象

### 5.1 Constitution

这不是一份文档，而是一组不可突破的系统边界。

宪法规定：

1. 哪些状态变化绝不能自动发生
2. 哪些动作必须人工批准
3. 哪些高层组件永远不能直接产生副作用
4. 哪些证据不足时必须降级
5. 哪些参数元认知可以调，哪些永远不能调
6. 哪些场景系统只能更保守，不能更激进

### 5.2 Situation

我坚持 `Situation` 必须是一等公民。

系统所有核心判断组件都不该各自乱读数据库，而应统一读取当前态势快照。

`AdmissionControl / Planner / RiskGate / Operator / Assessor / Tuner`

都应该以 `Situation` 作为统一输入。

否则系统只是在“很多规则一起跑”，不是“在统一世界模型上决策”。

### 5.3 EffectContract

我坚持副作用必须是显式契约，不允许继续隐式存在。

必须明确：

- `ReadSet`
- `WriteSet`
- `ExternalActions`
- `SideEffectLevel`
- `Rollbackability`
- `ApprovalClass`

真正把系统搞崩的，通常不是匹配错能力，而是副作用失控。

### 5.4 DecisionMeta

所有跨层决策都必须带：

- `confidence`
- `evidence_sufficiency`
- `reversibility`
- `blast_radius`

系统不只要判断“能不能做”，还必须判断：

- 我有多确定
- 证据够不够
- 做错了能不能回滚
- 影响面有多大

这四个量是整个自治系统的硬底座。

## 6. 六条终极稳定性铁律

### 6.1 高层永远不能直接改世界

高层只能：

- 提建议
- 做准入
- 调参数
- 写审计

高层不能：

- 直接执行动作
- 直接改状态
- 直接绕过闸门

### 6.2 目标层永远只能收紧系统

目标层只能：

- 拒绝
- 降级
- 缩预算
- 提高人工等级

不能：

- 放宽权限
- 降低人工要求
- 允许更高副作用

### 6.3 元认知层永远只能调参数

元认知层可以：

- 调阈值
- 调权重
- 调推荐顺序
- 调灰度比例

不能：

- 发起任务
- 触发回滚
- 推进状态
- 放宽宪法级约束

### 6.4 所有状态推进必须绑定证据

没有证据，不允许推进。

证据不足，不允许猜测式自动化。

### 6.5 LLM 只能分析、建议、精排，不能决定权限和状态

以下事项必须由规则或系统内核决定：

- 权限判断
- 风险闸门求值
- 状态迁移
- 锁、CAS、幂等
- 审计落库

### 6.6 学习机制优先学习边界，不优先学习激进最优

学习机制最先应该学会的是：

- 哪些场景必须人工
- 哪些场景证据永远不够
- 哪些场景副作用不可接受
- 哪些场景必须保守

而不是先学：

- 哪个模型更敢做
- 哪个执行器更省钱
- 哪个策略更激进

## 7. 我对 Goal / Situation / Learning 的最终态要求

### 7.1 Goal：约束，不是优化器

`ProjectObjective` 的本质不是“追求最优”，而是“划定边界”。

最关键的不是 DeliveryGoal，而是：

- `autonomy_level`
- `max_auto_retries`
- `max_auto_reworks`
- `max_auto_replans`
- `max_side_effect_level`
- `human_mandatory_points`

这些字段才是真正决定系统是否稳的东西。

### 7.2 Situation：统一世界模型

如果没有统一态势对象，Planner 和 AdmissionControl 最终都会演化成：

- 各自读自己的状态
- 各自推自己的结论
- 系统越来越不可解释

所以 `Situation` 不是可选项，而是终态必须项。

### 7.3 Learning：只允许保守自治

学习机制必须明确：

1. 小样本不更新
2. 陈旧样本权重衰减
3. 单次调整幅度受限
4. 激进方向必须人工批准

一句话：

`自动学习只能让系统更稳，不能自动让系统更大胆。`

---

## 8. 角色的最终定位

我认为角色不应该再是系统内部的主调度实体。

角色应该降级为：

`治理视图`

角色继续保留，但职责变成：

- 审计归属
- UI 展示
- 人工理解
- 责任边界
- 成本归集

系统真正的调度主键应该是：

- 目标
- 契约
- 态势
- 风险
- 证据

不是“哪个角色上”。

所以我最终承认五个稳定角色是有价值的，但它们不再是内核运行学的核心。

---

## 9. 为什么这比普通多 Agent 更高级

因为它追求的不是“更像人”，而是：

- 更像受约束的控制系统
- 更像带审计的运行内核
- 更像知道自己边界的自治系统

普通多 Agent 系统常见的问题是：

1. 会做很多事，但不知道哪些不能做
2. 有很多判断，但没有足够证据
3. 有很多学习，但越学越激进
4. 有很多角色，但边界并不硬

而自治宪法操作系统的核心是：

`知道什么时候必须停。`

这才是长期稳定无人化的前提。

## 10. 与当前仓库基线的关系

我认可当前仓库中的：

- [EasyMVP自治系统终极基线：七层自治模型.md](/www/wwwroot/project/easymvp/docs/EasyMVP自治系统终极基线：七层自治模型.md)

作为唯一实施基线。

我在本文档里保留的最终立场只有三条：

1. 终态必须是 `自治宪法操作系统`
2. 终态必须以 `StateChange 合法性` 作为核心治理对象
3. 终态必须坚持：
   - 宪法约束
   - 目标减法
   - 统一态势
   - 契约执行
   - 证据闭环
   - 保守学习

如果后续工程实现偏离这六个方向，即使功能更多，也不是我认可的终态。

---

## 11. 最终结论

如果只保留一句话来定义 EasyMVP 的终态，我给出的最终版本是：

`EasyMVP = 在目标边界内，对状态变化进行准入、执行、验收、恢复、学习与审计的自治宪法操作系统。`

再压缩成更短的一句：

`不是更会自动做事，而是更知道自己什么时候绝不能自动做事。`

这就是我作为架构师B给出的最终方案。
