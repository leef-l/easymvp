# 执行器统一抽象与生命周期设计文档

## 一、目标

定义执行器在系统中的统一抽象、状态机和生命周期，避免每个执行器各走一套。

---

## 二、统一对象

### 2.1 Executor

表示一个可注册的执行器实现。

### 2.2 ExecutionSession

表示某次任务执行实例。

### 2.3 ExecutionEvent

表示执行过程中的结构化事件。

---

## 三、生命周期

统一生命周期建议：

1. `created`
2. `starting`
3. `running`
4. `streaming`
5. `succeeded`
6. `failed`
7. `canceled`
8. `timed_out`

---

## 四、状态推进

### 4.1 DomainTask 与 ExecutionSession 的关系

- `DomainTask.pending`：尚未启动执行器
- `DomainTask.running`：执行器已进入 `starting/running/streaming`
- `DomainTask.completed`：执行器 `succeeded`
- `DomainTask.failed`：执行器 `failed/timed_out`

### 4.2 不允许的情况

1. 任务已完成但 session 仍是 running
2. session 已失败但任务仍是 running
3. cancel 后 session 继续向任务写成功结果

---

## 五、统一事件模型

建议事件类型：

- `execution.created`
- `execution.started`
- `execution.stdout`
- `execution.stderr`
- `execution.heartbeat`
- `execution.artifact`
- `execution.succeeded`
- `execution.failed`
- `execution.canceled`

---

## 六、公共请求结构

请求中统一传递：

1. 任务身份
2. 角色与模型
3. 工作目录
4. 写文件列表
5. 只读参考文件
6. 任务提示词
7. 超时与步数限制

---

## 七、取消模型

所有执行器都必须支持：

1. `context.Context` 取消
2. 超时取消
3. 项目/WorkflowRun 级暂停取消
4. 手动终止

CLI 型执行器通过 `exec.CommandContext` 实现。  
HTTP/Agent 型执行器通过远端 session cancel API 实现。

---

## 八、结果结构

统一结果：

- `exitCode`
- `summary`
- `rawOutput`
- `artifacts`
- `failureCategory`
- `retryable`
- `duration`

---

## 九、产物模型

产物包括：

- 修改文件列表
- diff
- 日志文件
- 执行报告
- 截图/录屏

---
