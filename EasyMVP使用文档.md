# EasyMVP 使用文档

> 版本：v2.0 | 更新日期：2026-04-02

---

## 1. 系统简介

EasyMVP 是一个 AI 协作开发平台。你只需要描述需求，系统会自动完成代码开发：

- **架构师 AI** 帮你分析需求、设计方案、拆分任务
- **实施员 AI** 通过 Aider 真实编辑代码文件
- **审计员 AI** 自动审查代码质量
- **看门狗** 自动监控任务状态，失败自动重试

---

## 2. 快速开始

### 2.1 前置配置

在使用 MVP 功能之前，需要先完成 AI 模块的配置：

#### 第一步：配置 AI 供应商

**导航：** AI管理 → 供应商管理

添加你的 AI 服务供应商（如腾讯云 Coding Plan）：

| 字段 | 示例 |
|------|------|
| 名称 | 腾讯云 Coding |
| 代码 | tencent-coding |
| 供应商类型 | anthropic |
| Base URL | `https://api.lkeap.cloud.tencent.com/coding/anthropic/v1` |

#### 第二步：配置 API 套餐

**导航：** AI管理 → 套餐管理

为供应商添加 API 密钥：

| 字段 | 示例 |
|------|------|
| 供应商 | 腾讯云 Coding |
| 名称 | Coding Plan 主套餐 |
| API Key | `sk-sp-xxxxx` |

#### 第三步：添加 AI 模型

**导航：** AI管理 → 模型管理

添加各角色需要的模型：

| 名称 | 模型代码 | 角色 | 说明 |
|------|---------|------|------|
| Auto (tc-code-latest) | tc-code-latest | architect | 架构师专用，最强模型 |
| GLM-5 | glm-5 | implementer | 实施员专用，代码编辑 |
| hunyuan-turbos | hunyuan-turbos | auditor | 审计员专用，代码审查 |
| MiniMax-M2.5 | minimax-m2.5 | coordinator | 协调员，轻量任务 |

**重要字段：**
- **角色（capability）**：选择 architect / implementer / auditor / coordinator
- **角色提示词（role_prompt）**：填写该模型的角色定位描述，越详细越好

#### 第四步：配置角色预设

**导航：** MVP → 角色预设

角色预设是新项目的默认配置模板。每创建一个项目，系统会自动根据预设创建该项目的角色配置。

建议配置：

| 角色类型 | 等级 | 模型 | 说明 |
|---------|------|------|------|
| architect | max | tc-code-latest | 架构师（最强模型分析需求） |
| implementer | pro | GLM-5 | 实施员-常规任务 |
| implementer | lite | Hunyuan-T1 | 实施员-简单任务 |
| implementer | max | GLM-5 | 实施员-核心任务 |
| auditor | pro | hunyuan-turbos | 审计员-常规审查 |
| coordinator | lite | MiniMax-M2.5 | 协调员 |

### 2.2 安装 Aider

Aider 是实施员 AI 用来编辑代码的工具，需要在服务器上安装：

```bash
pip3 install aider-chat
```

验证安装：
```bash
aider --version
# 输出: aider 0.86.2
```

## 3. 使用流程

### 3.1 创建项目

**导航：** MVP → 项目管理 → 新建项目

| 字段 | 说明 |
|------|------|
| 项目名称 | 如"招投标网" |
| 项目简介 | 详细描述你的需求（越详细越好） |
| 架构师AI模型 | 默认从角色预设读取，可手动更改 |

点击确认后，系统自动：
1. 创建项目记录
2. 根据角色预设创建所有角色配置
3. 创建架构师对话
4. 跳转到对话页面

### 3.2 与架构师对话

创建项目后自动进入对话页面，你可以和架构师 AI 沟通：

**对话示例：**
```
用户: 我想开发一个招投标网站，需要企业注册、项目发布、投标管理、评标打分等功能。
      技术栈用 Go + Vue3，数据库用 MySQL。

架构师: 收到。让我分析一下需求...
        [详细的需求分析和技术方案]

用户: 方案可以，请拆分任务。

架构师: 好的，以下是任务清单：
        {
          "tasks": [
            {"name": "数据库设计", "batch_no": 1, ...},
            {"name": "用户注册API", "batch_no": 2, ...},
            ...
          ]
        }
```

**关键点：**
- 你可以和架构师多轮对话，细化需求
- 让架构师输出 JSON 格式的任务清单
- 系统自动从 AI 回复中解析任务（页面顶部会显示"有 N 个待确认任务"）
- 架构师可以多次输出方案，系统以最新为准

### 3.3 填写代码工作目录

在确认方案之前，**必须**设置项目的代码工作目录：

**导航：** MVP → 项目管理 → 编辑（点击项目行的编辑按钮）

| 字段 | 说明 |
|------|------|
| 代码工作目录 | Aider 执行代码编辑时的项目根目录 |

**示例：**
```
/www/wwwroot/project/my-bidding-app
```

这个目录是实施员 AI 通过 Aider 编辑代码时的工作目录。请确保：
- 目录已存在
- 目录已初始化 Git 仓库（`git init`）
- 后端服务进程有读写权限

### 3.4 确认方案

在对话页面，确认任务拆分合理后：

1. 检查页面顶部显示的"待确认任务数"
2. 点击 **确认方案** 按钮
3. 系统弹出确认提示，点击"确认执行"

确认后系统自动：
1. 将所有 draft 任务转为 pending
2. 压缩架构师对话为全局上下文摘要
3. 启动任务调度器
4. 项目状态变为 **执行中（running）**

也可以在项目列表页点击"确认方案"按钮。

### 3.5 监控执行进度

#### 项目列表

**导航：** MVP → 项目管理

| 列 | 说明 |
|----|------|
| 状态 | 设计中 / 执行中 / 已暂停 / 已完成 |
| 任务进度 | 进度条 + 完成数/总数 |

#### 任务列表

**导航：** MVP → 任务管理

可以查看所有任务的状态：

| 状态 | 含义 |
|------|------|
| draft | 草稿（架构师拆分但未确认） |
| pending | 待执行（等待依赖完成或资源释放） |
| running | 执行中（AI 正在工作） |
| completed | 已完成 |
| failed | 失败（看门狗会自动重试） |
| bug_found | 审计发现 Bug（进入修复流程） |

#### 项目状态面板

在项目列表点击"查看详情"，可以看到：
- 各状态任务数量统计
- 正在执行的任务列表
- 失败任务和错误信息

### 3.6 暂停与恢复

#### 暂停项目

执行中的项目可以暂停：
1. 项目列表 → 点击"暂停"
2. 输入暂停原因
3. 所有 running 任务回退为 pending

暂停后可以回到对话页面，和架构师讨论调整方案。

#### 恢复项目

暂停的项目可以恢复：
1. 项目列表 → 点击"恢复"
2. 系统重新启动调度器
3. pending 任务继续执行

### 3.7 手动重试失败任务

除了看门狗自动重试，你也可以手动重试：

**导航：** MVP → 任务管理 → 找到 failed 任务 → 重试

## 4. 高级功能

### 4.1 角色模型分级

每个任务都有 `role_level`（等级），决定使用哪个模型：

```
任务A: role_level=max → 使用最强模型（如 tc-code-latest）
任务B: role_level=pro → 使用中等模型（如 glm-5）
任务C: role_level=lite → 使用轻量模型（如 hunyuan-t1）
```

架构师在拆分任务时会为每个任务标注合适的等级，核心模块用 max，简单任务用 lite，节省 API 成本。

### 4.2 任务依赖与并发

架构师拆分任务时会标注：
- **batch_no（批次号）**：同批次的任务可以并发执行
- **depends_on（依赖）**：必须等依赖任务完成后才执行
- **affected_resources（涉及资源）**：同时操作同一文件的任务不能并发

```
批次1: 数据库设计（无依赖）→ 可立即执行
批次2: 用户API、订单API（依赖数据库设计）→ 批次1完成后并发执行
批次3: 前端页面（依赖对应API）→ 批次2完成后执行
```

### 4.3 Bug 自动闭环

```
实施员完成任务 → 自动创建审计任务
  ↓
审计员审查代码
  ├─ 通过 → 任务完成
  └─ 发现 Bug → 上报
       ↓
     架构师分析 Bug 原因
       ↓
     实施员修复代码
       ↓
     审计员重新审查
```

整个过程全自动，无需人工干预。

### 4.4 上下文压缩

系统自动管理 AI 的上下文大小：

- 每个任务完成后，结果自动压缩为摘要
- 每个批次完成后，批次摘要合并进全局上下文
- 全局上下文保持在 ~3000 字以内
- 后续任务自动获得前置任务的关键信息

### 4.5 看门狗监控

系统后台运行看门狗，自动处理异常：

| 检测项 | 间隔 | 动作 |
|-------|------|------|
| 心跳检测 | 2 分钟 | 连续 3 次无进展 → 标记 failed |
| 自动重启 | 2.5 分钟 | failed 任务自动重试（最多 3 次） |
| 升级处理 | - | 重试 3 次仍失败 → 升级给架构师 |

## 5. 模型配置指南

### 5.1 角色提示词（role_prompt）

在 AI 管理 → 模型管理中，每个模型的"角色提示词"非常重要。建议针对不同角色编写专门的提示词：

**架构师模型提示词示例：**
```
你是一位资深软件架构师，擅长需求分析和系统设计。

你的职责：
1. 深入理解用户需求，提出澄清问题
2. 设计合理的技术方案
3. 将项目拆分为 80~200 个细粒度任务
4. 确保并行任务不修改同一个文件
5. 输出 JSON 格式的任务清单

任务清单格式：
{
  "tasks": [
    {
      "name": "任务名称",
      "description": "详细描述（让实施员能独立完成）",
      "role_level": "max/pro/lite",
      "batch_no": 1,
      "affected_resources": ["file1.go"],
      "depends_on": []
    }
  ]
}
```

**实施员模型提示词示例：**
```
你是一位专业的软件开发工程师。

你的职责：
1. 根据任务描述编写高质量代码
2. 遵循项目现有的代码风格和约定
3. 编写必要的注释
4. 考虑边界情况和错误处理
```

**审计员模型提示词示例：**
```
你是一位严谨的代码审计员。

你的职责：
1. 审查代码的正确性和完整性
2. 检查是否有安全漏洞（SQL注入、XSS等）
3. 检查代码风格和可维护性
4. 如果发现问题，详细描述问题和修复建议
```

### 5.2 当前支持的模型

通过腾讯云 Coding Plan，当前支持以下模型：

| 模型 | 代码 | 推荐角色 | 特点 |

| Kimi-K2.5 | kimi-k2.5 | architect (pro) | 长上下文分析 |

| MiniMax-M2.5 | minimax-m2.5 | coordinator | 协调总结 |

## 6. 常见问题

### Q1: 确认方案按钮点不了？

**原因：** 架构师还没有输出有效的 JSON 任务清单，或 JSON 中包含注释导致解析失败。

**解决：**
1. 在对话中明确要求架构师"请输出 JSON 格式的任务清单"
2. 检查页面顶部是否显示"有 N 个待确认任务"
3. 如果架构师已经输出了但任务数为 0，可能是 JSON 格式问题，让架构师重新输出

### Q2: 任务一直是 pending 状态？

**可能原因：**
- 依赖的任务还没完成
- affected_resources 与其他 running 任务冲突
- 并发数已满（默认最大 20）

**查看方法：** 检查该任务的 `depends_on` 和 `affected_resources` 字段。

### Q3: 实施员任务执行失败？

**可能原因：**
- work_dir 路径不存在或无权限
- Aider 未安装或 API Key 无效
- 模型返回非预期格式

**解决：**
- 检查项目的"代码工作目录"是否正确
- 运行 `aider --version` 确认安装
- 查看任务的 error_message 字段

### Q4: 如何调整已确认的任务？

暂停项目 → 修改任务 → 恢复项目。

### Q5: API 调用费用如何控制？

- 简单任务标记为 `lite`，使用轻量模型
- 上下文压缩减少 token 消耗
- 避免不必要的 max 等级任务
