# EasyMVP Control Doctrine

把钱学森《工程控制论》转成 EasyMVP 可执行的控制原则。

这不是一个“聊天风格”技能，而是所有设计、评审、编排、验收的顶层约束。

## 何时使用

出现下列任一情况时使用：

- 设计或修改 `WorkflowRun`、阶段状态机、风险闸门
- 设计 `acceptance / evidence / issue / verification / rework`
- 讨论 `brain-v3`、`easymvp-brain`、`EasyMVP` 三者边界
- 定义任务完成、失败、返工、恢复语义
- 讨论验证口径、远端验证环境、GitHub Actions 替代策略
- 讨论多脑协作是否稳定、是否互相干扰、是否容易振荡

## 核心原则

### 1. 闭环优先

- 执行成功不等于完成
- 必须经过验证、裁决、证据归档
- 没有 evidence，不算 acceptance

标准闭环：

```text
设计 → 编译 → 执行 → 验证 → 裁决
                    ↓
                  失败
                    ↓
             故障分析 → 返工重构 → 再执行
```

### 2. 稳定优先于盲目加速

- 不为追求自动化和并行，牺牲稳定性、恢复能力、审计能力
- 当系统处于高不确定、高耦合、高失败率状态时，优先降级、减速、分批、人工介入

### 3. 分层控制

固定分层，不混责：

- `brain-v3`：多脑运行时与控制面
- `easymvp-brain`：EasyMVP 领域判断与裁决
- `EasyMVP`：业务工作流与交付闭环

禁止：

- 让通用代码脑承担业务裁决
- 让业务层绕过合同直接把执行成功视为完成
- 让验收层直接变成执行层

### 4. 最小互扰

- 并行任务必须显式隔离资源
- 同一文件、模块、工作区、交付对象，不允许无规则并发修改
- 优先使用 worktree、resource lock、batch gate、depends_on

### 5. 时滞显式处理

必须假设这些都有延迟：

- 高配验证环境返回
- GitHub Actions 返回
- `brain serve` 状态同步
- 人工审批
- 浏览器采证

因此：

- 不能把“未返回”当“通过”
- 必须有轮询、超时、重试、降级和人工接管策略

### 6. 噪声过滤

以下都不是最终结论，只是原始信号：

- 模型自然语言输出
- 原始日志
- 单次运行结果
- 单条审计意见

必须经过：

- 结构化提炼
- 分级
- 证据归档
- 统一裁决

### 7. 约束下求最优

优化目标不能脱离约束：

- 上下文预算
- 远端验证资源
- 人工审批预算
- 风险等级
- 任务时效

当约束冲突时，优先级通常是：

```text
安全 / 数据一致性 / 交付正确性
> 稳定性
> 可恢复性
> 速度
> 体验优化
```

### 8. 误差可回流、可审计

- 错误不能被“总结成功”掩盖
- 必须进入 `review issue / accept issue / evidence / replay`
- 返工必须结构化，不允许临时补丁式闭嘴修

## 远端验证原则

最终验证口径：

- **高配验证环境**

当前阶段替代实现：

- **GitHub Actions**

本地低配开发机：

- 不承担最终编译与最终验收

因此：

- 不把本地低配机上的 `go test` / `go build` / 前端 build 视为最终通过依据
- 当前可以接受 GitHub Actions artifact / log / workflow run 作为正式验证证据
- 后续有高配验证机后，应把验证 contract 从“GitHub Actions 特定实现”提升为“远端高配验证通道”

## 与 ECC 的吸收关系

本技能吸收 `Everything Claude Code` 的这些机制，但要转换成 EasyMVP 语言：

### `search-first`

- 先看仓内已有实现
- 再查官方文档
- 再查外部成熟方案
- 最后才新增抽象或新实现

### `verification-loop`

- 执行完成只是中间态
- 必须进入验证闭环

### `context-budget`

- 长 prompt 不是资产，冗余上下文会直接降低控制质量
- 要限制角色提示、技能内容、合同模板长度

### `rules over memory`

- 能写成规则、合同、hook、状态机的，不要只靠模型记忆

## 设计检查清单

在设计工作流、阶段、验收、返工时，至少检查：

- [ ] 是否形成闭环，而不是“执行即完成”
- [ ] 是否区分了运行成功、交付成功、验收通过
- [ ] 是否处理了远端验证、人工审批等时滞
- [ ] 是否有噪声过滤，而不是直接吃原始输出
- [ ] 是否限制了任务互扰
- [ ] 是否提供了失败回流路径
- [ ] 是否允许降级或人工接管
- [ ] 是否把最终验证放在高配验证环境而非低配宿主机

## 输出要求

当使用本技能给方案或评审意见时，优先输出：

1. 控制结构是否闭环
2. 哪一层在越权
3. 哪些时滞未处理
4. 哪些噪声被误当结论
5. 哪些环节可能引发返工振荡
6. 最终验证口径是否正确

不要输出空泛的“建议加强稳定性”。要明确指出：

- 失稳点在哪
- 应改到哪一层
- 应增加什么 contract / evidence / gate / fallback
